# Phase 07.2 Plan 01: Group A confirmed _validate_config bug fixes -- Summary

Fixed 5 of the 6 Group A confirmed bugs from the Phase 7.1 audit follow-up.
Each component's `_validate_config()` was measuring content of fields that
may legitimately hold `${context.X}` references at validate time -- which
runs before `_resolve_expressions()`. Content checks were moved into
`_process()` (after Step 3 has resolved expressions), keeping
`_validate_config()` to key-presence and shape checks only.

## Pattern Applied

For each component (uniform across all five):

1. Stripped `_validate_config` to only key-presence and `isinstance` shape
   checks. Added a docstring note referencing CR-06 / quick task 260429-hc2
   for the analogous post-fix pattern in `file_output_delimited.py`.
2. Moved content-checks into `_process` at the earliest point each field
   is read, raising `ConfigurationError` with the same message strings
   the original `_validate_config` produced.
3. Imported `ConfigurationError` from `..exceptions` where missing.
4. Wrote regression tests proving:
   - `_validate_config` accepts `${context.X}` literal (returns `[]`).
   - `_process` resolves the context var correctly end-to-end with a
     `ContextManager` populated with the value.
   - Invalid resolved values still raise `ConfigurationError` (timing
     is later, behaviour is preserved).

## Out-of-Scope Items (deliberately retained)

Per CONTEXT.md decision A scope locking, the following content-checks
remain in `_validate_config` and are NOT addressed in this plan:

- `file_archive.py`: `ARCHIVE_FORMAT` enum membership check (line 92)
- `file_input_excel.py`: `footer` and `limit` `.isdigit()` checks (lines 130-141)

These are documented inline in the respective `_validate_config`
docstrings as candidates for a future cleanup phase if a Talend job
is found using a context var for those fields.

## Deviations from Plan

None substantive. All five tasks executed exactly as planned.

Two minor adjustments worth noting:

1. **Test helper detail (Rule 3 -- blocking issue, fixed inline):** All
   five test files needed a small adjustment to populate `comp.config`
   from the original config dict before calling `_validate_config()` in
   isolation. `BaseComponent.__init__` deepcopies into
   `_original_config` and sets `self.config = {}`; only `execute()` Step
   1 populates `self.config`. Direct unit tests of `_validate_config`
   that bypass `execute()` therefore see an empty `self.config` and all
   key-presence checks fail. Fixed inline by mirroring Step 1 in the
   `_make_component` helper of each new test file.

2. **file_input_excel test fixture detail:** `output_schema` was set to
   `None` on the component instance in test fixtures, since
   `_read_sheet` references it during execution and `BaseComponent.__init__`
   does not initialise it. Existing tests for other components do this
   either via a `schema=` kwarg on the helper or by leaving it
   unreferenced -- file_input_excel reads it unconditionally so explicit
   assignment was needed.

## Threat Flags

None. This is a pure engine-internal refactor moving validation from one

boundaries, no external input surface, no new auth / file access /
network paths.
