# Phase 07.2: validate-config bug sweep -- Context

**Captured:** 2026-04-29
**Status:** Ready for planning (decisions locked from prior analysis)

## Background

`BaseComponent.execute()` runs in this order (`src/v1/engine/base_component.py:204-260`):
1. Step 1: `self.config = copy.deepcopy(self._original_config)` -- raw config restored
2. Step 2: `self._validate_config()` -- runs on UNRESOLVED config
3. Step 3: `self._resolve_expressions()` -- context vars (`${context.X}`, `context.X`) and Java `{{java}}` markers resolved

**Bug pattern:** Any `_validate_config()` that inspects the *content* of a config field (length, regex, type coercion, enum membership, file-exists, numeric range) -- when that field can hold a context-var reference -- will spuriously reject or crash on valid configs because it measures the unresolved literal, not the runtime value.

This pattern produced the original Phase 7.1 CR-06 bug (already fixed in commit 43762c8 + quick task 260429-hc2). An audit of all 30+ engine components found 6 more confirmed instances + 5 suspects.

## Locked decisions

### A. Six confirmed bugs to fix (move content checks from `_validate_config` to `_process` post-resolution)

| File:line | Field | Issue | Failure mode |
|---|---|---|---|
| `src/v1/engine/components/file/file_archive.py:88-93` | `compression_level` | `int(value)` coercion | Crashes ValueError on `${context.LEVEL}` |
| `src/v1/engine/components/file/file_input_positional.py:100-106` | `pattern` | `int(x.strip())` per part | Crashes ValueError on `${context.WIDTHS}` |
| `src/v1/engine/components/file/file_input_positional.py:111-115` | `header_rows`, `footer_rows` | `int(value)` coercion | Crashes ValueError on `${context.ROWS}` |
| `src/v1/engine/components/transform/log_row.py:80-85` | `max_rows` | `int(value)` coercion | Crashes ValueError on `${context.MAX_ROWS}` |
| `src/v1/engine/components/transform/pivot_to_columns_delimited.py:108-109` | `field_separator` | `len() != 1` check | Rejects `${context.SEP}` (clone of CR-06) |
| `src/v1/engine/components/file/file_input_excel.py:117-120` | `header`, `first_column` | `.isdigit()` check | Rejects numeric `${context.HEADER_ROW}` |

**Fix pattern (uniform across all six):**
- Remove the content check from `_validate_config`
- Add equivalent check in `_process()` after `self.config` has been resolved (or after explicit `self._resolve_expressions()` if the component overrides execute)
- `_validate_config` retains only key-presence and container-shape checks (matches the new authoring rule)
- Preserve original error messages and exception types so behavior is identical at the right time

### C. Regression tests required for each fix

Each of the six confirmed fixes (and any suspect that ends up deferred) must add at least:
1. **Pre-fix demonstration** (committed but skipped/marked xfail before fix lands -- optional, may use existing tests if they fail with context vars): show `_validate_config` rejects/crashes on `${context.X}` value
2. **Post-fix happy path**: `_validate_config` accepts the unresolved string; resolution succeeds in `_process`; correct output produced
3. **Post-fix invalid resolved value still raises**: invalid resolved value (e.g. non-numeric string for an int field) raises the appropriate exception in `_process` -- timing is later but error is still surfaced

Test placement follows existing test file patterns under `tests/v1/engine/components/...`.

### D. Authoring rule documentation

New rule section in `docs/v1/standards/MANUAL_COMPONENT_AUTHORING.md`:

**Title:** "Rule: `_validate_config` may only check key presence and container shape -- never content"

**Content:**
- Why: BaseComponent lifecycle Step 2 (`_validate_config`) runs BEFORE Step 3 (`_resolve_expressions`). Content checks on unresolved config rejects valid context-var references.
- Allowed in `_validate_config`: key presence (`if not self.config.get("X"): raise`), container shape (`isinstance(value, list)`, `isinstance(value, dict)`), structural validity of immutable config (e.g. schema list shape).
- Disallowed in `_validate_config`: length checks (`len(x) != 1`), regex checks, `int()` / `float()` coercion, enum membership, file-exists / `os.path.exists`, numeric range comparisons.
- Where they belong: `_process()` after `self.config` is the resolved dict (or after explicit resolution).
- Code examples: good vs bad side-by-side, referencing BaseComponent step numbers.
- Cross-reference: `src/v1/engine/base_component.py` execute() lifecycle, this phase 07.2 sweep.

### E. Constraints

- ASCII-only across all changes (no emojis or unicode in code, comments, docstrings, doc additions, commit messages)
- Each component fix is one atomic commit
- Authoring rule addition is a separate commit
- Final pytest must show 0 NEW failures vs the baseline (we're fixing latent bugs that didn't have tests yet -- the new tests added in this phase prove the behavior, but should not break unrelated tests)

### F. Out of scope

- Schema validation, type coercion, output column ordering -- those live in BaseComponent steps 7b/7c (already correct)
- Java bridge expressions -- they resolve in step 3 alongside context vars (covered by deferring to _process)
- Adding NEW components -- only modifying the 11 in scope

## Specific implementation guidance

For Group A fixes, the per-component task structure should be:
1. Read the current `_validate_config` body
2. Identify the content-check lines (everything beyond key-presence and isinstance)
3. Move those lines verbatim into `_process` at the earliest point after `self.config` is the resolved dict (the engine's resolve_expressions runs as part of execute(), so by the time _process runs, config is resolved)
4. Add tests proving pre-fix-broken / post-fix-works
5. Commit with a per-component message format: `fix(07.2-NN): defer ${field} content check from _validate_config to _process`

For Group B suspects, the task structure is:
1. Read converter for the suspect component
2. Check if the converter ever emits `${context.X}` or `{{java}}` for the suspect field
3. If YES -> apply Group A fix pattern. If NO -> keep `_validate_config` as-is, document verdict in PROGRESS.md.

For Group D authoring rule:
- Single commit, single doc edit
- Bundle with phase summary
