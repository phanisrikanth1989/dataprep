### `_process` raises preserve original exception type (`ConfigurationError`) and message
When moving content checks from `_validate_config` to `_process`, the deferred check raises `ConfigurationError` with the same message string the original `_validate_config` used. No `ComponentExecutionError` re-wrapping, no message rewording.

**Rationale:** Caller / test contracts depend on exception type and message. Moving the *timing* of validation should not change *what* the caller observes. Preserves backward compatibility with all existing tests and consumers that catch `ConfigurationError`.
**Source:** 07.2-CONTEXT.md decision E; 07.2-01-SUMMARY.md; 07.2-02-SUMMARY.md (send_mail)

---

### send_mail smtp_port check placed OUTSIDE the broad `try/except Exception` in `_process`
The pre-existing `_process` body wraps work in `try: ... except Exception as e: raise ComponentExecutionError(...)`. A naive placement of the new smtp_port content check inside that block would have re-wrapped `ConfigurationError` as `ComponentExecutionError`, breaking the exception-type contract.

**Rationale:** Mirrors the pre-existing `_validate_config()` error path at line 145 of `send_mail.py`. The new check must run BEFORE the broad try/except so the original exception surfaces directly.
**Source:** 07.2-02-SUMMARY.md "send_mail ConfigurationError placement"; commit `ad74867`

### context_load KEEP comment lists all 4 valid policies, not 3

**Source:** 07.2-02-SUMMARY.md deviations; commit `0462e5e`

## Lessons

### BaseComponent lifecycle is non-obvious for direct unit testing of `_validate_config` and `_process`
`BaseComponent.__init__` sets `self._original_config = config` and `self.config = {}`. Only `execute()` Step 1 deepcopies `_original_config` into `self.config`. Tests calling `_validate_config()` or `_process()` directly (without going through `execute()`) see an empty `self.config` unless they manually populate it.

**Context:** All 7 new test files needed `comp.config = dict(config)` before calling `_validate_config()` / `_process()` in isolation. Without it, the deferred checks would always pass trivially because there's no key to check. Surfaced during Plan 01 Task 1 and confirmed in every subsequent plan task.
**Source:** 07.2-01-SUMMARY.md "Test helper detail (Rule 3 -- blocking issue, fixed inline)"; 07.2-02-SUMMARY.md "Test fixture pattern"

---

### Pre-resolution validation is fundamentally unsafe for any content-check
The original Phase 7.1 CR-06 contract (multi-char delimiter rejection) was over-strict for the same root reason as this phase's 6 confirmed bugs: `_validate_config` runs in Step 2, before context resolution in Step 3. Any content check (length, regex, int coercion, enum membership) on a config field that COULD hold `${context.X}` will spuriously reject valid configs.

**Context:** Rule 12 was added to `MANUAL_COMPONENT_AUTHORING.md` to lock this in. The pattern is now "key-presence and container-shape only in `_validate_config`; everything else in `_process`."
**Source:** 07.2-CONTEXT.md background; 07.2-PHASE-SUMMARY.md "Lessons Carried Forward"; quick task 260429-hc2 (CR-06 supersession)

---

### Naive placement of new checks inside existing `try/except` blocks can break exception-type contracts
A new `ConfigurationError` raise inside a block that ends `except Exception: raise ComponentExecutionError(...)` will be silently re-wrapped, breaking caller/test contracts. Discovered during send_mail Task 5.

**Context:** Always check the surrounding exception handler before adding a new raise. If the goal is for the original exception to surface to the caller, the new raise must run BEFORE the broad try/except.
**Source:** 07.2-02-SUMMARY.md "Auto-fixed Issues - Rule 1 - Bug"

---

### Plan-checker catches scope drift even when CONTEXT.md is detailed
Even with a 5-page CONTEXT.md listing exact field-by-field scope, the first iteration of Plan 01 silently widened file_input_excel from `header+first_column` to `header+first_column+footer+limit` and added a conditional ARCHIVE_FORMAT branch. Plan-checker iteration 1 surfaced both as blockers.

**Context:** Detailed CONTEXT.md is necessary but not sufficient. Plan-checker is the structural defense against unconscious scope expansion. Worth running on every non-trivial phase even when the planning input is well-locked.
**Source:** Plan-checker iteration 1 output (Blockers 1 + 2)

## Patterns

### Deferred-check pattern (Rule 12 enforcement)
Move content checks (int/float coercion, `len()`, regex, enum membership, file-exists, numeric range) from `_validate_config` (Step 2 of BaseComponent.execute()) to `_process` (Step 3, post `_resolve_expressions`). Same exception type (`ConfigurationError`), same message string, same caller-visible behavior -- only the timing of the check changes. `_validate_config` retains only key-presence and container-shape checks (`isinstance`, dict/list shape).

**When to use:** Any time a component config field can plausibly hold a `${context.X}` or `{{java}}` reference. New components in Phase 8+ must follow this pattern. Use the inline rationale comment template (see KEEP-rationale pattern below) when the field is closed-list and the check stays.
**Source:** 07.2-PHASE-SUMMARY.md "Patterns Established"; Rule 12 in `docs/v1/standards/MANUAL_COMPONENT_AUTHORING.md`

---

### Group B KEEP rationale comment template
For fields where the converter never emits context vars (closed-list enums, fields the converter explicitly emits as int via `_get_int`, fields the converter doesn't extract at all), keep the existing `_validate_config` check and add an inline rationale comment:

```
# Group B verdict (Phase 07.2): KEEP. <field> is <emitted-as-int|closed-list|never-extracted>
# by the converter at <converter file:line>; <isinstance | membership> check is shape-only,
# allowed by Rule 12.
```

**When to use:** Any time a `_validate_config` content check is reviewed against Rule 12 and judged safe. Future authors reading the code see WHY the check is allowed and do not "fix" it. Used in `file_input_raw`, `fixed_flow_input`, `unique_row`, `context_load`.
**Source:** 07.2-PHASE-SUMMARY.md "Patterns Established"; commits `807f213`, `c4155be`, `ca4f565`, `0462e5e`

---

### Test fixture pattern for direct `_validate_config` / `_process` calls
Tests calling `_validate_config()` or `_process()` in isolation (without invoking `execute()`) must populate `comp.config = dict(config)` before the call, mirroring `execute()` Step 1's `self.config = copy.deepcopy(self._original_config)`. Otherwise `self.config` is empty (`__init__` only sets `_original_config`).

**When to use:** Every unit test that exercises `_validate_config()` or `_process()` directly. Documented in test docstrings of all 7 new test files added by this phase.
**Source:** 07.2-01-SUMMARY.md "Test helper detail"; 07.2-02-SUMMARY.md "Test fixture pattern"

---

### Three-test pattern per Group A fix
For each component fix, add three tests proving:
1. `_validate_config` accepts a `${context.X}` literal (returns `[]` or does not raise)
2. `_process` resolves the context var correctly end-to-end (with a populated `ContextManager`)
3. `_process` raises the original exception type with the original message when the resolved value is invalid (preserves contract; just runs at the right time)

**When to use:** Any deferred-check pattern fix. Locks both the timing change AND the preserved behavior into regression tests. 37 tests across 5 Plan 01 components follow this template.
**Source:** 07.2-CONTEXT.md decision C; 07.2-01-SUMMARY.md test files

---

### Pinned baseline-vs-post pytest gate
Capture pre-phase pytest failure list at the pinned baseline commit; capture post-phase failure list at the post-phase HEAD; gate is `set(post_fails) - set(baseline_fails) == empty`. Allows the phase to land even when the suite has pre-existing unrelated failures (e.g. Java-bridge integration tests requiring a running JAR).

**When to use:** Any phase running against a non-green test suite baseline. Avoids the "must be 0 failures" gate that would block correct work because of unrelated pre-existing issues. Plan 04 used this for the closure verification.
**Source:** 07.2-VERIFICATION.md "Test Baseline vs Post-Phase"; 07.2-04-PLAN.md Task 1
