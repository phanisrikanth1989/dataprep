## Warnings

### WR-01: _find_quoted_ranges may read past end of string on trailing backslash

**File:** `src/v1/engine/components/transform/map.py:512`
**Issue:** In `_find_quoted_ranges`, when the parser encounters a backslash (`\\`) inside a quoted string, it skips two characters with `i += 2`. If the backslash is the very last character in the expression string (e.g., a malformed expression ending with `"hello\`), the inner `while` loop continues with `i` pointing past the string length. The next iteration's `expr[i]` check at line 511 or 514 would raise an `IndexError`. While this is an edge case (malformed expressions), this method is called on user-provided filter expressions that come from Talend job configs, which may contain unusual patterns.
**Fix:**
```python
# Line 511-512: Add bounds check before accessing expr[i] after skip
if expr[i] == '\\':
    i += 2
    if i >= len(expr):
        break
    continue
```

### WR-02: pd.isna() on Series in _substitute_row_refs may raise ambiguity error

**File:** `src/v1/engine/components/transform/map.py:583`
**Issue:** `_substitute_row_refs` calls `pd.isna(val)` where `val = row[col_name]`. While `row` is a `pd.Series` (single row from `iterrows()`), accessing a column by name normally returns a scalar. However, if the Series has duplicate index entries (which can happen after `pd.concat` of main_row and lookup_row at line 1022, where both may have identically-named columns), `row[col_name]` returns a Series rather than a scalar. Calling `pd.isna()` on a Series and then using the result in an `if` statement raises `ValueError: The truth value of a Series is ambiguous`. This would manifest when a main table and lookup table share column names that are not the join key (e.g., both have a `region` column and the combined Series from line 1022's `pd.concat` has duplicate `region` entries). In Phase 5.2's RELOAD_AT_EACH_ROW path, this only applies to the second lookup if there are multiple lookups with overlapping columns.
**Fix:**
```python
val = row[col_name]
# Guard against duplicate index entries from pd.concat
if isinstance(val, pd.Series):
    val = val.iloc[0]
if pd.isna(val):
    return "None"
```

### WR-03: Test mock uses eval() on untrusted-ish string input

**File:** `tests/v1/engine/components/transform/test_map.py:1802`
**Issue:** `_MockBridgeEscapeTest.execute_tmap_preprocessing` calls `eval(left_raw)` where `left_raw` comes from expression splitting. While this is test-only code (marked with `# noqa: S307`) and the input is controlled test data, `eval()` in test mocks can be a maintenance hazard if later copied or adapted. The pattern could be replaced with `ast.literal_eval` which only evaluates literal structures and is safe against code injection.
**Fix:**
```python
import ast
try:
    left_val = ast.literal_eval(left_raw)
except (ValueError, SyntaxError):
    results[expr_id] = np.array([False] * len(df))
    continue
```

### IN-01: Duplicate schema dict building logic

**File:** `src/v1/engine/components/transform/map.py:1191-1203` and `src/v1/engine/components/transform/map.py:1719-1731`
**Issue:** The schema dict construction logic (mapping pandas dtypes to string type identifiers like "int", "float", "str") is duplicated verbatim in `_evaluate_outputs_compiled` (lines 1191-1203) and `_evaluate_with_bridge` (lines 1719-1731). This is a classic DRY violation -- if a new type needs to be added (e.g., "date" handling), both locations must be updated in sync.
**Fix:** Extract to a private static method:
```python
@staticmethod
def _build_schema_dict(df: pd.DataFrame) -> dict[str, str]:
    schema = {}
    for col in df.columns:
        dtype = str(df[col].dtype)
        if "int" in dtype:
            schema[col] = "int"
        elif "float" in dtype:
            schema[col] = "float"
        elif "datetime" in dtype:
            schema[col] = "datetime"
        elif "bool" in dtype:
            schema[col] = "bool"
        else:
            schema[col] = "str"
    return schema
```

### IN-02: Test may be validating mock behavior rather than component logic

**File:** `tests/v1/engine/components/transform/test_map.py:1272-1342`
**Issue:** `test_reload_per_row_filter_numeric_comparison` uses `_MockBridgeNumeric` which parses `<=` expressions by splitting on the operator. The test's correctness depends on the mock correctly implementing the `<=` semantics. If the mock has a bug (e.g., wrong column lookup), the test passes but the actual Java bridge path is untested. This is mitigated by the integration test class `TestReloadAtEachRowIntegration` in `test_map_integration.py`, which uses a real Java bridge. The mock-based test still has value for CI without JVM, but the coupling between mock fidelity and test correctness is worth noting.
**Fix:** No code change needed. The existing `@pytest.mark.java` integration tests cover the real bridge path. Consider adding a comment to the mock-based tests noting they are supplementary to the integration tests.

### IN-04: _find_jar_path in integration tests uses subprocess without timeout

**File:** `tests/v1/engine/components/transform/test_map_integration.py:285`
**Issue:** `_find_jar_path` calls `subprocess.check_output(["git", "rev-parse", ...])` without a `timeout` parameter. While `git rev-parse` is fast, best practice for subprocess calls is to include a timeout to prevent hangs in edge cases (e.g., network-mounted git repos, corrupted git state).
**Fix:**
```python
common_dir = subprocess.check_output(
    ["git", "rev-parse", "--git-common-dir"],
    cwd=str(Path(__file__).resolve().parent),
    text=True,
    timeout=5,
).strip()
```
