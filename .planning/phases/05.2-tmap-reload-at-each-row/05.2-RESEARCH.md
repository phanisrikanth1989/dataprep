# Phase 05.2: tMap RELOAD_AT_EACH_ROW Fix - Research

**Researched:** 2026-04-15
**Domain:** Talend tMap RELOAD_AT_EACH_ROW lookup mode, per-row dynamic filtering
**Confidence:** HIGH

## Summary

This research covers the 4 bugs in `_join_reload_per_row` that prevent the RELOAD_AT_EACH_ROW lookup mode from matching Talend behavior. The critical finding comes from analyzing the Talaxie javajet source code (the actual code generation templates Talend uses). In Talend, RELOAD_AT_EACH_ROW **re-executes the lookup source subprocess** for each main row -- it does not re-filter a cached DataFrame. However, since our Python engine does not have subprocess re-execution (lookups are pre-loaded DataFrames), we must simulate the semantics by re-filtering the full lookup DataFrame per main row, which is what the current skeleton intends but fails to do correctly.

The Talaxie source also reveals that lookup expression filters in Talend are evaluated as **post-match filters** (after the hash lookup, not before), and that main row values are available directly as Java variables (e.g., `row1.region`) in the generated code -- not through globalMap indirection. For our engine, the filter expression evaluation must substitute main row values into the expression before evaluating it against each lookup row.

**Primary recommendation:** Fix all 4 bugs in-place within `_join_reload_per_row` using Python-side expression substitution for the per-row filter (no Java bridge round-trips per row). The filter should substitute main row values by replacing `{main_table}.{column}` references with actual values, then evaluate the expression against the lookup DataFrame in a single vectorized operation per main row.

## Talaxie Source Analysis (PRIMARY REFERENCE)

### How Talend Actually Implements RELOAD_AT_EACH_ROW

Source: `Talaxie/tdi-studio-se`, path `main/plugins/org.talend.designer.components.localprovider/components/tMap/` [VERIFIED: GitHub API direct file download]

#### Initialization (tMap_begin.inc.javajet, lines 155-181)

For RELOAD mode, Talend initializes the lookup hash table to **null**:
```java
// isOneOfReloadLookupMode = true
AdvancedMemoryLookup<lookupStruct> tHash_Lookup_tableName = null;
```
For LOAD_ONCE, it loads from globalMap (already populated by the upstream component). This means RELOAD lookups start empty -- they get populated fresh for each main row.

#### Per-Row Re-Execution (tMap_main.inc.javajet, lines 869-919)

For each main row, Talend does the following sequence:

1. **Set globalMap entries** (lines 884-891): If the lookup table has `globalMapKeysValues` configured, those key-value pairs are put into globalMap. The expressions reference main row columns directly (e.g., `row1.region`).
   ```java
   globalMap.put("region_param", row1.region);
   ```

2. **Re-execute the lookup subprocess** (line 894):
   ```java
   lookupSubProcessStartNodeProcess(globalMap);
   ```
   This re-runs the entire upstream subflow that feeds the lookup (e.g., a tFileInputDelimited or tDBInput with a WHERE clause parameterized by globalMap). The subprocess repopulates `tHash_Lookup_tableName` in globalMap.

3. **Fetch the fresh hash table** (lines 909-912):
   ```java
   tHash_Lookup_tableName = (AdvancedMemoryLookup<lookupStruct>)
       globalMap.get("tHash_Lookup_tableName");
   ```

4. **Perform hash lookup** (lines 917-919):
   ```java
   tHash_Lookup_tableName.initGet();
   tHash_Lookup_tableName.lookup(hashKey);
   ```

5. **Apply expression filter as post-match filter** (lines 1098-1130): After the hash lookup returns candidate rows, the expression filter is evaluated. If the filter evaluates to false, `fromLookup = null` and the row is treated as unmatched.
   ```java
   if (!rejectedInnerJoin && !(expressionFilter)) {
       fromLookup_tableName = null;
       tableName = tableName_Default;
       if (isInnerJoin) rejectedInnerJoin = true;
   }
   ```

### Mapping Talend Semantics to Python Engine

Talend re-executes the subprocess (e.g., re-queries a database with new parameters). Our engine cannot do that -- lookups are pre-loaded DataFrames. The correct simulation is:

1. **Keep the full unfiltered lookup** (don't pre-filter) -- matches Talend's "fresh data" semantic
2. **Set main row values in globalMap** per row -- matches Talend's globalMap.put() before subprocess
3. **Evaluate the filter expression per row** against the full lookup, with main row values substituted -- simulates the parameterized re-query
4. **Apply hash-key matching** on the filtered result -- matches Talend's hashKey.lookup()
5. **Expression filter is a post-match filter** on the lookup row -- if filter fails, treat as unmatched

This means the current skeleton's approach is fundamentally correct (iterate main rows, re-filter lookup per row), but it has 4 implementation bugs that break it.

### Bug 2 Fix: Per-Row Filter with Main Row Context (Core Fix)

**Recommended Approach: Python-side expression substitution + vectorized filter evaluation**

Based on the Talaxie analysis:
- Talend evaluates the filter as inline Java code where `row1.region` is a direct variable
- Our engine must substitute main row values into the expression string before evaluating

**Pattern: Create a `_apply_filter_per_row` method (new sibling to `_apply_filter`)**

```python
def _apply_filter_per_row(
    self,
    lookup_df: pd.DataFrame,
    filter_expr: str,
    lookup_name: str,

        lookup_df: Full unfiltered lookup DataFrame.
        filter_expr: Filter expression (may have {{java}} prefix).
        lookup_name: Name of the lookup table.

        main_name: Main table name (e.g., "row1").

    Returns:
        Filtered lookup DataFrame (rows matching the filter).
    """
    expr = self._strip_java_marker(filter_expr)

    # Substitute main row references: row1.col -> actual value
    resolved_expr = self._substitute_row_refs(
        expr, main_row, main_name
    )

    # Now evaluate the resolved expression against lookup rows
    # The expression now only contains lookup table references
    # Use _apply_filter with the resolved expression
    ...
```

**Expression substitution strategy:**

The filter expression `row1.region == row2.region` contains two types of references:
- Main table refs (`row1.region`) -- substitute with actual value from current main row
- Lookup table refs (`row2.region`) -- leave for per-row evaluation

The substitution uses `_ROW_REF_PATTERN` (already defined at line 34 of map.py) to find `table.column` references. For each match where `table == main_name`:
- Get the value from `main_row`
- Replace with a Python literal (string: `"value"`, numeric: `123`, None: `None`)

After substitution, `row1.region == row2.region` becomes `"US" == row2.region`, which can be evaluated as a simple column comparison on the lookup DataFrame.

**For simple comparisons** (the common case): After substitution, the expression reduces to `"value" == lookupTable.column` which can be evaluated as a vectorized pandas boolean mask:
```python
# After detecting pattern: literal == table.column
mask = lookup_df[col_name] == literal_value
filtered = lookup_df[mask]
```

**For complex expressions**: Fall back to Java bridge batch evaluation on the lookup DataFrame (one bridge call per main row, evaluating against all lookup rows simultaneously). This is O(n * bridge_calls) not O(n * m * bridge_calls).

**GlobalMap key format (D-07):** Use `{main_table_name}.{col}` format:
```python
# Correct format matching Talend semantics
self.global_map.put(f"{main_name}.{col}", main_row[col])
```
This is confirmed by Talaxie source where `row1.region` is a direct Java variable -- it uses the table name, not a generic "row" prefix. [VERIFIED: tMap_main.inc.javajet line 888 shows `globalMap.put(entry.getName(), entry.getExpression())` where expressions use table-qualified names]

**Performance assessment:**
- Substitution is pure string operations -- microsecond per row [ASSUMED]
- Simple comparison (==) resolved to vectorized pandas mask -- fast
- Complex expressions fall back to Java bridge -- one bridge call per main row (not per main-row x lookup-row)
- No per-row JVM round-trips for simple expressions (satisfies D-10)
- Java bridge fallback is one call per main row with full lookup DataFrame (batch, not per-cell)

**Confidence:** HIGH for the approach, MEDIUM for the exact implementation details

### Bug 3 Fix: Type-Aware Key Comparison

**Location:** `map.py` line 857 in `_join_reload_per_row`

Replace `str(main_val) != str(lookup_val)` with type-aware comparison. The `_auto_convert_join_keys` method in `_join_equality` (lines 1788-1856) already implements the pattern.

**Recommended approach:** Inline a type-aware comparator rather than calling `_auto_convert_join_keys` (which works on full DataFrames, not individual values):

```python
def _values_equal(self, a: Any, b: Any) -> bool:
    """Type-aware value comparison for join keys.

    Numeric types compared as numbers (int 1 == float 1.0).
    Otherwise compared as strings.
    Null/NaN never matches (handled separately by caller).
    """
    # Both numeric -> compare as float
    if isinstance(a, (int, float, np.integer, np.floating)) and \
       isinstance(b, (int, float, np.integer, np.floating)):
        return float(a) == float(b)
    # Otherwise string comparison
    return str(a) == str(b)
```

Then replace line 857:

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Type-aware comparison | Custom type coercion matrix | `_values_equal` helper with `isinstance` checks for numeric types + `np.integer`/`np.floating` | Edge cases with numpy types, NaN handling, pandas nullable types |
| Expression substitution | Regex-based string replacement | `_ROW_REF_PATTERN` (already defined at line 34) + `re.sub` with replacement function | Pattern already handles `table.column` detection correctly |
| Vectorized filter evaluation | Row-by-row Python loop over lookup rows | pandas boolean masking (`df[col] == value`) | Orders of magnitude faster for large lookups |

**Key insight:** The per-row loop is inherently O(n) over main rows, but the lookup filter within each iteration should be vectorized (pandas mask, not Python loop over lookup rows). This is where performance is preserved.

## Common Pitfalls

### Pitfall 1: Pre-Filter Leaks Into Reload Path
**What goes wrong:** The pre-filter at `_process` lines 282-288 reduces the lookup before the per-row loop even starts, making some valid matches invisible.
**Why it happens:** The pre-filter block runs for ALL lookup modes, and `lookup_mode` is read AFTER the filter block.
**How to avoid:** Read `lookup_mode` before the filter block. Guard with `lookup_mode != _RELOAD_AT_EACH_ROW`.
**Warning signs:** Tests that don't use filters pass, but filter-based tests fail or return too few rows.

### Pitfall 2: Expression Substitution Type Quoting
**What goes wrong:** Substituting a string value without quotes (`US` instead of `"US"`) causes the expression to be interpreted as a variable reference.
**Why it happens:** Main row values must be properly quoted as Python/Java literals when substituted into expressions.
**How to avoid:** String values wrapped in quotes, numeric values as-is, None as `null`/`None`.
**Warning signs:** Expressions like `"US" == row2.region` work but `US == row2.region` raises NameError.

### Pitfall 3: Pandas NaN Column Ordering
**What goes wrong:** When building a DataFrame from rows with inconsistent columns, pandas reorders columns alphabetically, breaking downstream column references.
**Why it happens:** `pd.DataFrame(result_rows)` infers columns from the union of all row keys, but order is dict-insertion-order dependent.
**How to avoid:** Explicitly set NaN-filled lookup columns on unmatched rows (Bug 4 fix). All rows have identical column sets.
**Warning signs:** Output DataFrame columns appear in wrong order, or downstream tMap outputs reference wrong columns.

### Pitfall 4: Java Bridge Per-Row Overhead
**What goes wrong:** Calling the Java bridge once per main row * per lookup row creates O(n*m) JVM round-trips, destroying performance.
**Why it happens:** Naively routing complex filter expressions through the bridge for each lookup row comparison.
**How to avoid:** For simple expressions (the vast majority), use Python-side substitution + pandas vectorization. For complex expressions, batch-evaluate once per main row against the full lookup DataFrame.
**Warning signs:** RELOAD_AT_EACH_ROW execution time increases by 10x+ compared to LOAD_ONCE.

### Pitfall 5: GlobalMap Key Format Mismatch
**What goes wrong:** Setting globalMap keys as `row.col` instead of `row1.col` means downstream components or expressions that reference `row1.col` in globalMap get None.
**Why it happens:** Current code uses `f"row.{col}"` instead of `f"{main_name}.{col}"`.
**How to avoid:** Use the main table name from config: `self.config["inputs"]["main"]["name"]`.
**Warning signs:** globalMap.get("row1.region") returns None even though it was set during the per-row loop.

### Expression Substitution Pattern (Recommended)

```python
# Source: Design based on Talaxie analysis and existing _ROW_REF_PATTERN
def _substitute_row_refs(
    self,
    expr: str,
    row: pd.Series,
    table_name: str,
) -> str:
    """Replace table.column references for a specific table with literal values.

    Args:
        expr: Expression string (e.g., "row1.region == row2.region").
        row: Row data (pd.Series) to substitute values from.
        table_name: Table name to match (e.g., "row1").

    Returns:
        Expression with matched references replaced by literals.
    """
    def _replace_ref(match):
        table, column = match.group(1), match.group(2)
        if table != table_name:
            return match.group(0)  # Not our table, leave as-is
        col_name = self._find_column_in_row(row, table, column)
        if col_name is None:
            return match.group(0)
        val = row[col_name]
        if pd.isna(val):
            return "None"
        if isinstance(val, str):
            # Escape quotes in string values
            escaped = val.replace("\\", "\\\\").replace('"', '\\"')
            return f'"{escaped}"'
        return str(val)

    return _ROW_REF_PATTERN.sub(_replace_ref, expr)
```

## Open Questions

1. **How to handle complex Java filter expressions in per-row context?**
   - What we know: Simple `row1.col == row2.col` comparisons can be resolved Python-side. The Java bridge can batch-evaluate expressions against a DataFrame.
   - What's unclear: What percentage of real jobs use complex filter expressions (e.g., `row1.amount > row2.threshold * 1.1`) in RELOAD_AT_EACH_ROW? Do any use Java method calls in the filter?
   - Recommendation: Implement Python-side substitution for simple expressions. For anything that still contains unresolved table references after substitution, fall back to `_apply_filter` which uses the Java bridge. This handles both cases. The bridge call is per-main-row (against full lookup DataFrame), not per-cell.

2. **Should `_apply_filter_per_row` be a new method or extend `_apply_filter`?**
   - What we know: `_apply_filter` has a clean interface (df, filter_expr, table_name, lookup_names). Adding a `main_row` parameter changes its signature.
   - What's unclear: Whether future features will also need row-context-aware filtering.
   - Recommendation: Create a new `_apply_filter_per_row` method that calls `_substitute_row_refs` then delegates to `_apply_filter`. Keeps existing method clean, single responsibility.
