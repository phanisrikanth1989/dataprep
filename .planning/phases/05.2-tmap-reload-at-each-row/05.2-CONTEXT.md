# Phase 5.2: tMap RELOAD_AT_EACH_ROW Fix - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix 4 bugs in `_join_reload_per_row` so the RELOAD_AT_EACH_ROW lookup mode matches Talend behavior. In Talend, this mode re-evaluates the lookup filter expression for each main row, with access to the current main row's values via globalMap. The current implementation has a working skeleton but per-row dynamic filtering is broken.

Scope: Fix the 4 documented bugs (double filtering, per-row filter resolution, string-cast keys, column misalignment). No new features, no other tMap changes.

</domain>

<decisions>
## Implementation Decisions

### Research Directive (MANDATORY)
- **D-01:** The research phase MUST study the Talaxie GitHub source (`Talaxie/tdi-studio-se`, path `main/plugins/org.talend.designer.components.localprovider/components/tMap/`) to understand exactly how Talend generates Java code for RELOAD_AT_EACH_ROW. Specifically: `tMap_begin.javajet` (lookup loading for reload mode), `tMap_main.javajet` (per-row filter evaluation, row context access), and `tMap_end.javajet` (cleanup).
- **D-02:** Implementation approach for Bug 2 (per-row filter evaluation) is deferred to research. The researcher must determine how Talend resolves main row values in per-row lookup filter expressions before any code is written.

### Bug 1: Double Filtering (HIGH)
- **D-03:** Guard the pre-filter at `_process` lines 282-288 with `if lookup_mode != _RELOAD_AT_EACH_ROW`. For RELOAD_AT_EACH_ROW, the full unfiltered lookup must reach the per-row loop. Straightforward fix.

### Bug 2: Per-Row Filter Can't Reference Main Row Values (HIGH -- Core Semantic Issue)
- **D-04:** Approach deferred to research (see D-01/D-02). The researcher will study the Talaxie javajet source and determine which approach matches Talend semantics.
- **D-05:** Three candidate approaches identified in handoff:
  1. Make `_apply_filter` globalMap-aware (add optional row context parameter)
  2. Substitute main row values into filter expression string before calling `_apply_filter`
  3. Use Java bridge for per-row filter evaluation (it already has globalMap access)
- **D-06:** Research decides whether refactoring `_apply_filter` is needed (adding row context parameter or creating a sibling method) vs. working around it inside the per-row loop. Refactoring is in scope IF the research shows it's necessary, but not speculative.
- **D-07:** GlobalMap key format must use `{main_table_name}.{col}` (e.g., `row1.customer_id`), not `row.{col}`. Research should confirm the exact key format Talend uses.

### Bug 3: String-Cast Key Comparison (MEDIUM)
- **D-08:** Replace `str(main_val) != str(lookup_val)` with type-aware comparison. Follow the `_auto_convert_join_keys` approach already used in `_join_equality`. Numeric types compared as numbers, otherwise as strings.

### Bug 4: Missing Lookup Columns on Unmatched Rows (MEDIUM)
- **D-09:** When LEFT_OUTER_JOIN has no match, build a combined Series with NaN-filled lookup columns before appending to result_rows. Ensures consistent column sets in the output DataFrame.

### Performance Constraint (NON-NEGOTIABLE)
- **D-10:** Per-row filter evaluation must NOT add per-row JVM round-trips or other overhead on top of the existing O(n*m) loop. If the Java bridge is used for filter evaluation, it must batch-evaluate (e.g., evaluate the filter for all lookup rows against one main row in a single bridge call), not call per-row-per-lookup-row.
- **D-11:** The existing performance warning for large datasets (>10K main x >10K lookup) stays. No additional performance degradation beyond what RELOAD_AT_EACH_ROW inherently requires.

### Backward Compatibility (NON-NEGOTIABLE)
- **D-12:** Existing RELOAD_AT_EACH_ROW functionality (basic key matching without filter expressions) must not regress. The 7 existing `TestReloadAtEachRow` tests must continue passing.
- **D-13:** All other tMap tests (LOAD_ONCE, join modes, expressions, etc.) must continue passing. Zero regressions.

### Test Strategy
- **D-14:** Claude's discretion on test depth. Guideline: bug-proving tests (the 4 from handoff) plus meaningful edge cases (multi-column keys, null main values, empty lookup after filter, large dataset perf guard). Estimated ~10-15 new tests.
- **D-15:** At least one `@pytest.mark.java` integration test that runs per-row filter evaluation through the real JVM bridge (not mocked). Mock-only tests gave false confidence before (per Phase 5.1 lesson).

### Claude's Discretion
- Internal method decomposition for the per-row filter fix (after research determines approach)
- Whether to add a separate `_apply_filter_per_row` method or extend existing `_apply_filter`
- Exact test count and edge case selection
- Whether to use `_auto_convert_join_keys` directly or inline the type comparison logic

</decisions>

## Existing Code Insights

### Fix Targets (All in map.py)
- `_process` lines 282-288: Pre-filter applied to ALL lookup modes including RELOAD_AT_EACH_ROW (Bug 1)
- `_join_reload_per_row` line 811: globalMap key format is `row.{col}` instead of `{table_name}.{col}` (Bug 2)
- `_apply_filter` line 380: No concept of current row context -- evaluates against DataFrame only (Bug 2)
- `_join_reload_per_row` line 857: `str(main_val) != str(lookup_val)` string-cast comparison (Bug 3)
- `_join_reload_per_row` line 872: Unmatched rows appended without lookup columns (Bug 4)

### Reusable Assets
- `_auto_convert_join_keys` in `_join_equality`: Type-aware key conversion already implemented. Same approach works for Bug 3.
- `_evaluate_with_bridge`: Batch expression evaluation via Java bridge. May be leveraged for per-row filter if research confirms batch approach.
- `_strip_java_marker`, `_is_simple_column_ref`, `_find_column`: Filter expression utilities already used by `_apply_filter`.

### Established Patterns
- Java bridge batch evaluation: Send DataFrame + expressions, get numpy arrays back. Avoids per-row JVM calls.
- GlobalMap sync: `self.global_map.put()` / `.get()` for inter-component state.
- Column prefixing: `{lookup_name}.{column}` for namespace isolation.
- Null key pre-filter: `pd.isna()` check before comparison (Phase 5 D-13).

### Integration Points
- `_join_reload_per_row` is called from `_process` at line 293 when `lookup_mode == _RELOAD_AT_EACH_ROW`
- Result feeds back into the main lookup processing loop (joined_df, rejects)
- GlobalMap variables set in per-row loop are visible to downstream components

</code_context>
