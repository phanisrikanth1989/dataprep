# Phase 5.2: tMap RELOAD_AT_EACH_ROW Fix -- Context Handoff

## What this phase is about

The `RELOAD_AT_EACH_ROW` lookup mode in tMap has a working skeleton but 4 bugs prevent it from matching Talend behavior. In Talend, this mode re-evaluates the lookup filter expression for each main row, with access to the current main row's values via globalMap. This enables dynamic per-row filtering that LOAD_ONCE cannot do.

## Current state (as of 2026-04-15)

- Implementation: `src/v1/engine/components/transform/map.py`, method `_join_reload_per_row` (line ~765)
- Routing: `_process` at line ~293 routes to `_join_reload_per_row` when `lookup_mode == "RELOAD_AT_EACH_ROW"`
- Tests: `tests/v1/engine/components/transform/test_map.py`, class `TestReloadAtEachRow` (7 tests, all pass)
- Test suite: 4114 passed, 23 skipped, 0 failed

## Bugs found (with exact locations)

### Bug 1: Double filtering (HIGH)

**Location:** `map.py` lines 282-296 in `_process`

The lookup filter is applied globally at lines 282-288 *before* the routing decision at line 291. Then `_join_reload_per_row` receives the already-filtered `lookup_df` and tries to re-filter at lines 814-821.

```python
# Line 282-288: This runs for ALL lookup modes, including RELOAD_AT_EACH_ROW
if (lookup_config.get("activate_filter")
        and lookup_config.get("filter")):
    lookup_df = self._apply_filter(
        lookup_df, lookup_config["filter"],
        lookup_name, joined_lookup_names
    )

# Line 291-296: Then routes to per-row handler with already-filtered data
if lookup_mode == _RELOAD_AT_EACH_ROW:
    joined_df, rejects = self._join_reload_per_row(
        joined_df, lookup_df, lookup_config  # <-- lookup_df already filtered!
    )
```

**Talend behavior:** For RELOAD_AT_EACH_ROW, the filter should NOT be pre-applied. The full unfiltered lookup must be available to the per-row loop.

**Fix direction:** Guard the pre-filter with `if lookup_mode != _RELOAD_AT_EACH_ROW`.

### Bug 2: Per-row filter can't reference main row values (HIGH -- core semantic issue)

**Location:** `map.py` lines 807-821 in `_join_reload_per_row`

The per-row loop sets globalMap at line 811:
```python
self.global_map.put(f"row.{col}", main_row[col])
```

Two problems:
1. Key format is `row.{col}` instead of `{main_name}.{col}` (e.g., `row1.customer_id`). Talend uses the table name prefix.
2. `_apply_filter` (called at line 817) does NOT resolve globalMap references. It handles:
   - Simple column refs via `_is_simple_column_ref` (e.g., `row2.active`)
   - Complex expressions via Java bridge `_evaluate_with_bridge`
   
   Neither path substitutes globalMap values into the filter expression. So a filter like `row1.region == customers.region` won't see the current main row's region value.

**Talend behavior:** In RELOAD_AT_EACH_ROW mode, the lookup filter expression can reference main row values. Talend evaluates the filter as a Java expression with the current row context available.

**Fix direction:** This is the hardest fix. Options:
- Make `_apply_filter` globalMap-aware for RELOAD_AT_EACH_ROW context
- Build a per-row filter evaluation that substitutes main row values before calling `_apply_filter`
- Use the Java bridge for filter evaluation when available (it already has globalMap access)

### Bug 3: String-cast key comparison (MEDIUM)

**Location:** `map.py` line 857 in `_join_reload_per_row`

```python
if str(main_val) != str(lookup_val):
```

This converts everything to string for comparison. Fails for:
- `int(1)` vs `float(1.0)` -> `"1"` != `"1.0"` (should match)
- `True` vs `"true"` -> `"True"` != `"true"` (edge case)

**Talend behavior:** Talend uses Java's `.equals()` with auto-boxing, which handles int/long/float equivalence.

**Fix direction:** Type-aware comparator. Check if both are numeric -> compare as numbers. Otherwise compare as strings. Similar to the `_auto_convert_join_keys` approach used in `_join_equality`.

### Bug 4: Missing lookup columns on unmatched rows (MEDIUM)

**Location:** `map.py` lines 868-872 in `_join_reload_per_row`

```python
if not matched:
    if join_mode == "INNER_JOIN":
        reject_rows.append(main_row)
    else:
        result_rows.append(main_row)  # <-- only main columns, no lookup NaNs
```

When LEFT_OUTER_JOIN has no match, `main_row` is appended without NaN-filled lookup columns. At line 876, `pd.DataFrame(result_rows)` builds a DataFrame from rows with inconsistent column sets.

pandas currently auto-fills NaN for missing columns, so this "works" but relies on undocumented behavior and column ordering can be wrong.

**Talend behavior:** Unmatched rows have all lookup columns set to null.

**Fix direction:** Build a combined Series with NaN-filled lookup columns:
```python
# Create NaN-padded row with all expected columns
combined = main_row.copy()
for col in lookup_prefixed_cols:
    combined[col] = None
result_rows.append(combined)
```

**None of these tests exercise the actual RELOAD_AT_EACH_ROW differentiator: per-row dynamic filtering.** They all pass with the current broken implementation because they don't use filter expressions.

## Tests needed to expose the bugs

1. **Per-row filter test:** Main table has rows with different `region` values. Lookup has rows with `region` column. Filter: `row1.region == customers.region`. Each main row should only match lookups with the same region. Currently ALL lookups match every row because the filter doesn't resolve `row1.region`.

2. **Double-filter test:** Set `activate_filter: true` with a filter that eliminates some lookup rows. Verify RELOAD_AT_EACH_ROW still has access to the full lookup (not the pre-filtered subset).

3. **Mixed-type key test:** Main has `int(1)`, lookup has `float(1.0)`. Verify they match.

4. **Unmatched column alignment test:** LEFT_OUTER_JOIN with unmatched rows. Verify the output DataFrame has consistent columns and the lookup columns are NaN (not missing) for unmatched rows.
