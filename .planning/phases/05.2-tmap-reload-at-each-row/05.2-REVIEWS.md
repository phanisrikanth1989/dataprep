# Cross-AI Plan Review -- Phase 05.2

## Gemini Review

### 1. Summary
The implementation plans for Phase 05.2 are technically sound, highly targeted, and demonstrate a deep understanding of the discrepancies between the current Python engine and Talend's `tMap` behavior. By simulating Talend's per-row re-execution through Python-side expression substitution and re-filtering of cached DataFrames, the plan effectively bypasses the performance bottleneck of JVM round-trips while achieving functional parity. The two-wave approach--prioritizing core logic fixes followed by exhaustive edge-case testing--is appropriate for a component as critical as the `tMap` join logic.

### 2. Strengths
- **Surgical Logic Fixes:** The plan directly addresses the four confirmed bugs with specific code-level strategies (e.g., guarding the pre-filter, fixing the `globalMap` key format).
- **Performance Awareness:** Explicitly avoids per-row JVM calls (D-10), opting for a regex-based expression substitution which is significantly more efficient for high-volume ETL processing.
- **Type-Safe Comparisons:** Moving away from `str()` casting to a float-promoted numeric comparison (`_values_equal`) prevents common ETL "false negatives" where `1.0 != "1"`.
- **Comprehensive Testing Suite:** The inclusion of 10-15 unit tests covering specific bug triggers, combined with a `@pytest.mark.java` integration test, provides high confidence in the fix and guards against regressions.
- **Contextual Integrity:** The use of `{main_table_name}.{col}` for `globalMap` keys correctly simulates the Java scope provided by Talend's `javajet` generated code.

### 3. Concerns
- **Regex Substitution Robustness (MEDIUM):** Using regex (`_ROW_REF_PATTERN`) to substitute `table.column` references in expressions can be brittle. If a column name appears inside a string literal within the expression (e.g., `row1.label == "Look at row1.label"`), the regex might incorrectly replace the content of the string.
- **NaN/None Handling in Expressions (MEDIUM):** When substituting `None` or `np.nan` into an expression string for evaluation, the plan must ensure the resulting Python expression is valid (e.g., substituting `None` into `val > 5` might throw a `TypeError` during evaluation).
- **Numeric Precision (LOW):** Comparing all numerics as `float` is generally safe, but for very large 64-bit integers or decimals where precision is critical, float promotion can occasionally lead to rounding errors.
- **Dependency on Internal State (LOW):** The plan assumes `_apply_filter` is already robust enough to handle the results of the per-row substitution.

### 4. Suggestions
- **Refine Regex Logic:** Enhance `_substitute_row_refs` to be aware of quotes/literals if possible, or at least document the limitation that column names shouldn't conflict with common string constants in the job.
- **Explicit Null-Safe Evaluation:** In `_apply_filter_per_row`, ensure that the substitution of `None` values includes a check to prevent evaluation crashes.
- **Value Equalization Edge Case:** In `_values_equal`, consider handling the case where one side is a string and the other is numeric by attempting a safe cast of the string to numeric before comparison.
- **Verify Column Alignment:** For Bug 4 (NaN-fill), explicitly verify that the column order of the "dummy" NaN row matches the expected schema of the lookup table.

### 5. Risk Assessment
**Risk Level: LOW**

The plan is well-grounded in research and targets a localized set of bugs in `map.py`. The "simulation" approach is significantly lower risk than attempting to modify the Java bridge for per-row execution. The primary risk is the complexity of the expression substitution (Bug 2), but the planned regression testing and the specific unit tests for numeric/null/quoted values mitigate this effectively.
