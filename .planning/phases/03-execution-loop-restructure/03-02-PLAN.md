    **SubjobPlan dataclass:**
    ```python
    @dataclass
    class SubjobPlan:
        subjob_id: str
        component_ids: list[str]        # Topologically sorted execution order
        component_set: frozenset[str]    # For O(1) membership checks
    ```

    **StreamingMetadata dataclass:**
    ```python
    @dataclass
    class StreamingMetadata:
        component_id: str
        requires_full_data: bool = False   # True for aggregate, sort
        streamable: bool = True            # True for filter, map, most transforms
    ```

    Types that require full data (for streaming metadata):
    - `AggregateRow`, `tAggregateRow`, `UniqueRow`, `tUniqueRow`, `tUniqRow`
    - `SortRow`, `tSortRow`
    - `AggregateSortedRow`, `tAggregateSortedRow`

    **TriggerEdge dataclass:**
    ```python
    @dataclass
    class TriggerEdge:
        from_component: str
        to_component: str
        trigger_type: str
        from_subjob: str | None = None
        to_subjob: str | None = None
        condition: str | None = None
    ```

    **ExecutionPlan class:**

    `__init__(self, components: list[dict], flows: list[dict], triggers: list[dict], subjobs: dict[str, list[str]] | None = None)`:

    1. Build `_component_to_subjob` mapping. If `subjobs` dict is provided (preferred), build from it. If not, auto-detect subjobs using connected-components on the flow graph (BFS, same logic as current `_find_connected_components`). **When auto-detecting, log at INFO level: "No subjobs dict in job config, auto-detecting subjob boundaries from flow graph"** (addresses review concern: auto-detection logging).
    2. Build SubjobPlan for each subjob:
       - Filter flows to only those whose `from` AND `to` are both in this subjob's component set.
       - Use `graphlib.TopologicalSorter` to compute execution order. Add all components as nodes, add flow edges as dependencies (`to` depends on `from`).
       - Call `static_order()`. Catch `CycleError` and re-raise as `ConfigurationError` with cycle info.
    3. Build trigger edge list from triggers config. Map each trigger's from/to component to their subjob. Normalize trigger keys: support both `from`/`to` and `from_component`/`to_component`.
    4. Compute `_initial_subjobs`: subjobs whose ID does not appear as `to_subjob` in any trigger edge.
    5. Build `_streaming_metadata` dict: for each component, check its type against the requires_full_data list.
    6. Store `_runif_target_subjobs`: set of subjob IDs targeted by RunIf triggers.
    7. **Build `_cross_subjob_flows`: list of flow dicts where the `from` component and `to` component are in different subjobs. Also build `_flow_consumers`: dict mapping flow_name -> set of consumer component IDs.** (Addresses review concern: cross-subjob flow lifecycle).

    `validate(self) -> None`:
    - Run reachability analysis from initial subjobs, following trigger edges.
    - Unreachable subjobs (not reachable AND not in `_runif_target_subjobs`) -> raise `ConfigurationError` listing the unreachable subjobs.
    - Per D-08: RunIf targets are conditionally reachable, do NOT flag them.

    Properties and methods:
    - `initial_subjobs -> list[str]`: subjob IDs not targeted by any trigger
    - `get_subjob_plan(subjob_id: str) -> SubjobPlan`: get pre-computed plan
    - `get_triggered_subjobs(trigger_type: str, source_component: str) -> list[TriggerEdge]`: get trigger edges from a source component of a given type
    - `get_all_trigger_edges_from_subjob(subjob_id: str) -> list[TriggerEdge]`: get all trigger edges originating from components in a subjob
    - `get_streaming_metadata(component_id: str) -> StreamingMetadata`: get streaming metadata
    - `component_to_subjob -> dict[str, str]`: mapping for lookups
    - `all_subjob_ids -> list[str]`: all subjob IDs in execution order (initial first, then by trigger chain)
    - **`get_cross_subjob_flows() -> list[dict]`**: returns flows where from-component and to-component are in different subjobs. Used by OutputRouter to avoid premature clearing. (Addresses review concern: cross-subjob flow lifecycle)
    - **`get_flow_consumers(flow_name: str) -> set[str]`**: returns set of component IDs that consume a given flow. Used to check if all consumers are complete before clearing.

    Important implementation details:
    - Trigger `from`/`to` fields reference COMPONENTS, not subjobs. The plan maps these to their containing subjobs.
    - OnSubjobOk triggers: `from_component` is the `is_subjob_start` component of the source subjob (the component Talend marks as the subjob start). The `to_component` is typically the start of the target subjob.
    - OnComponentOk triggers: fire after a specific component completes. The target may be in the same or different subjob.
    - For auto-detection fallback: use BFS from each unvisited component, following flows bidirectionally (same as current engine.py `_find_connected_components`).
