# Phase 05.4 Plan 02: Rewrite _route_inner_join_rejects with _NullRow Injection Summary

**Inner-join-reject column expressions are now evaluated per row against the reject output's own expression list -- via the shared _evaluate_output_columns_py helper -- with a new _NullRow sentinel that raises AttributeError when the reject expression dereferences a failing lookup. Hard-coded literals, renamed simple refs, and {{java}} reject columns no longer collapse to None.**

## Accomplishments

- Added `_NullRow` class to `map.py` (line 128) -- a read-only sentinel that raises `AttributeError` with an ASCII message (`lookup was not matched; column '<name>' unavailable on reject row`) on any attribute access.  Sits next to `_NullNamespace` with identical shape but opposite semantics.
- Extended `_build_namespace` and `_evaluate_output_columns_py` with an optional `failed_lookups: set[str] | None` kwarg.  Defaults to `None`/`{}` so all existing call-sites are unaffected.  Lookup names in the set get a `_NullRow()` binding; everything else uses the existing `_NullNamespace` / `SimpleNamespace` rules.
- Rewrote `_route_inner_join_rejects` (`map.py:2749-2840`) to:
  1. Loop over `inner_join_reject_dfs` per failing lookup (preserves per-row failing-lookup identity, D-05).
  2. Call `_evaluate_output_columns_py` with `failed_lookups={failing_lookup}` so only the failing lookup's namespace raises -- previously matched lookups retain their data bindings.
  3. Pass `var_columns={}` (variable values are not defined for the unmatched reject row source).
  4. Build the result via the helper's single-allocation `pd.DataFrame(...)` then one outer `pd.concat` across failing lookups -- replaces the legacy per-column `reject_df[col_name] = ...` loop that emitted `PerformanceWarning: DataFrame is highly fragmented` (R7).
  5. Preserve the empty fast path (`if not inner_join_reject_dfs: return`).
  6. Use `%`-style logger formatting (cheaper, no f-string allocation).
- Added 5 RED-first unit tests (`TestInnerJoinRejectPy`) covering: `_NullRow` import, `_NullRow` raise contract (ASCII message + exact phrases), literal-column evaluation, empty fast path, and the `_NullRow` injection contract (die_on_error=True raises `ComponentExecutionError`; die_on_error=False logs the `_NullRow` "was not matched" phrase as a warning).
- Verified zero behavioural regression: all 307 `test_map.py` tests pass; broader 1767-test transform unit suite also green.

## Decisions Made

- **Per-failing-lookup loop instead of single pd.concat.**  The legacy code concatenated all reject DataFrames into `all_rejects` then did one column-copy pass.  The rewrite loops per failing lookup so the per-row identity of which lookup failed is preserved -- required for correct `_NullRow` injection (D-04 says only the *failing* lookup raises; previously matched lookups, per D-05, keep their data bindings).  Trade-off: one extra `pd.concat` at the end if there are multiple failing lookups, which is cheap relative to the per-row evaluation cost.
- **`failed_lookups` as optional kwarg, not separate "reject-only" helper.**  Threading the parameter through `_evaluate_output_columns_py` and `_build_namespace` with a `None` default keeps all existing call-sites untouched and lets Plans 03/04 reuse the same mechanism if they need lookup-specific error semantics (filter-reject and catch-output-reject row sources are complete -- they generally won't need this -- but the option exists at no cost).  Alternative considered: a thin "reject variant" of the helper that wraps with the parameter set.  Rejected as redundant -- the kwarg approach has the same shape with fewer call-sites to maintain.
- **`var_columns={}` for reject routing.**  Per-row variable values were computed against the matched `joined_df` (in `_evaluate_variables_py`) and are positionally indexed against that frame; the reject row source has different rows (different indices), so the variable values do not apply.  Reject expressions referencing `Var.*` will resolve to `None` via `_VarBag.__getattr__` (which returns `None` for undefined attributes when the bag was never populated).  This matches Talaxie's `tMap_main.inc.javajet` behaviour where the reject-output Groovy method receives a freshly empty `Var` map.

## Deviations from Plan

- **Test 5 die_on_error=False scenario adapted to existing `_eval_expr` semantics.**  The plan's Task 1 Test 5 wording said "the reject row is dropped (result['rej'] is empty or omits that row) and an ERROR-level log line is emitted" -- but the existing `_eval_expr` (phase 05.3 plan 05 deliverable) catches per-column exceptions and substitutes `None` with a `logger.warning` call, it does NOT drop the row.  Adding row-dropping semantics to `_eval_expr` would be an architectural change well outside this plan's scope (it would affect every Python-eval call site, not just reject routing).
  - **Resolution:** Test 5 asserts that under `die_on_error=False` the `row2_val` column resolves to `None` (AttributeError from `_NullRow.name` caught) AND that the captured WARNING log line text contains the `_NullRow` contract phrase `"was not matched"`.  This is sufficient to PIN the D-04 contract because: (a) the legacy `_NullNamespace` would return `None` *without* any warning text mentioning "was not matched" (it returns `None` silently with no log), so a passing assertion proves `_NullRow` was injected, not `_NullNamespace`.  (b) The `die_on_error=True` branch of Test 5 still asserts `ComponentExecutionError` is raised, pinning the hard-error path.
  - **Impact:** R5 (null-lookup expression semantics) is **partially** completed by this plan: the hard-error path is fully wired (die_on_error=True raises); the soft-error path logs a warning rather than dropping the row.  The "row dropped" semantic from SPEC R5 / D-04 is a separate concern that should be addressed in a follow-up plan or revisit if it becomes a production requirement.  Marked `requirements-partial: [R5]` in frontmatter.
- **No deviation from R1, R7, R4.**  R1 (per-row expression evaluation) and R7 (single-allocation result) are fully delivered.  R4 (shared evaluator) was Plan 01's deliverable and is honoured here -- this plan calls the existing helper, not a new one.

## Issues Encountered

- **`comp.config` is `{}` until `execute()` runs.**  `BaseComponent.__init__` stores the user config into `_original_config` and leaves `self.config = {}`; `execute()` copies `_original_config -> config` at step 1.  Direct method-level tests (e.g. `comp._route_inner_join_rejects(...)`) need `comp.config` populated to access `self.config["inputs"]["main"]["name"]`.  Fix: the test helper `_make_map_comp_for_reject` calls `comp.config = copy.deepcopy(comp._original_config)` after construction.  This mirrors what `execute()` does at step 1; no production code change needed.
- **Confirmed the plan's `Plan 02 RED gate` matched reality.**  RED run produced 4 of 5 failures: the empty-fast-path test passed because the legacy implementation already had the same early-return guard.  This is documented in the RED commit message.

## Next Phase Readiness
