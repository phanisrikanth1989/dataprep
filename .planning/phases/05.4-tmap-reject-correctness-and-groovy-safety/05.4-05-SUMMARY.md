# Phase 05.4 Plan 05: Groovy safety -- `_groovy_escape_expression` + 10-site retrofit Summary

**One-liner:** Tokenises Talend Java expressions to escape `$` -> `\$` inside double-quoted string literals before any Groovy parse (compiled-script or bridge-eval), preventing GString interpolation crashes; closes R6 with the audit doc as the locked decision record.

---

## What changed

### 1. New helper `Map._groovy_escape_expression`

Location: `src/v1/engine/components/transform/map.py:3200` (placed immediately after `_strip_java_marker`).

Approach: single-pass state-machine tokenizer with one boolean state (`in_string`). Inside a double-quoted region:

- `\` followed by any next character is consumed as a two-character unit (preserves `\"` and `\\` so they cannot mis-detect the closing quote)
- `"` closes the region
- `$` is rewritten to `\$`
- everything else is passed through unchanged

Outside a double-quoted region, every character (including `$` as identifier prefix) is passed through unchanged. The output is safe to interpolate into a Groovy source line OR to submit directly as a Groovy script body.

## Audit-document summary (what the audit decided)

**Escaped (in-string only):** `$` -> `\$` (prevents Groovy GString runtime interpolation of `$identifier` and `${expr}` once the script reaches `GroovyShell.parse`).

**Left alone (intentional pass-through):** `$` in identifier position (legal in Java AND Groovy); standard backslash escapes (`\n`, `\t`, `\r`, `\\`, `\"`); single-quoted character literals (Talend column schemas don't define `char`, only `String`); triple-quoted strings (not in Talend's Java emission surface); dollar-slashy strings (not legal Java); Unicode escapes (same in both); Groovy-only keywords used as Java identifiers (Talend UI constrains these out); statement separators; numeric suffixes (`123L`, `1.5f`, `1.0d`); BigDecimal/BigInteger `G` suffix (not legal Java -- cannot appear).

**Out of scope of this helper:** lambdas, annotations, pattern bindings, type parameters, anonymous classes -- none can appear in Talend tMap expression text, so the audit explicitly does not inspect them.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Critical correctness] Retrofitted 4 additional Groovy-embedding sites beyond the 6 named by the plan**

- **Found during:** Task 2 implementation (while reading the bridge code to identify "sites 4-6")
- **Issue:** Plan paragraph at lines 258-263 says: "Many `_strip_java_marker` calls in the join methods send expressions to the PYTHON side (bridge_eval), not to the compiled Groovy script. Only embed sites that interpolate expression text directly into Groovy script lines need the escape." This is mechanically incorrect. `_bridge_eval` calls `_evaluate_with_bridge` which calls `JavaBridge.executeTMapPreprocessing`. The Java side iterates `for entry in expressions: compileShell.parse(entry.value)` and runs the resulting Script per row. The expression IS the Groovy script body. So every `_bridge_eval` / `_evaluate_with_bridge` call site that forwards a Java expression IS a Groovy-embedding site and is vulnerable to GString interpolation crashes from in-string `$`.
- **Fix:** Audited every `_strip_java_marker` call in `map.py`. Classified each as (a) reaches `GroovyShell.parse` (compiled-script or bridge-eval) -> retrofit, or (b) pure-Python downstream (`_eval_expr`, `ContextManager.resolve_string`, `_SIMPLE_COLUMN_RE` matching, locality classification) -> no retrofit. The plan's six-site list captured 3 compiled-script sites + 3 bridge-eval join-key sites. The audit added 4 more bridge-eval sites: `_apply_filter` (main/lookup filter), `_chunked_cross_product` (match expression), `_evaluate_variables` complex branch, and `_apply_output_filter`. Total: 10 sites retrofitted.
- **Files modified:** `src/v1/engine/components/transform/map.py`
- **Verification:** All 302 `test_map.py` tests pass; `grep -c "_groovy_escape_expression"` = 11 (1 def + 10 calls), exceeding the plan's `>= 7` minimum.
- **Why this is Rule 2 not a scope change:** the original plan demands GString-safety for the Groovy-embedding surface. Leaving 4 of 10 sites unprotected would defeat the helper's purpose for production Talend jobs that route through the cross-table or per-row computed-key paths. The audit doc was updated in Task 1 to enumerate the full 10-site surface so this is also reflected in the locked decision record.
