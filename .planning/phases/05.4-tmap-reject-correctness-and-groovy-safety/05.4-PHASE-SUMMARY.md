# Phase 05.4: tMap Reject Correctness and Groovy Safety -- Phase Summary

## Phase Goal

Restore tMap reject-routing fidelity to Talend semantics across all three reject mechanisms (`inner_join_reject`, `is_reject` filter-reject, `catch_output_reject`), perform a full Groovy-safety audit of expression embedding in the compiled-script generator, and eliminate the pandas `PerformanceWarning: DataFrame is highly fragmented` emitted by reject routing.

**Root cause:** Commit `9ab8120` (Phase 5 "rewrite from scratch") lost reject-routing fidelity at three sites and embeds Talend expressions raw into Groovy script lines without escaping Groovy-special characters. The `origin/main` Map (pre-restructure) and Talaxie's canonical tMap codegen template both got reject routing right; the current `feature/engine-restructure` branch was the outlier.

## What Was Delivered

### Three reject routing sites rewritten

| Site                                                                                    | Plan      | Change                                                                                                                                                |
| --------------------------------------------------------------------------------------- | --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `_route_inner_join_rejects` (formerly L2583-2626)                                       | 05.4-02   | Calls `_evaluate_output_columns_py` per unmatched-main row; `_NullRow` sentinel binds the failing-lookup row; single-allocation DataFrame (R1, R5, R7) |
| `is_reject` filter-reject routing inside `_evaluate_outputs_py` + `_apply_output_filter` | 05.4-03   | Evaluates the **reject** output's column expressions (not the source output's) against filter-failed rows; reject schema wins (R2, R7)                 |
| `_route_catch_output_rejects` (formerly L2542-2575)                                     | 05.4-04   | Iterates the catch output's user columns; framework reserves `errorMessage` / `errorStackTrace`; user expressions for those reserved names are ignored (R3, R6, R7) |

### `_evaluate_output_columns_py` shared helper

Introduced in Plan 05.4-01. Signature: `_evaluate_output_columns_py(row_source, output_cfg, var_columns, main_name, lookup_names) -> pd.DataFrame`. Builds the per-row namespace, evaluates filter and column expressions via `_eval_expr`, collects rows into a list, materialises via `pd.DataFrame.from_records(rows, columns=col_names)` (R4, R7 single-allocation). Empty-rejects fast path returns an empty DataFrame with the declared schema. Used by all three reject routing sites and the main-output path.

### `_NullRow` sentinel

Introduced in Plan 05.4-02. Module-level singleton bound into the reject row namespace for the failing lookup. Accessing any attribute raises `AttributeError` with ASCII-only message `lookup '<name>' was not matched; column '<col>' unavailable on reject row`. Propagates through existing `die_on_error` / `catch_output_reject` machinery (R5).

### `_groovy_escape_expression` helper + retrofitted embed sites

Plan 05.4-05 added the helper. Tokenises the Java expression to identify string-literal regions (text between unescaped `"` characters, respecting Java escape rules), and escapes `$` -> `\$` inside string literals only. Identifier-position `$` is left alone (both Java and Groovy permit it).

Ten embed sites in `_build_compiled_script` were retrofitted to call `_groovy_escape_expression` after `_strip_java_marker` and before f-string interpolation: output column expressions, output filters, variable expressions, join key expressions (main-side and lookup-side computed keys), lookup filters, main filter, and reserved-column placeholders for catch_output_reject (R6).

### Per-reject-output method emission in compiled script

Plan 05.4-06 extended `_build_compiled_script` to emit `evalOutput_<reject_name>(int i, RowWrapper main, RowWrapper lk1, ..., Map<String,Object> Var)` methods alongside the existing non-reject `evalOutput_<name>` methods (Talaxie tMap_main.inc.javajet parity). Dispatch uses Option A (D-09): the compiled script is invoked twice with a `rejectMode` boolean -- once with `joined_df` for matched rows (`rejectMode=false`, calls non-reject evalOutputs) and once with the rejects DataFrame (`rejectMode=true`, calls reject evalOutputs). No marker column in the unified input.
