# Phase 05.4 Plan 04: Catch-Output-Reject Column-Expression Evaluation Summary

**Rewrote `_route_catch_output_rejects` to evaluate each catch output's user-defined column expressions per row via the shared `_evaluate_output_columns_py` helper, with framework-reserved `errorMessage` / `errorStackTrace` column values overwriting any user expression (D-06, Talaxie tMap_main.inc.javajet:1925-1988 parity).**

## Accomplishments

- **D-06 reserved-column policy implemented.** `errorMessage` and `errorStackTrace` are filtered out of the user-expression eval pass; framework values are assigned afterwards on the materialised DataFrame. If a user declares either name with a custom expression, the framework value WINS (matches Talaxie `tMap_main.inc.javajet:1925-1988`).
- **User-defined catch columns now evaluated.** The legacy `error_df.copy()` + literal `errorMessage` append is replaced by `self._evaluate_output_columns_py(error_df, eval_cfg, {}, main_name, lookup_names)` -- so columns like `(original_id=row1.id, custom_tag='"ERR"')` produce the expected values instead of being silently dropped.
- **Single-allocation guarantee preserved (R7).** `user_df` comes from `_evaluate_output_columns_py` as a single materialised frame; the reserved-column overwrite is two scalar column-level writes on the already-allocated DataFrame -- not inside a per-row loop. The old fragmenting pattern (`output_df["errorMessage"] = ...` after `error_df.copy()`) is gone.
- **Dual payload acceptance.** `__errors__` is read as either:
  - `pd.DataFrame` (preferred / future-wired): rows align with errored input rows, and the bridge may carry `errorMessage` / `errorStackTrace` columns directly which seed `per_row_messages` / `per_row_traces`.
  - `dict` (current Groovy emission shape): `count`, `indices`, `messages` -- `messages` (a row-index -> str dict) seeds `per_row_messages`. Row data is unavailable in this shape so user expressions referencing `row1.*` resolve to `None` via `_NullNamespace`.
- **Empty / missing payload fast paths.** `error_key not in raw_result`, `error_payload is None`, empty `pd.DataFrame`, and empty `messages` dict all short-circuit without modifying `result`.

## Decisions Made

- **`errorMessage` only populated when declared.** Under D-06, the framework writes `errorMessage` only if the user lists that column in `output_cfg["columns"]`. The legacy code unconditionally appended an `errorMessage` column to the catch output even when it was not in the schema. That was incorrect (the schema is the contract). The legacy test pinning this was updated to declare the column explicitly.
- **Result column ordering restored via `reindex(columns=col_names)`.** `_evaluate_output_columns_py` returns non-reserved columns in their declared order; reserved columns are then assigned (appending them to the right). A final `reindex` makes the result frame match the declared `output_cfg` schema exactly -- otherwise a config declaring `(errorMessage, original_id)` would yield `(original_id, errorMessage)`.
- **Dual payload shape.** The bridge currently strips `__errors__` from the Arrow output (`JavaBridge.convertTMapOutputsToArrow:846`). The rewrite handles the dict shape (`count`, `indices`, `messages`) so the routing function does the right thing today, and pre-handles the DataFrame shape so a future bridge change that wires `__errors__` back can drop in without re-touching this function. CONTEXT.md called this out: "the bridge does not currently produce these keys; use a default value. Document any discrepancy in SUMMARY."
- **`var_columns={}` for error rows.** Variable values are computed against the matched DataFrame and are not defined for error rows. References to `Var.X` on error rows resolve to `None` via `_VarBag.__getattr__` semantics. Same convention as `_route_inner_join_rejects` (phase 05.4-02).

## Issues Encountered

- **Bridge does not populate `__errors__` back to Python.** `JavaBridge.convertTMapOutputsToArrow:846` explicitly skips the `__errors__` key during Arrow serialisation, so today `raw_result["__errors__"]` is never set on the Python side from the compiled path. The rewrite is wired-ready: if a future plan unpacks `__errors__` as a DataFrame on the Python side, this function consumes it without further edits. The live-bridge test `test_compiled_path_catch_output_reject` (test_map_bridge.py:271) exercises the compiled path with a catch output declared; it passes today because the test asserts only on the non-error main output and lets the catch path silently no-op.
