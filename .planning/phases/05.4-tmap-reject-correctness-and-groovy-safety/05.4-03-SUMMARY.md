# Phase 05.4 Plan 03: tMap is_reject Filter-Reject Routing Rewrite Summary

**Rewired the is_reject filter-reject routing in both Map (tMap) paths -- _evaluate_outputs_py (Python-eval) and _apply_output_filter (compiled) -- to evaluate each reject output's OWN column expressions against the failed-filter rows via _evaluate_output_columns_py, restoring Talend filter-reject semantics where reject-side-only columns (literal tags, codes, renames) are correctly populated.**

## Accomplishments

- Added `TestFilterRejectPy` class with 2 unit tests pinning the contract that the reject output's declared schema wins (literal columns + reject-side-only columns).
- Rewrote `_evaluate_outputs_py` filter-reject path (~lines 2593-2670):
  - Collect raw joined_df row source dicts in `reject_source_rows` (D-02) instead of pre-evaluated source-output rows.
  - Track `reject_source_indices` to slice pre-computed `var_columns` to the failing-row subset (`rej_var_columns`) so the helper sees per-row Var values aligned to the reject DataFrame.
  - Call `_evaluate_output_columns_py(rej_row_source, oc, rej_var_columns, main_name, lookup_names)` with the reject output's config (`oc`, not the source output's `col_defs`).
- Rewrote `_apply_output_filter` filter-reject path (~lines 2697-2731):
  - Replace `result[rej_name] = failed` with `_evaluate_output_columns_py(failed, oc, {}, main_name, lookup_names_list)` against the reject output's config.
  - `var_columns={}` -- the compiled path materialises Vars inside the Groovy script; they are not accessible to re-pass.  Reject expressions referencing `Var.*` resolve to `None` via `_VarBag.__getattr__` (same convention as `_route_inner_join_rejects`).
- Eliminated the three legacy patterns from filter-reject sites:
  - `reject_df = pd.DataFrame(reject_rows, columns=[c["name"] for c in col_defs])` (source-output col_defs)
  - `result[rej_name] = reject_df` (assignment of source-evaluated DataFrame)
  - `result[rej_name] = failed` (compiled path: raw failed rows leak through)

## Decisions Made

- **Var columns sliced positionally to failing-row subset (Python-eval path).** When the source output has an active filter, `var_columns` was computed against the full `joined_df`.  We collect `reject_source_indices` alongside `reject_source_rows`, then derive `rej_var_columns` by gather-indexing `var_columns` to just the failing rows.  This keeps per-row Var values aligned with the reject DataFrame rows when the reject output's expressions reference `Var.*`.
- **Compiled path uses `var_columns={}`.** Compiled-path filter rejection happens after the bridge round-trip.  Vars are materialised inside the Groovy script; we don't have a Python-side per-row Var snapshot to re-pass.  Reject expressions referencing `Var.*` therefore resolve to `None` via `_VarBag.__getattr__` -- consistent with `_route_inner_join_rejects`'s `var_columns={}` decision (Plan 05.4-02 D-04).  Documented inline in the call site.
- **Raw row source = `row_series.to_dict()` per D-02.** Captures the full joined-df row context (main + all lookups), matching the SPEC's D-02 row-source definition.  The materialised DataFrame uses `columns=list(joined_df.columns)` so column ordering is preserved and the helper's `_build_row_dicts_for_py` sees the same column layout it saw during the main output evaluation.
- **Concat-or-assign idiom preserved for both sites.** When `rej_name` already has rows in `result` (e.g. routed from a prior filter output in a multi-output config, or pre-initialised empty by the `is_reject` skip-when-already-populated guard), we still `pd.concat([result[rej_name], rej_df], ignore_index=True)`.  Otherwise direct assign.  The Python-eval path additionally checks `not result[rej_name].empty` to skip concat against the empty-placeholder DataFrame.

## Deviations from Plan

None -- plan executed exactly as written.  Two minor refinements made during implementation, both documented above and aligned with the plan's intent:

1. **Plan said "collecting raw row_series dicts: `reject_source_rows.append(row_series.to_dict())`".** Implemented exactly as specified.  Added the parallel `reject_source_indices` list because the plan's note about per-row Var alignment implied that `var_columns` indexed positionally against the failing rows -- but `var_columns` is computed against full `joined_df` indices.  The index list lets us gather-index to produce `rej_var_columns` for the helper without iterating `var_columns` twice.
2. **Plan said compiled path uses `var_columns={}`.** Implemented exactly as specified.  Rationale documented in the docstring/comment block of the call site so future readers don't think it's an oversight.
