# Phase 05.4 Plan 06: Compiled-Path Per-Reject-Output Method Emission + Dual-Invocation Dispatch Summary

**One-liner:** D-09 closes the `{{java}}`-marker gap on tMap reject outputs in the compiled-Groovy path -- the compiled script now emits a separate pass of helpers for reject outputs, and `_evaluate_outputs_compiled` invokes the same compiled script twice (active mode over `joined_df`, reject mode over the reject row source assembled from `inner_join_reject_dfs`) so reject column expressions are evaluated by the bridge instead of being silently dropped on the Python-eval fallback.

---

## D-09 Option A: rationale recap

The plan offered three shapes for closing the reject-output gap:

- **Option A (chosen):** Compile once, invoke twice. Same compiled script body services both passes; the row loop branches on a `rejectMode` flag.
- Option B: Compile twice (two different scripts, one per mode).
- Option C: Single invocation, both row sources interleaved with a per-row mode discriminator.

Option A wins for three reasons:

1. **Hot-path is unchanged.** The active-output pass runs the same row loop it ran before -- one extra `if (!rejectMode)` branch on a constant binding value is folded away by the JIT after the first chunk. No per-row cost.
2. **No bytecode bloat.** The script body grows by a constant amount (one helper pass + one branch in the row loop + one branch in the results emit). Both passes share the JVM method-size budget controlled by `_FILL_CHUNK`; reject outputs follow the same per-column chunk limit so neither pass ever overflows the 64 KB Code attribute.
3. **No Java-side change needed.** The mode flag rides on the `contextVars` map that `executeCompiledTMap` already `putAll`'s into the binding-visible `context`. We didn't have to add a binding parameter to the Java API.

The compile-twice shape (Option B) doubled the script-cache footprint and made the per-component compile a moving target whenever reject configuration changed mid-job. The interleaved shape (Option C) needed a per-row mode discriminator threaded through every helper -- the JIT would not be able to constant-fold it away.

## How `rejectModeBool` reaches the Groovy script

The plan's vocabulary used `rejectModeBool` as a binding name. Implementation: the flag is propagated via `context["__rejectMode__"]` (a key on the existing `context` binding, not a new top-level binding). The generated script reads:

```groovy
boolean rejectMode = (context != null && context.get("__rejectMode__") != null)
    ? ((Boolean) context.get("__rejectMode__")).booleanValue() : false;
```

- Default-false: any pre-existing script in the cache that lacks this binding (or any compiled-path invocation that does not pass `reject_mode=True`) sees `rejectMode=false` and behaves exactly as before.
- Type-safe cast: the bridge stuffs a Python `bool` into `self.context["__rejectMode__"]`; Py4J marshals it to a Java `Boolean`; the cast unwraps it.
- Scoped mutation: see "Try/finally context-flag scoping" below.

This avoids growing the `executeCompiledTMap` Java signature -- a far less invasive surface than a new `setBinding` call would have been.

---

## How `_evaluate_outputs_compiled` performs the second invocation

The control flow is:

```
script = _build_compiled_script(...)                      # one Groovy compile, both passes
java_bridge.compile_tmap_script(...)
# First invocation -- active outputs over joined_df
raw_result = java_bridge.execute_compiled_tmap_chunked(
    df=joined_df, reject_mode=False, ...)
# Second invocation -- reject outputs over reject row source
if reject_outputs and inner_join_reject_dfs:
    reject_source = self._build_reject_row_source(...)
    if reject_source is not None and not reject_source.empty:
        reject_raw = java_bridge.execute_compiled_tmap_chunked(
            df=reject_source, reject_mode=True, ...)
# Merge results
for output_cfg in outputs_config:
    out_name = output_cfg["name"]
    if out_name in raw_result:    result[out_name] = raw_result[out_name]
    elif out_name in reject_raw:  result[out_name] = reject_raw[out_name]
    else:                         result[out_name] = pd.DataFrame(columns=...)
```

Skips on either side:

- **No reject outputs in config** -> only the first invocation runs (back-compat with all pre-Plan-06 jobs).
- **`inner_join_reject_dfs` is None or empty** -> only the first invocation runs (no rejects to evaluate; firing the second pass with empty input is a wasted round trip).

The merge is mutually exclusive: the first pass returns only active-output keys, the second pass returns only reject-output keys (the script body's result-map emission is `rejectMode`-branched), so the merge never has to disambiguate or concat. If the bridge ever returns a key from both passes (a bug, not a current behaviour), the active-pass value wins -- this is the safer default because active-pass rows are always the larger set.

## How the reject row source DataFrame is constructed

`Map._build_reject_row_source(inner_join_reject_dfs, joined_columns)`:

1. Iterate per failing-lookup reject frame in `inner_join_reject_dfs`.
2. Skip frames that are None / empty.
3. `reindex(columns=joined_columns)` on each kept frame -- aligns each frame to the joined-frame schema. Failing-lookup columns become NaN (which the Arrow serialiser maps to null), previously-matched lookup columns remain populated (D-05 partial-match binding), main columns remain populated.
4. Single frame -> return it directly; multiple frames -> `pd.concat(..., ignore_index=True)`; no frames -> return None.

Why reindex onto the joined-frame schema specifically: the compiled script was compiled against `joined_df`'s schema (that's what `compile_tmap_script` recorded via `input_columns=list(joined_df.columns)`). The bridge's Arrow serialiser uses the script-time schema to bind RowWrappers; a reject frame with a different column order or missing columns would mis-bind. Reindex normalises both shape and order.

## Try/finally context-flag scoping in `bridge.py`

`execute_compiled_tmap_chunked` now accepts `reject_mode: bool = False`. Before delegating to the inner body the call mutates `self.context["__rejectMode__"]`:

```python
_prior_reject_flag = self.context.get("__rejectMode__", _MISSING)
self.context["__rejectMode__"] = bool(reject_mode)
try:
    return self._execute_compiled_tmap_chunked_body(...)
finally:
    if _prior_reject_flag is _MISSING:
        self.context.pop("__rejectMode__", None)
    else:
        self.context["__rejectMode__"] = _prior_reject_flag
```

The `_MISSING` sentinel distinguishes "key not present before this call" (so the finally branch `pop`s it) from "key present and explicitly None or False before this call" (so the finally branch restores the exact prior value). Without the sentinel, an `if _prior_reject_flag is None` check would have conflated absence with explicit None and incorrectly persisted the flag.

This scoping matters because the same `JavaBridge` instance is shared across many tMap components in one job; leaking `__rejectMode__` from one component into a downstream component's compiled script would silently corrupt that downstream component's row-loop dispatch.

## `_route_inner_join_rejects` and the compiled path -- coexistence

The Python-eval path still uses `_route_inner_join_rejects` downstream of `_evaluate_outputs` to materialise inner-join-reject outputs. The compiled path now does its own reject materialisation via the second bridge invocation. Running both would duplicate rows.

`_process` now branches:

```python
if _use_compiled:
    result = self._evaluate_outputs(..., inner_join_reject_dfs=inner_join_reject_dfs)
    # Compiled path already routed rejects via the second invocation.
else:
    result = self._evaluate_outputs_py(...)
    self._route_inner_join_rejects(result, inner_join_reject_dfs, outputs_config)
```

The early-return `joined_df.empty` branch still unconditionally calls `_route_inner_join_rejects` -- in that branch the compiled script is never invoked (no rows to feed it), so the Python-eval reject routing is the only path that produces reject output rows. This is a deliberate carve-out: dispatching the compiled script over an empty input would be a wasted bridge round trip and the Python-eval helper handles the empty-joined case correctly already.

## Complications encountered during implementation

1. **`inner_join_reject_dfs` was not in `_evaluate_outputs_compiled`'s signature.** The plan flagged this. Resolution: added it as an optional `inner_join_reject_dfs: dict | None = None` kwarg, threaded through `_evaluate_outputs` from `_process`. Defaults to None for any caller (including standalone tests) that does not have reject row data; the second invocation is then skipped.

2. **Bridge `execute_compiled_tmap_chunked` had no `rejectMode` parameter.** Resolution: added `reject_mode: bool = False` kwarg. The body was split into a public entry point that scopes the context-flag mutation and a private `_execute_compiled_tmap_chunked_body` that does the actual chunked execution; this keeps the try/finally tight and the body unchanged from before.

3. **Java-side change avoidance.** The plan suggested a `rejectModeBool` Groovy binding. A new top-level binding would require a Java-side change to `executeCompiledTMap` (a new `Map<String,Object> extraBindings` arg). Implementation chose the lower-risk path: piggyback on the existing `context` binding (which `executeCompiledTMap` already `putAll`'s into) and reserve the `__rejectMode__` key. The Groovy script reads `context.get("__rejectMode__")`. No Java rebuild needed, no API surface widened.

4. **Generated-script bytecode budget for the reject pass.** Reject-output helpers follow the same per-column chunk size (`_FILL_CHUNK = 1`) as active outputs, so the same 64 KB per-method JVM limit applies independently to each helper. Reject outputs typically have far fewer columns than active outputs (mostly carry main-row columns + a small set of error tags), so this is not a tight constraint in practice.

5. **`_route_inner_join_rejects` double-route bug almost shipped.** Before the `_process` carve-out the first integration would have run both reject paths -- the compiled path's second invocation AND the Python-eval helper -- duplicating reject rows. Caught during the dual-invocation test's authoring. The carve-out is the smallest scoped fix; the helper is still reachable via the Python-eval path and the empty-joined branch.

## Decisions Made

- **D-09 Option A chosen** for the reasons enumerated in the "rationale recap" section.
- **`__rejectMode__` reserved-key namespace on `context`.** The leading-and-trailing double-underscore convention matches the existing `__errors__` key emitted by the compiled script (and consumed by `_route_catch_output_rejects`). User-defined Talend context variables are unlikely to collide; if they do collide the bridge owns the namespace.
- **Schema alignment via `reindex(columns=joined_columns)`.** Considered the alternative of compiling a separate reject-pass script keyed on the reject row source's own schema; rejected as overkill given that the failing-lookup columns are the only difference and Arrow handles missing columns as null.
- **Skip `_route_inner_join_rejects` in the compiled path.** Considered the alternative of letting both paths run and dedup'ing afterwards; rejected as fragile (dedup'ing on row identity across pandas concat is order-dependent).
- **Live-bridge dual-invocation test deferred to Plan 07.** The unit-level mock-bridge test pins the dispatch contract; the live-bridge round trip would need a real Groovy compile + Arrow serialisation pair which is Plan 07's test matrix scope.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `_route_inner_join_rejects` would have double-routed rejects on the compiled path**

- **Found during:** Task 2 implementation, while writing the dual-invocation dispatch.
- **Issue:** The plan said "after the first bridge invocation, if there are reject outputs ... invoke the compiled script a second time". It did not flag that `_process` ALSO calls `_route_inner_join_rejects` downstream of `_evaluate_outputs`, which would re-route the same reject rows on top of the bridge's already-populated reject output frames. Result: duplicate rows in every reject output.
- **Fix:** `_process` now branches: compiled path skips `_route_inner_join_rejects`; Python-eval path keeps the call. The early-return `joined_df.empty` branch keeps the call too (compiled script is not invoked in that branch).
- **Files modified:** `src/v1/engine/components/transform/map.py` (the `_process` step-9 block).
- **Verification:** No new test added for this specific path (the existing `TestInnerJoinReject` class exercises the Python-eval path which is unaffected). The dual-invocation unit test verifies the bridge is invoked twice; the live-bridge test deferred to Plan 07 would catch any row-count regression. The carve-out is a one-line `if not _use_compiled:` guard whose intent is documented inline.
- **Why Rule 1 not Rule 4:** This is a bug introduced by the dual-invocation wiring (without the carve-out, the new code would double-emit reject rows). It is not an architectural change -- both paths exist already; the carve-out just picks the correct one per execution mode.

**2. [Rule 2 - Critical correctness] Bridge `_MISSING` sentinel for exact context-flag restore**

- **Found during:** Task 2 implementation, while writing the try/finally context-flag scope.
- **Issue:** The naive restore -- `self.context["__rejectMode__"] = _prior_reject_flag` after `_prior_reject_flag = self.context.get("__rejectMode__")` -- would conflate "key was absent before this call" with "key was present and explicitly None before this call". The first case demands `pop`; the second case demands restoring the explicit None.
- **Fix:** Introduced a module-level `_MISSING = object()` sentinel; `self.context.get("__rejectMode__", _MISSING)` distinguishes absence from explicit-None; the finally branch picks `pop` or `assign` accordingly.
- **Files modified:** `src/v1/java_bridge/bridge.py`.
- **Why Rule 2:** Without the sentinel, a downstream consumer that sets `context["__rejectMode__"] = None` (e.g. to clear a stale flag) before this call would have that explicit None silently dropped after the call returns. The correctness guarantee of "the call leaves `self.context` in exactly the prior state" is required because the JavaBridge instance is shared across components.
