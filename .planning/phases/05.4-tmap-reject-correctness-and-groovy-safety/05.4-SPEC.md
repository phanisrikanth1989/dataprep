# Phase 05.4: tMap Reject Correctness and Groovy Safety — Specification

**Created:** 2026-05-15
**Ambiguity score:** 0.173 (gate: ≤ 0.20)
**Requirements:** 7 locked

## Goal

Restore tMap reject-routing fidelity to Talend semantics across all three reject mechanisms (`inner_join_reject`, `is_reject` filter-reject, `catch_output_reject`), perform a full Groovy-safety audit of expression embedding in the compiled-script generator, and eliminate the pandas `PerformanceWarning: DataFrame is highly fragmented` emitted by reject routing. End-state: every reject-output column expression is evaluated against its source row (matching Talaxie's `tMap_main.inc.javajet` and the historical `origin/main` Map implementation), Groovy compilation no longer fails on Talend expressions containing `$` or other Groovy-special characters, and the reject-routing code path no longer triggers pandas fragmentation warnings.

## Background

Commit `9ab8120` (2026-04-15) — *"feat(05-01): rewrite tMap engine component from scratch with full Talend feature parity"* — rewrote `src/v1/engine/components/transform/map.py` (~1800 lines). The rewrite's goal was Talend parity, but it lost reject-routing fidelity at three sites and embeds Talend expressions raw into Groovy script lines without escaping Groovy-special characters. The `origin/main` Map (pre-restructure) and Talaxie's canonical tMap codegen template both got reject routing right; the current `feature/engine-restructure` branch is the outlier. Phase 05.3 (just merged) did not touch any of the buggy code paths and inherited the bugs.

End-to-end repro confirmed via `.item → converter → engine` pipeline at `/tmp/repro_inner_reject/Job_inner_reject.item`. With 5 inner-join-unmatched rows expected to produce a `rej` output with 7 columns, only 2 columns (those whose names coincidentally match main DataFrame columns) are populated; the other 5 (renamed simple refs, hard-coded literals, java expressions) are all None. The pandas `PerformanceWarning` is emitted from `map.py:2614` (`reject_df[col_name] = None`) during this routing.

Talaxie / Talend OS reference: `tdi-studio-se/main/plugins/org.talend.designer.components.localprovider/components/tMap/tMap_main.inc.javajet:1740-1796` (loop emitting per-column-expression assignment) and `1925-1988` (catch-output-reject loop with reserved `errorMessage` / `errorStackTrace`).

## Requirements

1. **R1 — inner_join_reject column-expression evaluation**: Inner-join-reject output columns are evaluated per row using the reject output's own expression list.
   - Current: `map.py:2583-2626 _route_inner_join_rejects` does name-based DataFrame column copy from `all_rejects`. Hard-coded literals, renamed simple refs, and `{{java}}` expressions all collapse to None. Only columns whose name matches a main DataFrame column survive.
   - Target: When unmatched-main rows route to an `inner_join_reject: True` output, evaluate the reject output's column expression list against each unmatched row, producing a DataFrame whose columns match the reject output's declared schema. Behavior applies uniformly across all five join paths: `_join_equality`, `_join_main_side_computed_key`, `_join_lookup_side_computed_key`, `_join_cross_table` / `_chunked_cross_product`, `_join_reload_per_row`.
   - Acceptance: For each of the 5 join paths × 4 column kinds (same-name simple ref, renamed simple ref, hard-coded literal, java expression) = 20 tests, the resulting reject DataFrame contains the expected literal/computed value (not None) for every column.

2. **R2 — is_reject filter-reject column-expression evaluation**: When an output's `activate_filter` predicate rejects a row, the destination `is_reject` output's own column expressions are evaluated on that row.
   - Current: `map.py:2470-2484` (in `_evaluate_outputs_py`) and `map.py:2520-2532` (in `_apply_output_filter`) build the reject DataFrame using the SOURCE output's `col_defs`, then `result[rej_name] = reject_df` overwrites the placeholder's schema. The reject output's own column list is never evaluated; reject-side-only columns vanish.
   - Target: Filter-rejected rows are routed to the first `is_reject: True, inner_join_reject: False` output by evaluating THAT output's column expressions against each rejected row. The reject output's declared schema wins; source-output columns are not silently carried over.
   - Acceptance: Test with one source output having columns `(id, tag="KEEP")` and a reject output declaring `(id, tag="FILTER_FAIL", code=42)` — every reject row in the reject output has `tag="FILTER_FAIL"` and `code=42`, and the reject DataFrame has 3 columns, not 2.

3. **R3 — catch_output_reject column-expression evaluation**: User-defined column expressions on a catch-output-reject output are evaluated per row, with only `errorMessage` and `errorStackTrace` columns auto-filled by the framework.
   - Current: `map.py:2542-2575 _route_catch_output_rejects` copies the raw `error_df`, appends an `errorMessage` literal column, and writes the result. User-defined column expressions on the catch output are ignored.
   - Target: Mirror Talaxie's `tMap_main.inc.javajet:1925-1988` — iterate the catch output's column list, evaluate each user expression on the source row, reserve `errorMessage` and `errorStackTrace` for framework-supplied values (exception text + stack trace).
   - Acceptance: Test with a catch-output-reject output declaring `(original_id=row1.id, custom_tag="ERR", errorMessage)` — when an expression error fires on a main row, the catch output has `original_id` populated from the row, `custom_tag="ERR"`, and `errorMessage` containing the exception text.

4. **R4 — Shared per-output evaluator**: A single helper evaluates one output's column expressions over an arbitrary row source, callable from all reject routing sites and the main-output path, on both the python-eval and compiled paths.
   - Current: `_evaluate_outputs_py` and `_evaluate_outputs_compiled` each iterate `outputs_config` and contain main-flow + reject routing logic inline. Reject routing duplicates structure without sharing the evaluator.
   - Target: Extract a helper (signature TBD in discuss-phase, but conceptually `_evaluate_one_output(row_source, output_cfg, var_columns, main_name, lookup_names, java_bridge_or_none) -> pd.DataFrame`). All three reject sites call this helper. Main-output path may also delegate to it. NO copy-paste from `origin/main`; the helper is designed for the current dual-path architecture (post-05.3) with the `_has_any_java_marker()` dispatch already in place.
   - Acceptance: A `grep -c "reject_df\[" src/v1/engine/components/transform/map.py` returns 0 (no incremental column assignment for reject construction); each reject routing site reduces to one helper call plus result-dict insertion.

5. **R5 — Null-lookup expression semantics**: When a reject output column expression references a row that is null in this routing context (e.g., `row2.country_name` on an unmatched inner-join row), the expression raises naturally and is handled by the existing `die_on_error` / `catch_output_reject` machinery — matching Talend.
   - Current: With the current bug, the reject column was unconditionally None so the question never arose.
   - Target: Match Talend behavior. In the compiled path, `row2.country_name` on a null `row2` raises NPE (Groovy), which the bridge surfaces as an exception. In the python-eval path, `row2.country_name` on a `None`-like row source raises `AttributeError`. Both propagate through the existing per-output error handling: if `die_on_error=true`, the job fails; if `catch_output_reject` is configured for that output, the row routes there; otherwise the row is dropped with an error logged. No new error-handling layer is introduced.
   - Acceptance: Test where reject output references `row2.col` on inner-join-unmatched row: with `die_on_error=true`, job raises ComponentExecutionError; with `catch_output_reject` configured, the row appears in the catch output; with neither, the row is silently dropped and an ERROR is logged.

6. **R6 — Groovy-safety in compiled-script embedding**: The compiled-script generator escapes/handles every character Groovy parses differently from Java, in every site where Talend expression text is interpolated into a Groovy source line.
   - Current: `map.py:2782` and all sibling sites (`_build_compiled_script` body — variables at line ~2855, output filters at line ~2791, join key keys, lookup filters) use raw f-string interpolation: `lines.append(f"    row[{abs_idx}] = {expr};")`. Talend expressions containing `$` cause Groovy compile errors (e.g. `"Total: $100"` becomes `"Total: $100"` in Groovy, which tries to interpolate `$100` as a GString identifier and fails). Other Groovy-special characters likely have similar issues but have not yet been audited.
   - Target: Full audit of Groovy-vs-Java embedding differences. At minimum: `$` inside string literals must be escaped to `\$`. Audit also covers backslash escape interactions, unicode quote characters, raw-string boundary characters, GString-vs-Java-string semantics, and Groovy identifier rules. Every identified difference is documented and either (a) escaped at embedding time, or (b) flagged as unsupported with a clear error. The audit deliverable is a written list of every Groovy-vs-Java parsing difference encountered with the corresponding handling choice.
   - Acceptance: A Talend expression `"Total: $100"` (column expression in any output) compiles and produces the literal string `"Total: $100"` in output. The audit document exists at `.planning/phases/05.4-tmap-reject-correctness-and-groovy-safety/05.4-GROOVY-AUDIT.md` enumerating each Groovy-vs-Java difference plus its handling. Test suite includes one test per audited character class.

7. **R7 — Eliminate reject-path pandas fragmentation warning**: Reject DataFrames are constructed in a single allocation, not by incremental column assignment.
   - Current: `_route_inner_join_rejects:2612-2614` (`reject_df[col_name] = ...` in a loop) emits `PerformanceWarning: DataFrame is highly fragmented` when reject outputs have more than a few columns. `_route_catch_output_rejects:2572` adds `output_df["errorMessage"] = "..."` after a `.copy()`, also fragmenting. Both observed by the manager in production logs.
   - Target: Both reject sites construct the result DataFrame in a single `pd.DataFrame(rows, columns=...)` call (or equivalent dict-based construction). No incremental `df[col] = ...` writes inside loops in reject routing.
   - Acceptance: Running the test matrix with `warnings.filterwarnings("error", category=pd.errors.PerformanceWarning)` produces zero `DataFrame is highly fragmented` warnings from any reject-routing code path. Note: this scope is limited to reject sites; the four other incremental-column sites in map.py (L1091, L1093, L1906-1915 — joined_df construction in computed-key joins and column re-assignment) are intentionally deferred to a separate follow-up phase per user direction.

## Boundaries

**In scope:**
- Rewrite of the three reject-routing functions (`_route_inner_join_rejects`, the `is_reject` paths inside `_evaluate_outputs_py` and `_apply_output_filter`, `_route_catch_output_rejects`) so each reject output's column expressions are evaluated per row.
- Extraction of a shared per-output expression evaluator helper, designed for the current dual-path architecture (python-eval + compiled).
- Full Groovy-safety audit and corresponding escaping/handling code in the compiled-script generator (every site where Talend expression text is embedded into a Groovy line).
- Removal of the `reject_df[col_name] = ...` and `output_df["errorMessage"] = ...` incremental-column patterns in reject routing only.
- Test matrix: 5 join paths × 3 reject mechanisms × 4 column kinds = ~60 reject tests, plus Groovy-safety tests (one per audited character class). All assertive — assert exact expected column values, not just row counts.
- End-to-end `.item → converter → engine` regression fixture promoting `/tmp/repro_inner_reject/Job_inner_reject.item` into `tests/talend_xml_samples/Job_05_4_*.item`.
- Updated documentation: SUMMARY.md per plan, plus a `05.4-GROOVY-AUDIT.md` artifact.

**Out of scope:**
- Fragmentation cleanup of the four other incremental-column sites in `map.py` (L1091, L1093, L1906, L1908, L1913, L1915) — deferred to a separate follow-up phase per user direction. These are inside join-construction code, not reject routing, and rewriting them is its own design exercise.
- Re-architecting the dual-path (`_evaluate_outputs_py` vs `_evaluate_outputs_compiled`) split introduced by phase 05.3 plan 05 — the new helper is layered on top, not a replacement.
- Reverting the phase-5 rewrite to main-branch code — the regression is fixed by a properly-designed adaptation to the current architecture, not by a copy-paste.
- Backward-compatibility flag for the old buggy behavior — clean break per user direction; any job relying on the old None-everywhere reject behavior was already producing wrong data.
- Talend Studio diff-based testing (running actual Talend OS / Talaxie and byte-comparing output) — tests use behavioral assertions only, derived from Talaxie codegen semantics.
- D-07 (Phase 05.3 deferred: empty-lookup → downstream `nullable=false` crash) — remains tracked in REQUIREMENTS.md as its own future phase. Not touched here.

## Constraints

- **Feature parity is non-negotiable** (per `CLAUDE.md`): the engine must produce identical output to Talend on the same job. The shared helper's behavior is validated against Talaxie's `tMap_main.inc.javajet` per-column-assignment semantics.
- **Both dispatch paths must be supported** post-phase-05.3: python-eval (no `{{java}}` markers anywhere) and compiled (markers present). The shared helper either dispatches internally or has python-eval and compiled variants that share interface.
- **No converter change**: the JSON shape produced by `src/converters/talend_to_v1/components/transform/map.py:_parse_outputs` (`inner_join_reject`, `is_reject`, `catch_output_reject` flags + column lists with `expression`) is correct and must not be re-formatted. The fix is engine-only.
- **No regression to phase 05.3 work**: all 8 phase-05.3 plans' tests must continue to pass. The `_has_any_java_marker()` dispatch (plan 01), locality classifier (plan 02), computed-key joins (plan 03), chunked cross-product (plan 04), python-eval path (plan 05), file_output filepath expression (plan 06), and e2e suite (plan 07) are downstream consumers of map.py — none should regress.
- **Existing 11 `inner_join_reject: True` test sites in `test_map_bridge.py`** must have their assertions strengthened during this phase: their current assertions pass only because of coincidental name-matching between reject output columns and main DataFrame columns. After this phase, those tests must assert on the actual evaluated reject output values.
- **Phase 14 coverage gate (95% per-module floor)** must still pass post-fix. Note: phase 05.3 plan 07 SUMMARY observed map.py at 88.3% — that's pre-existing gap; the new reject-routing code must come with sufficient tests to not push the floor lower.
