## Implementation Decisions

### D-01 — Shared evaluator design (LOCKED)
- **Extract `_evaluate_output_columns_py(row_source, output_cfg, var_columns, main_name, lookup_names) -> pd.DataFrame`** from the body of `_evaluate_outputs_py:2415-2486`. The new helper does the per-row loop (build namespace per row from `row_source`, evaluate filter if active, evaluate each column expression via `_eval_expr`, collect into row dicts, return as `pd.DataFrame.from_records(rows, columns=[c["name"] for c in col_defs])`). NO incremental column assignment.
- **`_evaluate_outputs_py` becomes a thin wrapper** that loops over non-reject outputs and calls `_evaluate_output_columns_py` for each. Reject outputs are still skipped here — they are evaluated by the reject routing functions calling the same helper.
- **Compiled path: extend `_build_compiled_script` to emit per-reject-output methods** alongside the existing non-reject methods, mirroring Talaxie's `tMap_main.inc.javajet` per-output method emission. The existing single-bridge-call execution in `_evaluate_outputs_compiled` then produces ALL outputs in one shot, including reject ones — reject routing in Python just reads from the bridge result map without a second bridge call. This avoids a per-row extra bridge round trip and keeps wall-clock parity with main-branch behavior. (See D-09 for the dispatch wiring inside the Groovy row loop.)
- **Dispatcher per reject routing site:** `self._has_any_java_marker()` (phase 05.3 plan 01 helper) picks the path. Compiled path reads the bridge result map; python-eval path calls `_evaluate_output_columns_py` directly with the appropriate row source DataFrame.
- **No refactor of `_evaluate_outputs_compiled` interface** — the new reject-method-emission in `_build_compiled_script` is internal; the function signature and return shape stay the same.

### D-02 — Row source DataFrames per reject mechanism (LOCKED)
- **`inner_join_reject`:** row source = the rejects DataFrame returned by `_join_equality` / `_join_main_side_computed_key` / `_join_lookup_side_computed_key` / `_join_cross_table` / `_join_reload_per_row` (lines 595-596 collect these into `inner_join_reject_dfs`). This DataFrame contains: main columns (bare) + columns from PREVIOUSLY-JOINED lookups (with `lookup.col` prefix) + NaN/missing for columns from the lookup that failed. Reject output expressions referencing the failing-lookup row produce `None`/raise (per D-05 below).
- **`is_reject` filter-reject:** row source = the rows that failed the source output's `activate_filter` predicate. This is the SAME `joined_df` slice that the source output sees, just filtered to keep only the failing rows. Reject output's expressions evaluate against full joined_df row context (main + all lookups), since the row is "complete" — it failed the filter, not the join.
- **`catch_output_reject`:** row source = the rows where an upstream expression raised. Same context as `is_reject` (joined_df row) but enriched with `errorMessage` and `errorStackTrace` columns populated by the framework. User-defined column expressions evaluate against the joined_df row; reserved column names `errorMessage` / `errorStackTrace` are framework-filled (Talaxie pattern, `tMap_main.inc.javajet:1925-1988`).

### D-03 — Reject DataFrame allocation strategy (LOCKED — addresses R7)
- **All three reject sites build the DataFrame in a single `pd.DataFrame.from_records(rows, columns=col_names)` allocation.** Rows are accumulated as a list of dicts during the per-row evaluation loop, then materialized once. No `df[col] = ...` writes inside loops. No `.copy()` followed by column append.
- **Empty-rejects fast path:** if the row source DataFrame is empty, return an empty DataFrame with the output's declared schema (`pd.DataFrame(columns=col_names)`). Skips the per-row loop entirely.

### D-04 — Null-row & error propagation (LOCKED — addresses R5)
- **Python-eval path:** for `inner_join_reject`, when the reject row source's columns are missing the failing lookup's columns, the expression namespace binding for that lookup is a `_NullRow` sentinel (a singleton object that raises `AttributeError`/`KeyError` with the message `lookup '<name>' was not matched; column '<col>' unavailable on reject row` — ASCII-only per project preference).
- **Compiled path:** for `inner_join_reject`, the Groovy script's `RowWrapper` for the failing lookup is bound to `null`. Accessing `row2.col` on a null `RowWrapper` raises NPE in Groovy — caught by the existing per-row try/catch in the compiled script (already in place for `die_on_error=false` and `catch_output_reject` routing).
- **Error propagation chain (both paths):** any exception during reject column evaluation propagates through the existing error handling layer:
  - If `die_on_error=true` on the component: raises `ComponentExecutionError` from `_process()`.
  - If `catch_output_reject` is configured for this output: row is routed to the catch output with `errorMessage` populated.
  - Otherwise: row is dropped, ERROR logged (ASCII), reject output count is decremented.
- **No new error-handling layer.** The bug fix is purely about routing expression-evaluation through the right code; error handling stays as-is.

### D-05 — Multi-lookup partial-match reject binding (LOCKED — Talaxie parity)
- **When INNER_JOIN fails on lookup B but the main row had previously matched lookup A successfully**, the reject row source carries main columns + lookup A columns. Reject output expressions referencing `row2_A.col` resolve correctly; expressions referencing `row2_B.col` raise per D-04.
- **Talaxie parity:** `tMap_main.inc.javajet` row loop binds each lookup's `RowWrapper` once at the top; if a later lookup fails, prior lookups' rows remain bound. Mirror that.
- **Implementation note:** the existing rejects accumulation in `_process` (lines 595-596) already carries the partial-join state correctly — just needs the reject routing to USE that information by evaluating the reject output's expressions against the partial row, not by name-based copy.

### D-06 — Catch-output-reject reserved column policy (LOCKED — Talaxie parity)
- **Two column names are framework-reserved:** `errorMessage` (populated with `exception.message`) and `errorStackTrace` (populated with the stack trace string). User-defined column expressions for any OTHER column name evaluate normally against the joined_df row context.
- **Talaxie reference:** `tMap_main.inc.javajet:1925-1988` skips these two names during user-expression evaluation:
  ```java
  if (outputColumnName.equals("errorMessage") || outputColumnName.equals("errorStackTrace")) {
      continue;
  }
  ```
- **If a user defines a column named `errorMessage` or `errorStackTrace` with their own expression**, the framework value WINS (matches Talaxie). Document this in SUMMARY.md so users know these names are reserved.

### D-07 — Groovy escape strategy (LOCKED — addresses R6)
- **Location:** inside the engine's `_build_compiled_script` and its sibling expression-embedding sites, just before f-string interpolation into a Groovy line. NOT in the converter (keeps converter pure XML→JSON; converter has no view of Groovy semantics). NOT in the Groovy template (Groovy has no raw-string mode that handles all chars).
- **Strategy:** new helper `_groovy_escape_expression(java_expr: str) -> str` that tokenizes the input to identify string-literal regions (text between unescaped `"` characters, respecting Java escape rules), and applies Groovy-safety escapes inside those regions only. Non-string portions (operators, identifiers, numeric/null literals) are passed through unchanged because Java and Groovy share that syntax.
- **Escapes applied inside string literals:**
  - `$` → `\$` (unconditional; covers GString interpolation marker)
  - Other characters TBD by audit; default to "pass through unchanged" if Java and Groovy agree.
- **Identifier handling:** Java allows `$` in identifiers (e.g. `$tmp`). Groovy also allows it. No escaping needed for identifier `$`. The audit must distinguish "$ inside string literal" (escape) from "$ in identifier" (leave alone) — same lexer-state distinction Java uses.
- **Embedding sites to retrofit (every call site in `_build_compiled_script`):**
  1. Output column expressions — `lines.append(f"    row[{abs_idx}] = {expr};")` (~L2782)
  2. Output filters — `lines.append(f"    if (!({clean_filter})) return null;")` (~L2791)
  3. Variable expressions — `lines.append(f'        Var.put("{var_name}", {expr});')` (~L2855)
  4. Join key expressions (only paths that send keys to bridge — main-side and lookup-side computed keys)
  5. Lookup filters and main filter (when present)
  6. Reserved-column placeholders in catch_output_reject (the `errorMessage` and `errorStackTrace` framework assignments — these construct Groovy string literals from the exception, so they need their own bridge-side escape rule)
- **Audit deliverable:** `05.4-GROOVY-AUDIT.md` enumerating every Groovy-vs-Java parsing difference with the chosen handling (escape / leave alone / unsupported). Tests in `test_map_groovy_safety.py` cover one per audited difference.

### D-08 — Test matrix organization (LOCKED — addresses acceptance criteria)

### D-09 — Compiled-script reject-output method generation (LOCKED — refines D-01)
- **Per-reject-output evaluator methods** are emitted by `_build_compiled_script` alongside the non-reject `evalOutput_<name>` methods. Each reject output gets `evalOutput_<reject_name>(int i, RowWrapper main, RowWrapper lk1, ..., Map<String,Object> Var)` that evaluates the reject output's columns and returns an `Object[]`.
- **Row-loop dispatch inside the compiled script:** the existing per-row loop (`for (int i = 0; i < rowCount; i++)`) needs awareness of which input row is matched-vs-rejected. Two options the planner can pick from:
  - **Option A (preferred):** invoke the compiled script TWICE — once with `joined_df` (matched rows) emitting only non-reject outputs, once with the rejects DataFrame emitting only reject outputs. Same compiled script object, two `executeCompiledTMap` calls with different inputs and a `reject_mode` flag binding. No new script structure beyond a top-level `if (rejectMode) { call reject evalOutputs; } else { call normal evalOutputs; }` block.
  - **Option B:** add a marker column to the unified input DataFrame indicating reject-vs-matched, branch inside the row loop. More invasive to the script structure; rejected as default for this phase.
- **Planner's call** between A and B during plan-phase. SPEC.md and CONTEXT.md don't lock the choice, but Option A is recommended.

- **One plan in this phase is dedicated to this audit-and-strengthen pass.** Without it, the fix could regress and existing tests would silently pass.

## Existing Code Insights

### Reusable Assets
- **`_eval_expr`, `_build_namespace`, `_VarBag` (phase 05.3 plan 05)** at `map.py:2079-2280` — per-row Python expression evaluator. The new `_evaluate_output_columns_py` is a thin orchestrator over these.
- **`_has_any_java_marker()` (phase 05.3 plan 01)** at `map.py:3009-3060` — single source of truth for path dispatch. Reject routing functions use this to pick python-eval vs compiled.
- **`_bridge_eval` (phase 05.3 plan 02)** at `map.py:2984` — single call site for bridge expression evaluation. Compiled-path reject evaluation goes through this (or directly through `_evaluate_outputs_compiled` per D-01).
- **`_strip_java_marker`** at `map.py:~3060` — strips the `{{java}}` prefix before embedding in compiled script. The new `_groovy_escape_expression` runs AFTER this.
- **`_BridgeShim` pattern (phase 05.3 plan 07)** — used in e2e tests to disable ETLEngine's auto-start of JavaBridgeManager when a session bridge is passed, preventing port contention under `pytest -n auto`. New live-bridge tests in this phase follow the same pattern.

### Established Patterns
- **Per-row namespace building:** `_build_namespace(row_dicts, var_bag)` constructs a dict of `{row1: dict, row2: dict, Var: _VarBag}` for `eval(expr, ns)`. Reject row sources fit this pattern with `row2 = _NullRow()` for unmatched-lookup variants.
- **DataFrame from records:** the python-eval path already uses `pd.DataFrame(rows, columns=[c["name"] for c in col_defs])` at L2464-2466. The new helper formalizes this pattern.
- **Per-output evaluator method emission in compiled script:** `_build_compiled_script` already emits `evalOutput_<name>(...)` methods per non-reject output (L141-153 of the function body, anchored to L2632+ in the file). Adding reject outputs follows the same emission pattern.
- **Result-map structure from bridge:** `_evaluate_outputs_compiled` returns `dict[str, pd.DataFrame]` keyed by output name. Reject outputs naturally fit this shape.

### Integration Points
- **`_evaluate_outputs_py` and `_evaluate_outputs_compiled`** are called from `_process` after the join phase. Reject routing is called after these. The new design keeps this sequencing.
- **`_route_inner_join_rejects` is called TWICE in `_process`:** once on the early-exit path (no main rows survived joins, L604) and once on the normal-exit path (L655). The rewrite must preserve both call sites.
- **Catch output reject `__errors__` key in raw_result:** `_route_catch_output_rejects` reads `raw_result["__errors__"]` produced by the compiled script. The script's error map is the row source for D-06 user-expression evaluation. Documentation lives in `_build_compiled_script` near where errors are captured.
- **Phase 05.3 test_map_05_3_e2e.py** at `tests/v1/engine/components/transform/` — this phase's e2e fixture follows the same shape and lives alongside it.
