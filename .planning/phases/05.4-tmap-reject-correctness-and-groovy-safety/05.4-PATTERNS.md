### `_evaluate_output_columns_py` (new helper extracted from `_evaluate_outputs_py`)

**Analog:** `map.py:2361-2386` (`_eval_output_row`) and `map.py:2435-2467` (per-row loop in `_evaluate_outputs_py`).

for row_idx, row in enumerate(joined_df.itertuples(index=False)):

**Column eval pattern** (`map.py:2361-2386`):
```python
def _eval_output_row(self, col_defs, ns, out_name):
    row_dict: dict[str, Any] = {}
    for col_cfg in col_defs:
        col_name = col_cfg["name"]
        col_expr = col_cfg.get("expression", "")
        if not col_expr:
            row_dict[col_name] = None
            continue
        expr = self._strip_java_marker(col_expr)
        row_dict[col_name] = self._eval_expr(expr, ns, out_name, col_name)
    return row_dict
```

**Pattern to follow:**
- Signature per D-01: `_evaluate_output_columns_py(self, row_source: pd.DataFrame, output_cfg: dict, var_columns: dict[str, list], main_name: str, lookup_names: list[str]) -> pd.DataFrame`.
- Inner loop mirrors the existing `_evaluate_outputs_py` per-row body exactly.
- DataFrame materialized once via `pd.DataFrame(rows, columns=col_names)` -- NOT incremental column assignment (R7). Empty fast path: `pd.DataFrame(columns=col_names)` when `row_source` is empty.
- `_eval_output_row` call is unchanged; the new helper is a thin loop wrapper over it.

**Deviation required:** The new helper receives an arbitrary `row_source` DataFrame (can be `inner_join_reject_dfs` content, failed-filter rows, or `joined_df`) rather than always `joined_df`. The `_build_row_dicts_for_py` call must use `row_source` and `lookup_names` from the caller's context. When a reject row source is missing lookup columns for the failing lookup, `_NullRow` is injected for that lookup (see `_NullRow` pattern below).

### `_NullRow` sentinel (new class in `map.py`)

**Analog:** `map.py:112-125` (`_NullNamespace`).

```python
class _NullNamespace:
    """Read-only namespace where every attribute access returns None."""

    def __getattr__(self, name: str) -> None:
        return None

    def __setattr__(self, name: str, value: Any) -> None:
        pass  # discard -- null namespaces are read-only
```

**Pattern to follow:**
- Same class-level docstring style (triple double-quotes, one-sentence summary + usage context).
- Same `__getattr__` / `__setattr__` override shape.
- Placed alongside `_NullNamespace` and `_VarBag` at the top of `map.py` (lines ~78-126), before any method definitions.
- Module-level singleton or instantiate-per-use -- CONTEXT.md D-04 says singleton or per-use both acceptable; pick whichever yields cleaner error messages.

**Deviation required:** Unlike `_NullNamespace` (which returns `None` silently to mirror Talend left-outer semantics), `_NullRow` must RAISE `AttributeError` with an ASCII message: `lookup '<name>' was not matched; column '<col>' unavailable on reject row`. This is the key semantic difference (D-04): inner-join-reject null-lookup references are errors, not silently None.

### `_groovy_escape_expression` (new helper in `map.py`)

**Analog A:** `map.py:3066-3077` (`_strip_java_marker`) -- simplest single-pass string helper shape.

```python
def _strip_java_marker(self, expr: str) -> str:
    """Remove {{java}} prefix from expression."""
    if expr.startswith(_JAVA_MARKER):
        return expr[len(_JAVA_MARKER):]
    return expr
```

**Analog B:** `src/converters/talend_to_v1/expression_converter.py:163-228` (`ExpressionConverter.convert`) -- state-aware tokenize-and-transform over an expression string, identifies string-literal regions via regex.

**Pattern to follow:**
- Instance method (not staticmethod) matching the rest of `map.py`'s helper style.
- Placed in the "Helper Methods" section near `_strip_java_marker` (around `map.py:3062+`).
- Docstring follows Google-style with `Args:` / `Returns:` (engine module convention per `CLAUDE.md`).
- Tokenizer distinguishes string-literal regions (between unescaped `"` chars, respecting Java escape rules) from non-string regions. Inside string literals: `$` -> `\$`. Outside string literals: pass through unchanged.
- D-07 locks the minimum escape: `$` in string literals. The audit may add more; all go here.
- Called at every expression-embedding site in `_build_compiled_script` AFTER `_strip_java_marker`.

**Deviation required:** `_strip_java_marker` is a trivial prefix strip with no tokenization. `_groovy_escape_expression` needs a real tokenizer loop to distinguish string vs non-string regions. The implementation approach (single-pass state machine vs regex alternation) is at the executor's discretion per CONTEXT.md `Claude's Discretion`.

### Per-reject-output method emission in `_build_compiled_script`

**Analog:** `map.py:2761-2800` -- existing `fillOutput_<name>_chunkN` + `evalOutput_<name>` emission for `active_outputs`.

```python
for output in active_outputs:          # line 2761
    out_name = output["name"]
    num_cols = len(output["columns"])
    # ... emit fillOutput_<name>_chunk0 ... chunkN methods (lines 2768-2784)
    lines.append(f"Object[] evalOutput_{out_name}({helper_params}) {{")
    if activate_filter and filter_expr:
        clean_filter = self._strip_java_marker(filter_expr)
        lines.append(f"    if (!({clean_filter})) return null;")
    lines.append(f"    Object[] row = new Object[{num_cols}];")
    for chunk_idx in range(num_chunks):
        lines.append(f"    fillOutput_{out_name}_chunk{chunk_idx}({chunk_fill_args});")
    lines.append("    return row;")
    lines.append("}")
```

**Pattern to follow:**
- D-09 Option A (recommended): emit reject-output `evalOutput_<name>` methods in a SEPARATE pass after the existing `active_outputs` pass. Use the same `fillOutput_<name>_chunkN` + coordinator pattern.
- `_FILL_CHUNK = 1` constant applies equally to reject output methods.
- `helper_params` and `chunk_fill_params` strings are already computed above the loop; reuse them.
- `_groovy_escape_expression` is called after `_strip_java_marker` at every `col_expr` embedding site (line 2779 analog): `expr = self._groovy_escape_expression(self._strip_java_marker(col_expr))`.
- The reject methods are called by a SEPARATE bridge invocation in Python (Option A): `_evaluate_outputs_compiled` dispatches the normal script, then reject routing dispatches the same compiled script with `reject_mode=True` binding (or equivalent two-call design per D-09 Option A). This is wired in Python, not in the Groovy row loop itself.
- `active_outputs` filter (line 2715-2718) stays: `not o.get("is_reject") and not o.get("inner_join_reject")`. The new reject-output methods are emitted from a SEPARATE list: `reject_outputs = [o for o in outputs if o.get("is_reject") or o.get("inner_join_reject")]`.

**Deviation required:** The existing row loop dispatches only `active_outputs`. The reject-output `evalOutput_<name>` methods are emitted but called from a separate Python-triggered script invocation, not from the same row loop (D-09 Option A). The compiled script body needs a `rejectMode` guard so the row loop skips reject helpers on the normal pass and skips non-reject helpers on the reject pass.
