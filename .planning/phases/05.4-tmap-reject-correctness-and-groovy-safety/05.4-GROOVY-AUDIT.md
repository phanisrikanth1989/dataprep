# Phase 05.4: Groovy-vs-Java Expression Embedding Audit

**Audited:** 2026-05-15 (HEAD: 9defcc91275dfbb1ad9903fea797bded5e1b157d)
**Method:** Static analysis of the Groovy 3.0 language specification vs the Java 11 language specification for the subset of syntax that appears in Talend tMap column / variable / filter / join-key expressions.
**Scope:** Every site in `src/v1/engine/components/transform/map.py` where Talend Java expression text is forwarded to the Groovy runtime -- either by being interpolated into a Groovy source line emitted by `_build_compiled_script`, or by being passed as a complete Groovy script body through `JavaBridge.executeTMapPreprocessing` (which calls `GroovyShell.parse(expression)` on each expression string and runs the resulting `Script`).

This audit underwrites the helper `Map._groovy_escape_expression` introduced in plan 05.4-05 (D-07). It enumerates every character class the helper considers, the chosen disposition for each, and the unit test that locks the disposition in.

## Character Class Differences

| Char Class | Java Semantics | Groovy Semantics | Difference | Handling | Test |
|---|---|---|---|---|---|
| `$` inside a double-quoted string literal | literal dollar character | GString interpolation trigger -- Groovy tries to resolve `$identifier` or `${expr}` after the `$` | YES -- causes `MissingPropertyException` or arbitrary identifier evaluation at runtime | escape `$` -> `\$` inside the string region only | test_groovy_escape_dollar_inside_string_literal |
| `$` outside a string literal (identifier / operator position) | legal Java identifier character (`$tmp`, `$$x`) | legal Groovy identifier character (`$tmp`, `$$x`) | None | leave alone | test_groovy_escape_dollar_in_identifier_not_escaped |
| Backslash escape sequences (`\n`, `\t`, `\r`, `\\`, `\"`) inside double-quoted strings | standard Java string escapes | identical Java-compatible escapes (Groovy GString accepts the same set) | None | leave alone -- the tokenizer consumes both characters of an escape sequence as a unit so it cannot mis-detect a string boundary | N/A -- behavioral test exercises a string containing `\"` and verifies `$` after it is still escaped (test_groovy_escape_handles_escaped_quote_then_dollar) |
| `'` single-quoted character literal (`'a'`) | `char` primitive literal | `java.lang.String` of length 1 (no `char` literal in Groovy) | Low risk: `Map`-marker pipeline rarely emits single-quoted strings, and a 1-char `String` is still assignable wherever Talend code expects `String`. A 1-char `String` is NOT assignable to a `char` field -- but Talend column schemas don't define `char` columns, only `String`. | leave alone | N/A |
| `"""..."""` triple-quoted strings | not supported before Java 15 (not emitted by Talend's Java codegen) | Groovy GString multi-line | Low risk: never appears in Talend `.item` expression text | leave alone (the tokenizer's `"` state-machine still parses each pair of `"` correctly even when triple-quoted regions are NOT present, which is the actual case) | N/A |
| `$/.../$` "dollar-slashy" strings | not legal Java syntax (would not be emitted by Talend) | Groovy-specific raw string with limited interpolation | Cannot appear in Java expression text | leave alone | N/A |
| Unicode escape sequences `\uXXXX` | resolved by the Java compiler before tokenization | same resolution rules at the Groovy parser stage | None | leave alone | N/A |
| Method / variable names that collide with Groovy-only reserved words (`in`, `def`, `trait`, `as`, `with`, `it`) | valid Java identifiers | Groovy keywords -- ambiguous in expression position | Low risk: Talend column / variable identifiers are constrained by the Talend UI which does not allow these as bare identifiers in column-name slots. They might appear inside method bodies as locals, but Talend-generated Java does not emit them. | leave alone; if encountered they would require backtick quoting which is a much larger transformation outside the scope of this helper | N/A |
| `;` statement terminator | required | optional | None (we always emit `;` ourselves in the wrapping Groovy line; the embedded expression body never ends with a free-standing `;`) | leave alone | N/A |
| Numeric literal suffixes (`123L`, `1.5f`, `1.0d`) | Java primitive type tags | same in Groovy | None | leave alone | N/A |
| `BigDecimal` / `BigInteger` literal forms (`123G`, `1.5G`) | not legal Java | Groovy-specific tag | Cannot appear in Java expression text emitted by Talend | leave alone | N/A |

## Embedding Site Inventory

There are two distinct Groovy-evaluation paths in `map.py`. Both consume the expression text as Groovy source, so both need `_groovy_escape_expression`.

**Path A: compiled-script embed** -- `_build_compiled_script` interpolates the expression text into Groovy source lines that wrap a `for` loop. The resulting script body is compiled once with `JavaBridge.compileTMapScript`.

**Path B: bridge-eval embed** -- expressions go through `Map._bridge_eval` (or `_evaluate_with_bridge` directly) which forwards them to `JavaBridge.executeTMapPreprocessing`. The Java side calls `compileShell.parse(expression)` on each expression string. The expression IS the Groovy script.

Both paths require GString escaping because both reach `GroovyShell.parse`.

## Decision Log

- `$` in string literals: ESCAPE (`$` -> `\$`). This is the locked minimum per D-07. Without this, Talend expressions like `"Total: $" + amount` cause Groovy to attempt interpolation of the bare `$"` GString, which raises a parse error.
- `$` in identifiers: LEAVE ALONE. Groovy allows `$` in identifiers (e.g. `$tmp`, `var$1`). The tokenizer distinguishes "inside double-quoted string" vs "outside" via a single boolean state so this determination is unambiguous.
- Backslash sequences inside string literals: LEAVE ALONE. Java and Groovy use the same `\n`, `\t`, `\r`, `\\`, `\"` set. The tokenizer consumes both bytes of an escape sequence as a unit so a `\"` inside a string never falsely closes the string.
- Single-quoted character literals: LEAVE ALONE. Talend column types are `String`-flavored; a 1-char Groovy `String` is type-compatible.
- All other character classes (triple-quoted, dollar-slashy, Groovy keywords as identifiers, numeric suffixes, Unicode escapes): LEAVE ALONE. None of these classes can be produced by Talend's Java code emission for the tMap component, so the helper does not need to transform them. Should a real-world failure surface a counter-example, this audit table is the place to add a new row plus a behavioral test.

## Groovy-vs-Java Confidence Notes

The matrix above is the result of a Groovy-vs-Java character-class comparison constrained to the subset of Java syntax that Talend Open Studio emits for tMap expressions. The Java emission surface for tMap is narrow (variable refs, column refs `rowN.col`, method calls, ternary, arithmetic, string concatenation, regex, plus the `TalendString` / `TalendDate` routine helpers). Constructs the audit deliberately did NOT inspect because they cannot appear in that surface: annotations, lambdas, switch expressions, type parameters, anonymous classes, `instanceof` pattern bindings.
