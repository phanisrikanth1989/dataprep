# Phase 05.3 Plan 03: Computed-Key Join Methods Summary

Two new join methods fixing issues 2a and 2b (single-side computed join keys
producing 0 matches). O(n+m) hashmap parity via batch-eval then pd.merge,
replacing the O(n*m) cross-table fallback.

## What Was Built

### Task 1 (TDD): `_join_main_side_computed_key` + `_join_lookup_side_computed_key` + dispatch wiring

**New method `_join_main_side_computed_key`:**
- Batch-evaluates all main-side join key expressions ONCE on `joined_df` via
  `_bridge_eval(joined_df, exprs, joined_lookup_names)` (D-04: passes full
  prior-join list so `row3.col` refs resolve when row3 was joined earlier).
- Adds computed values as temporary `__jk_main_N__` columns on a copy of `joined_df`.
- Delegates to standard `pd.merge` using temp columns as left keys (O(n+m)).
- Mirrors full `_join_equality` structure: matching mode dedup, null key pre-filter,
  inner-join reject routing (MAP-02/MAP-03), column cleanup.
- Drops temp columns and `__dup__` columns from output; removes them from rejects too.

**New method `_join_lookup_side_computed_key`:**
- Symmetric counterpart. Batch-evaluates lookup-side key expressions on `lookup_df`.
- Calls `_evaluate_with_bridge(lookup_df, exprs, lookup_name, [])` directly
  (NOT `_bridge_eval`) because the bridge must see the lookup as its "main" table
  (`lookup_name` as `main_table_name`) with no joined-with-others context.
  A comment explains this exception to the D-04 rule.
- Adds `__jk_lookup_N__` temp columns, prefixed after `_prefix_lookup_columns`,
  merges on them as right keys.
- Same matching mode / null prefilter / reject routing / cleanup as main-side method.

**Dispatch wiring in `_process`:**
- All-main-or-context localities + computed exprs: removed `# TODO(05.3-03)` +
  cross-table fallback, replaced with `_join_main_side_computed_key(...)` call.
- All-lookup-side localities + computed exprs: same replacement with
  `_join_lookup_side_computed_key(...)`.
- Zero `TODO(05.3-03)` markers remain in `map.py`.

## Deviations from Plan

None. Plan executed exactly as written. The one design note:

**`_join_lookup_side_computed_key` calls `_evaluate_with_bridge` directly (not `_bridge_eval`).**
The plan's action item 2 explicitly called out this as a "rare case" and provided
the exact rationale: `_bridge_eval` always uses the config's main_name; for lookup-side
eval the bridge needs to receive the lookup's own name as `main_table_name`. A comment
in the code explains why this is intentional.
