# Phase 05.3 Plan 04: Chunked Cross-Product + Issue 2c Fix Summary

Memory-bounded chunked cross-product (D-05) and issue 2c crash fix.
`_join_cross_table` now delegates to `_chunked_cross_product` which iterates
main_df in chunks instead of materializing the full N*M cross-product.

## What Was Built

### Task 1 (TDD): Chunked cross-product + dispatch fix + config aliasing

**New `_compute_cross_chunk_size(lookup_rows)` static method:**
- Formula: `max(100, min(10_000, 100_000_000 // max(1, lookup_rows)))`
- Bounds peak intermediate-frame cells to ~100M regardless of lookup size
- Zero lookup rows treated as 1 (auto-tune to 10000 chunk size)

**New `_chunked_cross_product(main_df, lookup_df, match_expr, ...)` method:**
- Iterates main_df in auto-tuned (or config-overridden) chunks
- For `match_expr=None`: pure cartesian, no bridge call
- For `match_expr` set: evaluates via `_bridge_eval` with D-04 joined_lookup_names
- Empty lookup_df: returns empty frame immediately
- Respects `cross_join_chunk_size` config key (override) or auto-tune fallback

**New `_chunked_cross_product_with_keys(...)` method:**
- Inner logic of `_join_cross_table` for non-empty join_keys (two-sided)
- Iterates in chunks, evaluates join key expressions via `_bridge_eval` per chunk
- Builds match mask per chunk from key expr == prefixed lookup column comparison

**Rewritten `_join_cross_table`:**
- No longer materializes full main x lookup cross-product
- Delegates to `_chunked_cross_product_with_keys` (non-empty keys) or
  `_chunked_cross_product` (empty keys / pure cartesian)
- Lookup columns prefixed before cross-product (no double-prefix issue)
- INNER_JOIN reject routing preserved

**Issue 2c fix in `_process` dispatch:**
- Empty join_keys + `activate_filter=True` previously crashed with
  `list index out of range` when `_apply_filter` received a two-sided expression
  (referencing both `row1.col` and `row2.col`) applied to just the lookup DataFrame
- Fix: before pre-filtering, classify the filter locality
  - `LOCALITY_LOOKUP_SIDE`: safe to pre-filter (reduces cross-product size)
  - All others (two-sided, main-side, context): deferred to `_chunked_cross_product`
    as `match_expr` -- evaluated on the actual cross-product row pairs
- The crash was that `_classify_key_locality` was never called on an empty
  join_keys list; now the empty-keys path has its own explicit dispatch branch

**Config aliasing (D-05):**
- `output_chunk_size` added as canonical config key for compiled output chunk size
- `rows_buffer_size` honored as legacy alias for back-compat (no re-conversion needed)
- Default: 50,000 rows (was 2,000,000 in original code, already fixed in prior phases)
- `cross_join_chunk_size` config key for cross-product chunk size override

**Bridge fix: pandas 3.0 StringDtype multi-batch IPC bug:**
- `pd.concat` with string columns in pandas 3.0 produces `StringDtype` arrays
- `pa.Table.from_pandas` converts these to chunked arrays → multiple IPC record batches
- Java `executeCompiledTMap` reads only the first batch (`loadNextBatch()` once)
- Fix: `arrow_table.combine_chunks()` before `write_table` in `_df_to_arrow_bytes`
- This affected any compiled-path execution with cross-product DataFrames from chunked concat

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `comp.config = {}` when calling internal methods directly**
- **Found during:** Task 1 (test implementation)
- **Issue:** `BaseComponent` sets `self.config = {}` at init; only populates from `_original_config` at `execute()` start. Tests calling `_chunked_cross_product` directly got `KeyError: 'inputs'` from `_bridge_eval`.
- **Fix:** Added `_make_comp_with_live_config()` helper that manually populates `comp.config` for direct-method unit tests (equivalent to what `execute()` does).
- **Files modified:** `tests/v1/engine/components/transform/test_map.py`
- **Commit:** edb5ccc

**2. [Rule 1 - Bug] pandas 3.0 StringDtype produces multi-batch Arrow IPC stream**
- **Found during:** Task 1 (test_b11 live-bridge test)
- **Issue:** `pd.concat` of cross-product DataFrames in pandas 3.0 produces `StringDtype` columns. `pa.Table.from_pandas` converts these to chunked arrays serialized as 2 record batches. Java `executeCompiledTMap` calls `reader.loadNextBatch()` once, reading only the first batch (80 of 120 rows).
- **Fix:** Added `arrow_table = arrow_table.combine_chunks()` in `_df_to_arrow_bytes` before `write_table`. Forces single-batch IPC stream regardless of column dtype.
- **Files modified:** `src/v1/java_bridge/bridge.py`
- **Commit:** edb5ccc

**3. [Rule 2 - Missing correctness] Test 6 redesigned to avoid compiled path**
- **Found during:** Task 1 (test_6 implementation)
- **Issue:** `{{java}}` in filter expression forces compiled path, but the compiled path needs a proper mock for `compile_tmap_script` and `execute_compiled_tmap_chunked`. For a unit test verifying crash-fix (not output correctness), using a no-marker filter was cleaner.
- **Fix:** Test 6 uses a no-marker filter expression (dispatch classification is independent of the marker). The live-bridge test (test_b10) covers the real `{{java}}` filter scenario end-to-end.
- **Files modified:** `tests/v1/engine/components/transform/test_map.py`
