# Phase 05.3: tMap Join Correctness, Marker Rule, and Filepath Expression - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix six bugs reported against tMap on `feature/engine-restructure` (and one related bug in
`file_output_delimited`). All six were reproduced end-to-end via real `.item` -> converter ->
engine pipelines (samples persisted in `/tmp/repro/Job_*.item` during brainstorming).

The unifying root cause is that the engine-restructure rewrite introduced a per-column
hybrid path in tMap's non-compiled flow that conflates "is this expression simple?" with
"can it be evaluated without the bridge?". Talend's actual semantics (verified against
Talaxie `tMap_main.inc.javajet`) classify by **expression locality**, not complexity.

Fixes target: tMap engine component (locality-based join classifier, chunked cross-product,
universal `{{java}}` marker rule), tMap-related shared infrastructure
(`_evaluate_with_bridge` plumbing for `joined_lookups`), and the converter+engine for
`file_output_delimited.filepath`.

Out of scope: Audit/migrate other expression-evaluating components (filter_rows,
file_input_*, file_output_xml/excel/json, java_row, python_row, swift_*, etc.) for the
same marker rule -- tracked as follow-up phase. Performance benchmarks at production
scale -- Phase 16 (Integration Testing & Performance) covers this. Java bridge JAR
itself -- only Python-side calls change.

</domain>

<decisions>
## Implementation Decisions

### D-01 -- LOCKED -- Universal `{{java}}` marker rule is the only signal for evaluation routing
For every config field that may contain an expression:
- If value starts with `{{java}}`: Java bridge is **MANDATORY**. If unavailable at startup,
  raise ConfigurationError. Send the (stripped) expression to the bridge for evaluation.
- If no marker: pure-Python evaluation. If the expression isn't valid Python, raise
  ExpressionError with a clear message.

**No per-column hybrid.** A given evaluation path is either all-bridge or all-Python.
**No silent fallbacks.** Marker present + bridge missing = hard error, not empty cells.

The converter's `_java_expr` wraps every non-empty tMap expression with `{{java}}` today
-- that stays as-is. The change is engine-side: stop trying the bridge per-column and
silently degrading to None.

### D-02 -- LOCKED -- tMap engine dispatch driven by marker presence anywhere
Replace the existing `_has_java_expressions(outputs_config)` + simple/compiled fork with:

- ANY of {output cols, output filter, variables, join key exprs, lookup filters, main filter}
  has a `{{java}}` marker -> use compiled Groovy path (existing `_evaluate_outputs_compiled`
  + variable evaluation inside the script).
- NO markers anywhere -> use NEW Python-eval path (mirrors `py_map.py` patterns).

The existing dead constant `_DEFAULT_CHUNK_THRESHOLD = 100000` (line 57 in current map.py)
is removed during this refactor.

### D-03 -- LOCKED -- Locality-based join classifier replaces `_classify_join_type`
For each join key expression (after stripping `{{java}}`):

| Refs to | Classification |
|---|---|
| Nothing (literal, context, globalMap only) | `context` |
| Only main side or only previously-joined lookups (anything in `joined_df`) | `main_side` |
| Only the current lookup side | `lookup_side` |
| Both main and current lookup | `two_sided` |

**Dispatch logic:**
- `join_keys` empty + `activate_filter` + filter is two-sided -> chunked cross-product, filter
  is the match condition.
- `join_keys` empty + `activate_filter` + filter is lookup-side-only -> filter the lookup,
  then chunked cross-product.
- `join_keys` empty + no filter -> chunked cross-product (pure cartesian) with warning.
- All keys are `context` or `main_side` -> batch-eval main-side keys ONCE on `joined_df`
  (passing `joined_lookups` to bridge so prior-lookup refs resolve), add as temp columns,
  standard `pd.merge`. O(n+m). [Talend hashmap path equivalent.]
- All keys are `lookup_side` -> batch-eval lookup-side keys ONCE on `lookup_df`, standard
  `pd.merge`. O(n+m).
- Any key is `two_sided` -> chunked cross-product + per-pair eval.

### D-04 -- LOCKED -- Bridge calls always pass `joined_lookups` list
Today `_join_cross_table` (and a few other call sites) pass only `[current_lookup_name]`
to `_evaluate_with_bridge`. Change all bridge call sites in tMap to pass the full
`joined_lookups` list so expressions like `row3.col.trim()` (where `row3` is a previously-
joined lookup) resolve correctly. This single change is what closes issue 3.

### D-05 -- LOCKED -- Memory-bounded chunked cross-product
| Knob | Default | Where used |
|---|---|---|
| `output_chunk_size` (config; alias `rows_buffer_size` for back-compat) | **50,000** (was 2,000,000) | Compiled output script execution chunking |
| `cross_join_chunk_size` (auto-tuned; override via config) | `max(100, min(10_000, 100_000_000 // lookup_rows))` | NEW: cross-product paths (filter-only join + true two-sided joins) |

Cross-product implementation pattern:
```python
chunk_size = max(100, min(10_000, 100_000_000 // max(1, len(lookup_df))))
result_chunks = []
for start in range(0, len(main_df), chunk_size):
    chunk = main_df.iloc[start:start + chunk_size]
    chunk_cross = pd.merge(chunk, lookup_df, how='cross')
    eval_results = evaluate(filter_or_join_expr, chunk_cross, ...)
    result_chunks.append(chunk_cross[eval_results])
result = pd.concat(result_chunks, ignore_index=True)
```

This mirrors Talend's row-by-row streaming behavior with bounded peak memory (~100M cells per
chunk regardless of lookup size).

### D-06 -- LOCKED -- Converter marks file_output_delimited filepath via mark_java_expression
- `src/converters/talend_to_v1/components/file/file_output_delimited.py:62`: replace the
  plain string assignment with `ExpressionConverter.mark_java_expression()`. This already
  detects `+` operators, ternaries, `.method()` calls, routines and prefixes with
  `{{java}}` only when needed.
- Engine `src/v1/engine/components/file/file_output_delimited.py`: after
  `context_manager.resolve_dict` substitutes context refs, if `filepath` still starts
  with `{{java}}`, send to bridge for evaluation. Apply the same treatment to other
  expression-bearing string params (`streamname`, etc.).
- For literal context-only filepaths (`/data/output.csv`), `mark_java_expression`
  returns the string unchanged -- no bridge round-trip needed.

### D-07 -- OPEN -- Empty lookup -> downstream nullable=false crash (issue 4)
Empty lookup is correctly skipped by Map; downstream component fails on
`validate_schema` because lookup-derived columns are NULL but declared `nullable=false`
in the .item-derived schema. Talend itself does not enforce `nullable=false` at runtime.

Three options, decision deferred:
1. **Loosen `validate_schema` globally to warn-not-raise on `nullable=false` violations.**
   One-line change in `base_component.py:1050-1053`. Matches Talend exactly. Cross-cutting
   policy change but minimal code.
2. **Surgical Map-side fix.** When skipping a lookup, explicitly add the lookup's columns
   to `joined_df` as all-NaN. Map manages its own output. Doesn't touch downstream
   validation policy. But Map can't easily override schema metadata declared by the
   converter, so downstream nullable=false enforcement may still fire.
3. **Add config flag `strict_nullability` (default `false` matching Talend).** Job-level
   toggle. Most flexible but most API surface.

The planner should produce a sub-plan for issue 4 ONLY after the user picks an option.
Mark this clearly as a blocker for the issue-4 sub-plan, not for the rest of the phase.

  `_join_main_side_computed_key`, etc.).
- Whether to wholesale-replace `_evaluate_outputs_simple` or leave it for non-tMap callers
  (no current non-tMap caller exists, so wholesale replace is acceptable).
- The Python evaluator for the no-marker path can either be inlined in the new method or

- Test harness organization (group by issue or by code area).
- Whether the bridge's hard-fail at startup (D-01) raises ConfigurationError vs a new

<code_context>
## Existing Code Insights

### What's already correct in engine-restructure
- Compiled Groovy path (`_evaluate_outputs_compiled` + `_build_compiled_script`) handles
  all expression types correctly when bridge is available -- it generates `row[N] = expr;`
  per output column, has a shared `Var` HashMap for chained variables, and produces
  correct results for literals, concatenations, ternaries, routines.
- `_evaluate_with_bridge` correctly calls `execute_tmap_preprocessing` with Arrow
  serialization. The bug is at the call sites (which `lookup_names` they pass), not in
  the helper itself.
- Matching modes (UNIQUE_MATCH, FIRST_MATCH, LAST_MATCH, ALL_MATCHES), lookup modes
  (LOAD_ONCE, RELOAD_AT_EACH_ROW, CACHE_OR_RELOAD), inner-join reject routing, and
  catch-output reject routing all work correctly. Don't touch.
- Schema validation in `base_component.py` works correctly for the cases it's intended
  for; the question is whether to loosen it for the empty-lookup case (issue 4 open).

### What needs replacing
- `_classify_join_type(join_keys: list[dict]) -> str` -- replace with
  `_classify_join_keys_locality(join_keys, main_name, joined_lookups) -> list[str]`.
- `_evaluate_outputs_simple(...)` -- replace with a Python-eval path that processes ALL
  outputs in one shot using a safe Python evaluator (mirror `py_map.py` patterns).
- `_join_cross_table` -- rewrite to use chunked iteration instead of full-cross-product
  materialization.
- `_join_equality` -- extend to handle `main_side` and `lookup_side` complex expressions
  (compute key once via batch-eval, add as temp column, then merge).
- `_has_java_expressions` -- expand to check ALL marker-bearing fields (variables, filters,
  join keys, main filter, lookup filters), not just output columns.

### What needs adding
- New `_join_main_side_computed_key` method (mirrors `_join_equality` but with batch-eval
  step before merge).
- New `_join_lookup_side_computed_key` method (symmetric).
- New `_chunked_cross_product` helper used by `_join_cross_table` and the filter-only
  path.
- Auto-tune helper `_compute_cross_chunk_size(lookup_rows: int) -> int`.
- New config keys: `output_chunk_size` (replaces/aliases `rows_buffer_size`),
  `cross_join_chunk_size` (auto-tuned, override-able).
- Engine-side hard-fail check at component init: if any expression in this component is
  marked `{{java}}` and the bridge is unavailable, raise `ConfigurationError` with a
  clear message.

### Integration points to watch
- The compiled path's chunk size is read from `rows_buffer_size` config (line 1316). Need
  to keep back-compat: if old jobs have `rows_buffer_size: "2000000"` in their JSON, the
  engine should still accept it and use it. New default for missing key drops to 50,000.
- `_evaluate_with_bridge` signature is `(df, expressions, main_name, lookup_names)`. All
  callers in tMap must pass `joined_lookup_names` (the full list of already-joined
  lookups, not just `[current_lookup]`).
- The Python-eval path needs access to: row data (joined_df columns), `Var.*` map (built
  sequentially from variable evaluation), context, globalMap. `py_map.py:_build_namespace`
  already shows the pattern.
