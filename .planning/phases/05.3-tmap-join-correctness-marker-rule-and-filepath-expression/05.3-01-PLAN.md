<objective>
Implement the universal `{{java}}` marker rule in tMap (D-01) and rewire the
dispatch logic to use marker-presence-anywhere as the only signal (D-02).
This plan is the **infrastructure** for the rest of the phase: every later plan
depends on the new dispatch boundary being in place.

Purpose:
- Eliminate the per-column hybrid path that conflates "is this expression
  simple?" with "can it be evaluated without the bridge?". That conflation is
  the root cause of issue 5 (chained variables silently emit empty cells when
  outputs are simple refs but variables need the bridge).
- Convert silent-empty failures (issue 1, issue 5) into either correct results
  (when the bridge is available) or a hard-fail at startup (when it is not).

Output:
- New `_has_any_java_marker(self) -> bool` on Map that scans ALL marker-bearing
  fields (variables, output filters, join keys, main filter, lookup filters,
  output column expressions).
- Hard-fail check wired into `_validate_config` (or earliest available hook) so
  marker-present + bridge-missing raises ConfigurationError before any data is
  processed.
- Dispatch logic in `_process` and `_evaluate_outputs` updated to use the new
  universal scanner. The OLD `_has_java_expressions(outputs_config)` is removed.
- The dead constant `_DEFAULT_CHUNK_THRESHOLD = 100000` (map.py:57) is removed.

This plan deliberately leaves the no-marker dispatch branch routing to a
sentinel (e.g. `raise NotImplementedError("Python-eval path lands in plan 05")`
or temporarily delegate to the existing simple path). Plan 05 replaces that
sentinel with the real Python-eval implementation. This split keeps each plan
focused and within the context budget.
</objective>
