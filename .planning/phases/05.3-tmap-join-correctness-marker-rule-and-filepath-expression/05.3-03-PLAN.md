<objective>
Add the two new join methods that handle the "all keys single-side, at least
one is computed" cases (D-03). These are the Talend-equivalent of the hashmap
join with a key-derivation step: compute keys once, then `pd.merge` -- O(n+m)
not O(n*m).

Purpose:
- Issues 2a (`row1.region.trim()` join key) and 2b (routine call as join key)
  currently fall through to `_join_cross_table` which is O(n*m) and (per the
  bug in current code) silently produces empty results when the bridge call
  drops `joined_lookups` plumbing. Plan 02 fixed the plumbing AND inserted a
  temporary cross-table fallback for these cases. This plan replaces the
  fallback with the real O(n+m) implementations.

Output:
- New `_join_main_side_computed_key(joined_df, lookup_df, join_keys, main_name, lookup_name, joined_lookup_names, ...) -> tuple[pd.DataFrame, ...]` method.
- New `_join_lookup_side_computed_key(joined_df, lookup_df, join_keys, main_name, lookup_name, joined_lookup_names, ...) -> tuple[pd.DataFrame, ...]` method.
- Both methods are wired into the dispatch in `_process` -- the TODO(05.3-03)
  markers from plan 02 are replaced with real routing.

Both new methods reuse the existing `_join_equality` helpers (matching mode
dedup, null-key prefilter, lookup column prefixing, inner-join reject routing)
to avoid duplication.
</objective>
