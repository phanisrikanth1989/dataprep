# Phase 05.3 Verification Report

## 1. Goal Recap

Phase 05.3 targeted six reproducible bugs in tMap on `feature/engine-restructure` plus one related
`tFileOutputDelimited` filepath bug. The unifying root cause was a per-column hybrid evaluation path
that conflated expression complexity with routing, producing silent empty cells, O(n*m) joins for
single-side transforms, a crash on empty `join_keys` + filter, silent zero-match results for
lookup-to-lookup transformed keys, and a broken filepath expression that wrote files with literal `+`
operators in the path. The phase introduced a universal `{{java}}` marker rule, locality-based join
classification, memory-bounded chunked cross-product, a pure-Python eval path for no-marker configs,
and bridge-backed filepath evaluation. D-07 (Issue 4: empty-lookup downstream nullable=false crash)
was explicitly deferred to a follow-up phase by user decision on 2026-05-15.

### D-05 â€” Memory-bounded chunked cross-product

**Status: PASS**

pandas 3.0 StringDtype multi-batch deserialization bug.

Perf test: `test_map_05_3_perf.py::TestChunkedCrossMemoryBound` (4 tests, `@pytest.mark.slow`)

## 3. Per-Issue Verification (Issues 1, 2a, 2b, 2c, 3, 5, 6)

### Issue 1 â€” Literal output column empty with bridge OFF

**Status: PASS**

Root cause: `_evaluate_outputs_simple` silently returned None for non-bare-column expressions when
bridge was unavailable. Fix: `_evaluate_outputs_simple` deleted; D-01 raises `ConfigurationError`
immediately; no silent fallback exists.

Test: `TestIssue1CleanBaseline.test_issue_1_clean_baseline` â€” runs `clean.json` (job with salary >
60,000 filter, `"L"` literal output column `side`). Asserts `len(rows) == 6` and all rows have

Additionally, `TestMarkerPresentNoBridgeHardFail.test_marker_present_no_bridge_hard_fails`
verifies the hard-fail (D-01 sentinel) explicitly without a bridge.

### Issue 2a â€” Main-side `.trim()` join key routed to O(n*m) cross-table

**Status: PASS**

Root cause: `_classify_join_type` routed any non-simple expression to cross-table.
Fix: `_classify_key_locality` classifies `row1.country_code.trim()` as `MAIN_SIDE` (computed);
dispatches to `_join_main_side_computed_key` (O(n+m) batch-eval then merge).

Test: `TestIssue2aMainSideTrim.test_issue_2a_main_side_trim` â€” runs `join_trim.json` with
`{{java}}row1.country_code.trim()` as join key. Asserts `len(rows) == 6`.

### Issue 2b â€” Routine call as join key routed to cross-table

**Status: PASS**

Same root cause and fix as 2a. `routines.StringHandling.UPCASE(row1.country_code)` is classified
as `MAIN_SIDE` (computed).

Test: `TestIssue2bRoutineJoinKey.test_issue_2b_routine_join_key` â€” runs `join_routine.json` with
`{{java}}routines.StringHandling.UPCASE(row1.country_code)` as join key. Asserts `len(rows) == 6`.

### Issue 2c â€” Empty `join_keys` + filter crashes with `list index out of range`

**Status: PASS**

Root cause: empty `join_keys` fell through to `_join_equality` which called `pd.merge(left_on=[],
right_on=[])` then indexed into an empty list.
Fix: empty-keys path in `_process()` (lines 479-533) classifies filter locality first; two-sided
or main-side filters become `match_expr` in `_chunked_cross_product`; lookup-side filters are
pre-applied; no `pd.merge(left_on=[], right_on=[])` ever called.

Fixture `filter_join.json` has `join_keys=[]` and
`{{java}}row1.country_code.trim().equals(row2.country_code)` as filter. Confirmed above via
`python3` inspection.

Test: `TestIssue2cFilterJoinNoCrash.test_issue_2c_filter_join_no_crash` â€” runs `filter_join.json`,
asserts no crash, asserts `len(rows) >= 1`.

### Issue 3 â€” Lookup-to-lookup transformed key produces 0 matches

**Status: PASS**

Root cause: `_join_cross_table` passed only `[current_lookup]` to `_evaluate_with_bridge`; the
bridge had no RowWrapper for `row2`, causing `MissingPropertyException` and null key â†’ 0 matches.
Fix: D-04 `_bridge_eval` wrapper always passes `joined_lookup_names`; verified that both
`_chunked_cross_product` (line 1527-1530) and `_join_main_side_computed_key` (line 1081) use it.

Fixture `l2l_trim.json` has two lookups: row2 joined on `{{java}}row1.country_code`, row3 joined
on `{{java}}row2.region.trim()`. Confirmed above.

Test: `TestIssue3LookupToLookupTrim.test_issue_3_lookup_to_lookup_trim` â€” runs `l2l_trim.json`,
asserts `len(rows) > 0` and `continent` column is populated.

### Issue 5 â€” Chained intermediate variables produce empty outputs

**Status: PASS**

Root cause: `_evaluate_variables` (simple path) called bridge once per variable without shared `Var`
binding; `Var.var1` was undefined when `Var.var2 = Var.var1 + "_USD"` executed.
Fix: D-01/D-02 â€” any `{{java}}` marker on a variable forces the compiled Groovy path which has a
shared `Var` HashMap. The `_evaluate_variables_py` method handles the no-marker case with `_VarBag`
for live sequential mutation.

Fixture `vars_simple.json` has three `{{java}}`-marked variables (var1, var2, var3) and all output
columns also marked, so it uses compiled path where chaining works correctly.

Test: `TestIssue5ChainedVarsSimpleOutputs.test_issue_5_chained_vars_simple_outputs` â€” runs
`vars_simple.json`, finds John Smith's row (salary=85000), asserts `v2 == "85000_USD"` and
`v3 == "LONG"`.

### Issue 6 â€” `file_output_delimited` filepath as expression writes literal `+` operators

**Status: PASS**

Root cause: converter read FILENAME as plain string; engine treated result as literal path after
context substitution, leaving `+` operators in the path.
Fix: Converter uses `ExpressionConverter.mark_java_expression()`; engine calls
`_resolve_java_expr_param()` which sends `{{java}}`-marked values to the bridge.

Fixture `filepath_expr.json` has `{{java}}context.OUT_DIR + context.OUT_NAME + context.OUT_EXT`
as the filepath. Confirmed above.

Test: `TestIssue6FilepathExpr.test_issue_6_filepath_expr` â€” sets `OUT_DIR`, `OUT_NAME`, `OUT_EXT`
context vars, runs the job, asserts file exists at `tmp_path/output_via_expr.csv`, asserts no file
with literal `+` in its name exists, asserts output file is non-empty.
