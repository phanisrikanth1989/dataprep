# Phase 05.3 Plan 02: Locality-Based Join Classifier + Bridge Call Plumbing Summary

Locality-based per-key join classifier (`_classify_key_locality`) replacing the coarse
`_classify_join_type`, plus `_bridge_eval` single-source-of-truth wrapper for D-04
compliance. Fixes issue 3 (lookup-to-lookup with transformed key producing 0 matches).

## What Was Built

### Task 1 (TDD): Locality classifier + _bridge_eval + dispatch rewrite + call site audit

**New constants (module-level, near `_JAVA_MARKER`):**
```python
_LOCALITY_CONTEXT = "context"       # only context/globalMap/Var refs
_LOCALITY_MAIN_SIDE = "main_side"   # only main + previously-joined lookups
_LOCALITY_LOOKUP_SIDE = "lookup_side"  # only current lookup
_LOCALITY_TWO_SIDED = "two_sided"   # both main and current lookup
```

**New method `_classify_key_locality(self, expr, main_name, current_lookup, joined_lookup_names) -> str`:**
- Strips `{{java}}` marker before classification
- Uses `_ROW_REF_PATTERN.findall()` returning tuples; extracts table names
- Filters out `context`, `globalMap`, `Var` (non-row refs)
- `joined_df` = main + all previously-joined lookups; any ref to those is `MAIN_SIDE`
- Correctly handles joined lookup refs (test 10: `row3.col` with `row3` in joined list -> `MAIN_SIDE`)

**New method `_bridge_eval(self, df, exprs, joined_lookup_names) -> dict`:**
- Single-source-of-truth wrapper around `_evaluate_with_bridge`
- Uses `self.config["inputs"]["main"]["name"]` for `main_name`
- Guarantees joined_lookup_names is always the full prior-join list per D-04

**Deleted `_classify_join_type`:**
- Removed entirely (only a comment reference remains)
- Replaced at the single dispatch call site with locality list comprehension

**Dispatch rewrite in `_process`:**
- Empty `join_keys` -> `_join_cross_table` (cartesian)
- Any `TWO_SIDED` locality -> `_join_cross_table`
- All `CONTEXT` or `MAIN_SIDE` + simple column refs -> `_join_equality`
- All `CONTEXT` or `MAIN_SIDE` + computed exprs -> `_join_cross_table` + `# TODO(05.3-03): route to _join_main_side_computed_key`
- All `LOOKUP_SIDE` + simple refs -> `_join_equality`
- All `LOOKUP_SIDE` + computed -> `_join_cross_table` + `# TODO(05.3-03): route to _join_lookup_side_computed_key`
- Mixed -> `_join_cross_table`

**Issue 3 fix in `_join_cross_table`:**
- Added `joined_lookup_names: Optional[list[str]] = None` parameter
- Changed bridge call from `[lookup_name]` to `(joined_lookup_names or []) + [lookup_name]`
- All 5 dispatch call sites updated to pass `joined_lookup_names`

**Bridge call site audit:**
- `_join_cross_table`: FIXED (the issue 3 bug) - now passes full list via `_bridge_eval`
- `_evaluate_variables`: fixed to use `_bridge_eval`
- `_evaluate_outputs_simple` per-column fallback: fixed to use `_bridge_eval`
- `_apply_output_filter`: fixed to use `_bridge_eval`
- `_apply_filter`: INTENTIONALLY kept on `_evaluate_with_bridge` directly -- this method
  is called for both main-table and lookup-table filtering with different `table_name` values.
  D-04 is addressed by the callers (map.py:385-388 already passes full `joined_lookup_names`).

**1. [Rule 1 - Bug] _apply_filter reverted from _bridge_eval to preserve table_name semantics**
- **Found during:** GREEN phase (running live-bridge tests)
- **Issue:** Changing `_apply_filter` to use `_bridge_eval` broke `RELOAD_AT_EACH_ROW` tests because `_apply_filter` is called for both main-table and lookup-table filtering.
