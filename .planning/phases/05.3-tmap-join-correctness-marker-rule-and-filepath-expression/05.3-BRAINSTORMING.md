## Background

Six bugs reported by testers against the tMap engine and one related file-output filepath bug.
Reproductions performed against `feature/engine-restructure`. The `feature/map_issues_debug` branch
made partial attempts at fixes (computed-key join, variables in `_has_java_expressions`,
`$` in DQ strings) â€” those attempts inform the design but are NOT to be cherry-picked.

The unifying root cause across most issues is that **engine-restructure introduced a per-column
hybrid path** in the tMap simple/non-compiled flow that conflates "is this expression simple?"
with "can it be evaluated without the bridge?". Talend's actual semantics (verified against
Talaxie `tMap_main.inc.javajet`) are different.

## The six issues

### Issue 1 â€” hardcoded literal in output column comes through as empty
- **Reproduced** with `Job_clean.item` containing output column `<mapperTableEntries name="side" expression="&quot;L&quot;" .../>`.
- With Java bridge **ON**: works correctly (compiled Groovy path produces `"L"`).
- With Java bridge **OFF**: ALL non-bare-column expressions return empty (literals, concatenations, ternaries, output filters). Engine logs `Cannot evaluate expression for column 'side' in output 'out' -- Java bridge not available` and silently writes None.
- **Root cause:** `_evaluate_outputs_simple` (`src/v1/engine/components/transform/map.py:1355-1436`) tries the bridge for any expression beyond `table.column` / `Var.x`. When the bridge is None, it returns None per column.

### Issue 2 â€” joins with routines / `.trim()` / `.replaceAll()` / filter expression go through cross-table O(nĂ—m)
- **Reproduced 3 sub-cases:**
  - 2a: join key `row1.country_code.trim()` â†’ `WARNING [tMap_2] Cross-table join with 'row2' -- O(n*m) evaluation` (correct result, slow path)
  - 2b: join key `routines.StringHandling.UPCASE(row1.country_code)` â†’ same cross-table path
  - 2c: `join_keys=[]` + filter as match condition (`row1.x.equals(row2.y)`) â†’ **CRASH** `list index out of range` from `pd.merge(left_on=[], right_on=[])`
- **Root cause:** `_classify_join_type` (`map.py:701-724`) only recognizes bare `table.column` as `_JOIN_EQUALITY`. Anything else falls to `_JOIN_CROSS_TABLE` which materializes the full pandas cross product. Empty `join_keys` falls through to `_JOIN_EQUALITY` and crashes.
- **Memory risk:** `_join_cross_table` materializes the full nĂ—m DataFrame in memory. 1M Ă— 100K â†’ OOM. Talend (verified against Talaxie javajet) iterates row-by-row instead.

### Issue 3 â€” lookup-to-lookup joins not working (transformed keys silently produce 0 matches)
- **Reproduced** via `Job_l2l_trim.item` with row3 join key `row2.region.trim()` (row2 was the prior lookup).
- Bare `row2.region` works (resolved against joined_df after row2 join).
- `row2.region.trim()` goes to cross-table path, **and silently returns 0 matches** because `_join_cross_table` calls `_evaluate_with_bridge` with `lookup_names=[current_lookup]` only â€” bridge has no RowWrapper for `row2`, throws MissingPropertyException internally, eval returns null for the join key, no rows match, LEFT_OUTER fills with NaN.
- **Same root cause as issue 2 + missing `joined_lookups` plumbing.**

### Issue 4 â€” empty lookup â†’ downstream crashes (Talend ignores)
- **Reproduced** with `country_lookup_empty.csv` (header only).
- Map correctly skips empty lookup with `WARNING [tMap_2] Lookup 'row2' is empty, skipping`.
- Map output rows have NULL for `country` and `region` (which referenced `row2.country_name` and `row2.region`).
- **Crash happens in tLogRow downstream**: `validate_schema` (`base_component.py:1050`) raises `DataValidationError: Column 'country' has NULL values but is not nullable` because the .item declared these columns `nullable=false`.
- **Talend behavior:** `nullable=false` is design-time metadata; tLogRow and most downstream components do NOT enforce it at runtime. Job runs fine; downstream consumes nulls.
- **OPEN â€” design decision deferred.** Three options:
  1. Loosen `validate_schema` globally to warn-not-raise on nullable=false violations (matches Talend exactly; one-line cross-cutting change).
  2. Surgical Map-side fix: when skipping a lookup, explicitly add the lookup's columns to `joined_df` as all-NaN (bounded, doesn't touch downstream policy, but Map can't easily override schema metadata).
  3. Add config flag `strict_nullability` (default false matching Talend; opt-in strict).

### Issue 5 â€” chained intermediate variables come through empty
- **Reproduced** via `Job_vars_simple.item` with `Var.var1 = row1.salary.toString()`, `Var.var2 = Var.var1 + "_USD"`, `Var.var3 = Var.var2.length() > 4 ? "LONG" : "SHORT"`, when ALL output columns happen to be simple refs (so `_has_java_expressions` returns False â†’ simple path).
- Result: `v1=85000` âś“, `v2=` âś—, `v3=` âś—.
- **Root cause:** Simple path's `_evaluate_variables` does one bridge call per variable. Each call has no shared `Var` binding, so when var2's expression `Var.var1 + "_USD"` runs, `Var.var1` is undefined â†’ bridge returns null silently.
- The `feature/map_issues_debug` commit `1a5f40f` partially fixed this by scanning variables in `_has_java_expressions`; that's the right direction but should be subsumed by the broader marker rule from issue 1.

### Issue 6 â€” file_output_delimited filepath as expression broken
- **Reproduced** via `Job_filepath_expr.item` with FILENAME `context.OUT_DIR + context.OUT_NAME + context.OUT_EXT`.
- Result: file created with literal name `+ output_via_expr + .csv` (literal `+` and spaces in the actual filesystem name).
- **Root cause:** Converter `file_output_delimited.py:62` reads FILENAME as plain string, no `_java_expr` / `mark_java_expression`. Engine `context_manager.resolve_dict` substitutes context patterns but leaves `+` operator as literal. file_output_delimited treats result as a literal Path.
- The `feature/map_issues_debug` commit `892b051` tried a regex hack to strip `"+"` runtime artifacts â€” that's a band-aid in the wrong place.

## Locked design decisions

### Universal rule: marker presence is the ONLY signal

(established under issue 1, applies everywhere expressions are evaluated)

```
For every config field that may contain an expression:
  if value starts with "{{java}}":
      â†’ Java bridge is MANDATORY. If unavailable, raise ConfigurationError at startup.
      â†’ Send the (stripped) expression to the bridge for evaluation.
  else:
      â†’ Pure-Python evaluation. If syntax is not valid Python, raise ExpressionError.
      â†’ Engine has a small Python-eval helper; no bridge round-trip.
```

**No per-column hybrid.** A given evaluation path is either all-bridge or all-Python.
**No silent fallbacks.** Marker present + bridge missing = hard error, not empty cells.

The converter's `_java_expr` (in `talend_to_v1/components/transform/map.py`) wraps every
non-empty tMap expression with `{{java}}` today â€” that stays as-is.

For tMap engine dispatch (replaces current `_has_java_expressions` + simple/compiled split):
- ANY of `{output cols, output filter, variables, join key exprs, lookup filters, main filter}`
  has a `{{java}}` marker â†’ use **compiled Groovy path** (existing
  `_evaluate_outputs_compiled` + variable evaluation inside the script).
- NO markers anywhere â†’ **Python-eval path** (NEW; mirrors `py_map.py` patterns).

### Issue 2/3 unified design: locality-based join classification

Replace `_classify_join_type` with **per-key locality classification**:

| Key expression refs (after stripping `{{java}}`) | Classification |
|---|---|
| Nothing (literal, context, globalMap only) | `context` |
| Only main side or only previously-joined lookups (anything in `joined_df`) | `main_side` |
| Only the current lookup side | `lookup_side` |
| Both main and current lookup | `two_sided` |

**Dispatch:**
```
if join_keys is empty:
    if activate_filter and filter is two-sided:
        â†’ chunked cross-product, filter is the match condition (case D)
    elif activate_filter and filter is lookup-side-only:
        â†’ filter the lookup, then chunked cross-product (no further filter)
    else:
        â†’ chunked cross-product (pure cartesian)
elif all keys are 'context' or 'main_side':
    â†’ batch-eval main-side keys ONCE on joined_df (passing joined_lookups
      to the bridge so prior-lookup refs resolve), add as temp columns,
      standard pd.merge. O(n+m). [Talend hashmap path equivalent.]
elif all keys are 'lookup_side':
    â†’ batch-eval lookup-side keys ONCE on lookup_df, standard pd.merge. O(n+m).
elif any key is 'two_sided':
    â†’ chunked cross-product + per-pair eval.
```

**Bridge calls always pass `joined_lookups` list** so expressions can reference any
previously-joined table. This single change closes issue 3.

### Chunking design (memory-bounded cross-product)

| Knob | Default | Where used |
|---|---|---|
| `output_chunk_size` (config; alias `rows_buffer_size` for back-compat) | **50,000** (was 2,000,000) | Compiled output script execution chunking |
| `cross_join_chunk_size` (auto-tuned, override via config) | `max(100, min(10_000, 100_000_000 // lookup_rows))` | NEW: cross-product paths (filter-only join + true two-sided joins) |

Cross-product path becomes:
```python
chunk_size = max(100, min(10_000, 100_000_000 // max(1, len(lookup_df))))
result_chunks = []
for start in range(0, len(main_df), chunk_size):
    chunk = main_df.iloc[start:start + chunk_size]
    chunk_cross = pd.merge(chunk, lookup_df, how='cross')
    eval_results = evaluate(filter_or_join_expr, chunk_cross, ...)
    result_chunks.append(chunk_cross[eval_results])
result = pd.concat(result_chunks, ignore_index=True)
```

Note: dead constant `_DEFAULT_CHUNK_THRESHOLD = 100000` should be removed.

### Issue 5 â€” covered by marker rule

The all-or-nothing marker rule from issue 1 means: any `{{java}}` marker on a variable
forces the compiled path, which has a shared `Var` Map across all variable evaluations
inside the generated Groovy script. Chained `Var.var1 + "_USD"` style expressions just
work. Deletes the simple-path's per-variable bridge calls entirely.

### Issue 6 â€” converter marks filepath, engine sends to bridge

- Converter (`file_output_delimited.py:62` and similar string params) calls
  `ExpressionConverter.mark_java_expression()` instead of plain string assignment.
  `mark_java_expression` already detects `+` operators, ternaries, `.method()` calls,
  routines, etc. and prefixes with `{{java}}` only when needed.
- Engine reads `filepath` after `context_manager.resolve_dict` substitutes context refs.
  If the resolved value still has `{{java}}` prefix, send to bridge for evaluation.
  Otherwise use as literal path.
- Same pattern applies to other string params (FILENAME_STATS, FILENAME_LOGS, etc.).
- The broader audit/migration of OTHER components (filter_rows, file_input_*, java_row, etc.)
  is task #13 / a follow-up phase, not in scope here.

## Talaxie reference (verified during brainstorming)

From `Talaxie/tdi-studio-se/main/plugins/org.talend.designer.components.localprovider/components/tMap/tMap_main.inc.javajet`:

- Talend builds an `AdvancedMemoryLookup<Struct>` (in-memory) or `PersistentLookupManager`
  (on-disk) keyed by the lookup-side join column. For each main row it computes the join
  expression (which may be `row1.x.trim()` or `routines.UPCASE(row1.x)`) and uses the
  result as the hashmap key. **O(n+m) for any single-side transform** â€” exactly the
  locality-classifier `main_side` path.

- For empty `aKeysValues` (no join keys): Talend iterates EVERY lookup row for EVERY main
  row and applies the filter expression INSIDE the per-pair loop. **Same O(nĂ—m) complexity
  as cross-product â€” no clever optimization.** The win is that Talend STREAMS row by row;
  it never materializes the cross product as a single data structure. Our chunked
  cross-product approach mirrors this with bounded memory.

- The filter expression (`activateExpressionFilter`) is ALWAYS evaluated per-pair inside
  the row loop. NEVER pre-applied to the lookup container.
