## The Bug (Issue 4)

When a tMap lookup is empty:
- Map correctly skips the lookup (no rows to merge).
- Downstream component (e.g., a tFilterRow or tMap consumer) receives the
  joined_df with NULL values in lookup-derived columns.
- The downstream component's `validate_schema` (in `base_component.py:1050-1053`)
  raises DataValidationError because those columns are declared `nullable=false`
  in the .item-derived schema.
- **Talend itself does NOT enforce nullable=false at runtime.** This is a
  feature-parity gap.

## The Three Options

### Option 1: Loosen `validate_schema` globally to warn-not-raise

**Change:** `src/v1/engine/base_component.py:1050-1053` -- replace `raise
DataValidationError(...)` with `logger.warning(...)`. One-line change.

**Pros:**
- Matches Talend exactly.
- Minimal code change.
- Single source of truth -- no cross-cutting per-component logic.

**Cons:**
- Cross-cutting policy change. ALL components are affected, not just Map.
- May mask genuine schema-mismatch bugs in other component paths that
  *should* fail loud.
- Hard to opt back in later (would need a config flag, which is option 3).

**Implementation outline (DO NOT EXECUTE without user OK):**
```python
# base_component.py:1050-1053
if not nullable and result[col_name].isna().any():
    logger.warning(
        f"[{self.id}] Column '{col_name}' has NULL values but is "
        f"declared nullable=false. (Talend parity: warning, not error.)"
    )
```

### Option 2: Surgical Map-side NaN backfill

**Change:** In Map's skip-empty-lookup branch (around `map.py:362-367` per
PATTERNS.md), explicitly add the lookup's columns to `joined_df` as all-NaN
so downstream sees the columns *exist* but are null.

**Pros:**
- ZERO cross-cutting risk -- only Map changes.
- The exact pattern is already in `py_map.py:269-277` (PyMap already does this).
- Verbatim copy from analog -- low-risk implementation.

**Cons:**
- Doesn't fix downstream `nullable=false` enforcement -- the columns are
  still NaN, so the downstream check still raises.
- May need to be combined with option 1 or option 3 to fully solve.
- Per CONTEXT.md "But Map can't easily override schema metadata declared by
  the converter, so downstream nullable=false enforcement may still fire."

**Implementation outline (per PyMap analog):**
```python
# map.py: in the skip-empty-lookup branch
import numpy as np
if lookup_df is not None and len(lookup_df.columns) > 0:
    for col in lookup_df.columns:
        prefixed = f"{lookup_name}.{col}"
        if prefixed not in joined_df.columns:
            joined_df[prefixed] = np.nan
```

### Option 3: Add `strict_nullability` config flag (default false matching Talend)

**Change:** Add a top-level `strict_nullability` boolean config flag to
`BaseComponent.__init__`; gate the validate_schema check on it.

**Pros:**
- Most flexible -- per-job toggle.
- Default `false` matches Talend; teams that want strict enforcement can opt in.
- Discoverable via config.

**Cons:**
- Most API surface; new public flag to document and test.
- Adds a config-key migration concern (existing JSON jobs don't have it; need
  default fallback).
- No direct analog in the codebase (closest is `die_on_error` flag handling).

**Implementation outline:**
```python
# base_component.py
class BaseComponent:
    def __init__(self, ...):
        ...
        self.strict_nullability = self._original_config.get("strict_nullability", False)
    
    def validate_schema(self, ...):
        ...
        if not nullable and result[col_name].isna().any():
            msg = f"[{self.id}] Column '{col_name}' has NULL values but is not nullable"
            if self.strict_nullability:
                raise DataValidationError(msg)
            logger.warning(msg)
```

## Recommendation (per PATTERNS.md File 4)

> **Strong recommendation:** Option 2 has zero cross-cutting risk and a
> verbatim analog in the codebase already.

But: per CONTEXT.md D-07, option 2 alone may not fully solve issue 4 because
the nullable=false check still fires downstream. A pragmatic combo would be
**option 2 + option 1 selectively for lookup-derived columns** -- but that
combo is itself a fourth design decision and requires user approval.
