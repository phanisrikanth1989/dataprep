<objective>
Implement the locality-based join classifier (D-03) and fix all bridge call
sites to pass the full `joined_lookup_names` list (D-04). This plan replaces
the entire-list-coarse `_classify_join_type` (map.py:701-724 pre-Plan-01)
with a per-key classifier, and audits every `_evaluate_with_bridge` call
site in tMap to ensure it receives the full prior-join list.

Purpose:
- D-03: Talend's actual semantics classify join keys by **expression locality**,
  not complexity. The current `_classify_join_type` returns a single bucket
  for the whole list -- which is wrong because issue 2c (filter_join: empty
  join_keys + filter as match) and issue 2a/2b (single-side computed key)
  need different dispatch paths than equality.
- D-04: Issue 3 (lookup-to-lookup with `row3.region.trim()`) silently produces
  0 matches because `_join_cross_table` passes only `[lookup_name]` to the
  bridge -- the bridge has no context for `row3` and the expression evaluates
  to None for every row, so the filter mask is all-False.

Output:
- New constants: `_LOCALITY_CONTEXT`, `_LOCALITY_MAIN_SIDE`,
  `_LOCALITY_LOOKUP_SIDE`, `_LOCALITY_TWO_SIDED`.
- New method `_classify_key_locality(self, expr, main_name, current_lookup, joined_lookup_names) -> str`.
- The OLD constants `_JOIN_EQUALITY`, `_JOIN_CONTEXT_ONLY`, `_JOIN_CROSS_TABLE`
  remain (still used by the existing dispatch through plan 03 lands), but
  `_classify_join_type` is REMOVED. The new classifier is plumbed into the
  dispatch logic returning the **list of locality classifications**, which
  the dispatch then aggregates per the D-03 truth table.
- New private helper `_bridge_eval(self, df, exprs, joined_lookup_names)` per
  PATTERNS.md "Shared Patterns -> Bridge call always passes full
  joined_lookup_names" -- single source of truth for tMap bridge calls.
- All 5 audited call sites (`_apply_filter` / `_join_cross_table` / `_evaluate_variables` simple-path / `_evaluate_outputs_simple` / `_apply_output_filter`) pass through `_bridge_eval`.
- `_join_context_only` is left CALLABLE BUT UNROUTED in this plan (deferred
  deletion -- see Step 4 below). Behavior diff vs `_join_equality` is left
  to a future cleanup task; rerouting all-context cases to `_join_equality`
  is the only routing change made here.

This plan does NOT yet add the new join methods (`_join_main_side_computed_key`
and `_join_lookup_side_computed_key`) -- those land in plan 03 and use the
classifier output from this plan.
</objective>

       ```python
       def _classify_key_locality(
           self,
           expr: str,
           main_name: str,
           current_lookup: str,
           joined_lookup_names: list[str],
       ) -> str:
           stripped = self._strip_java_marker(expr).strip()
           matches = _ROW_REF_PATTERN.findall(stripped)
           # _ROW_REF_PATTERN matches tuples; first element is the table name.
           row_refs = {m[0] for m in matches if m[0] not in ("context", "globalMap", "Var")}
           if not row_refs:
               return _LOCALITY_CONTEXT
           # Anything in joined_df = main side. joined_df contains main + all previously-joined lookups.
           main_side_tables = {main_name, *joined_lookup_names}
           refs_main = bool(row_refs & main_side_tables)
           refs_lookup = current_lookup in row_refs
           if refs_main and refs_lookup:
               return _LOCALITY_TWO_SIDED
           if refs_lookup:
               return _LOCALITY_LOOKUP_SIDE
           return _LOCALITY_MAIN_SIDE
       ```
