## What Changed

### Converter (`src/converters/talend_to_v1/components/file/file_output_delimited.py`)

Added `ExpressionConverter` import (exact same pattern as `file_input_delimited.py:42`).

Replaced plain `_get_str` for `FILENAME` and `STREAMNAME` with:
```python
config["filepath"] = ExpressionConverter.mark_java_expression(
    self._get_str(node, "FILENAME", "")
)
config["streamname"] = ExpressionConverter.mark_java_expression(
    self._get_str(node, "STREAMNAME", "outputStream")
)
```

Other `_get_str` params (fieldseparator, encoding, separators) are constants -- not marked.

### Engine (`src/v1/engine/components/file/file_output_delimited.py`)

Added `ComponentExecutionError` import and `_JAVA_MARKER = "{{java}}"` module constant.

Added `_resolve_java_expr_param(value, name) -> str` instance method:
- Returns value unchanged when no `{{java}}` prefix
- Raises `ConfigurationError` when marker present + `self.java_bridge is None` (D-01 hard-fail)
- Sends stripped expression to `bridge.execute_tmap_preprocessing` with 1-row placeholder DataFrame
- Raises `ComponentExecutionError` on bridge exception

Wired into `_process()` after reading config values:
```python
filepath = self._resolve_java_expr_param(filepath, "filepath")
streamname = self._resolve_java_expr_param(streamname, "streamname")
```

## Issue 6 Fix

**Before:** `context.outdir + "/" + context.name + ".csv"` wrote a file named literally
`context.outdir + /context.name + .csv` (after context substitution still had `+` operators).

**After:** The converter marks it with `{{java}}`, the engine detects the marker and evaluates
via the bridge, then uses the result (e.g., `/tmp/repro/output_via_expr.csv`) as the actual
file path.
