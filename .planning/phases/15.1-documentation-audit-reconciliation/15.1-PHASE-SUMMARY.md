## What Worked

- **Wave-isolated parallel execution.** Plans 15.1-01..08 had zero file overlap; all ran as
  parallel worktree agents. Wave structure (per-component first, cross-cutting second, closeout
  third) enforced naturally by D-D2.

- **SHA inventory pre-locked in RESEARCH.md.** Executors did not need to do live
  commit-archaeology for the heavy phases (5, 5.1, 5.2, 11, 12, 13, 14). The Section C
  commit-SHA inventory gave each agent the key fix SHAs upfront.

- **D-C1 strike-through convention.** Preserving historical record while marking truth -- readers
  can grep `[RESOLVED in Phase N` for the closed-issue census. No information loss.

- **D-A5 scope discipline.** Non-shipped audit docs left untouched kept scope tractable and
  focused. The 20 non-shipped docs remain honest in their broken state rather than misleadingly
  "cleaned up" without underlying engines.

- **Verify-before-claim (D-C3).** Every SHA cited was grep-confirmed. No audit claims were
  rolled back post-commit.

---

## What Was Hard

- **The ASCII audit was continuous work.** Every doc batch contained pre-existing non-ASCII
  characters (em-dashes, arrows, checkmarks, smart quotes) from the original 2026-04-03/04
  auditors. Plans 15.1-02 through 15.1-08 all logged ASCII cleanup deviations. The most
  intensive were tExtractXMLField.md (18+ occurrences) and tFileInputFullRow.md (16+).
  tAggregateSortedRow.md slipped through Plan 15.1-07 and was caught at Plan 15.1-12 closeout.

- **RED->GREEN status flips required full section rewrites.** Plans 15.1-01 and 15.1-08 each
  had components whose audit docs were written before the engine existed. Flipping from
  "no engine implementation" to GREEN meant updating Sections 1/5/6/7/8/9/10 -- not just
  applying a strike-through. tReplace.md (Plan 15.1-08) was the clearest case.

- **Cross-cutting doc regeneration (15.1-09 through 15.1-11) required holding the full
  per-component reconciliation state in mind.** SUMMARY_SCORECARD.md had to recompute rating
  distributions, traffic-light matrices, and priority counts from all 67 docs simultaneously.
  CROSS_CUTTING_ISSUES.md required processing all 42 H3 issues and determining
  RESOLVED/STILL LIVE status for each.

---

## Key Lessons

- **Per-component audit-doc maintenance is a continuous activity.** The 67 docs will drift again
  as Phases 16+ add features and fix bugs. A post-phase trigger (whenever a phase closes
  >= 5 issues) should regenerate CROSS_CUTTING_ISSUES.md at minimum.

- **SUMMARY_SCORECARD.md drifts fastest.** It aggregates data from all 67 per-component docs
  and is the first doc a new team member reads. It should be regenerated at each major phase
  close (Phases 16, 17, etc.).

- **The SHA-inventory approach (Section C in RESEARCH.md) was the right call.** Parallel
  executors working across 67 docs in 8 plans cannot each do independent commit-archaeology
  without risk of citing wrong SHAs. Pre-locking the key fix SHAs in the research file made
  the executor work safe to parallelize.

- **D-C5 header addition (Reconciled) avoids overwriting audit history.** Adding `> **Reconciled**`
  as a new line below `> **Audited**` preserves the original audit date, which is meaningful
  context (it tells readers how long the doc was stale before reconciliation).
