**Gathered:** 2026-05-11

<domain>
## Phase Boundary

Phase 15.1 is a Talend-vs-code parity re-validation. For every component this engine currently ships, the corresponding `docs/v1/audit/components/<category>/<name>.md` is brought back into truth against (a) the current converter + engine source in `src/` and (b) the Talaxie GitHub javajet / Java template that the implementation was originally built from. Stale issues are marked resolved; net-new gaps are surfaced inline; cross-cutting docs are regenerated with section shape preserved. **No source code changes** -- doc-only phase, mirroring Phase 15 D-E3.

The audit dir is not a clean-up exercise. It is the project's gap-analysis surface between "what Talend does" and "what our code does". When Phase 15.1 closes, the surviving audit content is the team's source of truth for what is shipped, what is partially shipped, and what real bugs exist that future phases must fix.

**In scope (66 + 1 + 3 = 70 doc edits / additions):**
- 66 per-component audit doc reconciliations for shipped components (components registered in `src/v1/engine/component_registry.py` REGISTRY -- `aggregate` 2, `context` 1, `iterate` 1, `database` 3, `file` 24, `transform` 35; `control` is zero by construction)
- 1 net-new audit doc for `FileOutputXML` (shipped via Phase 12, never audited; flagged in 15-03-SUMMARY as Phase 15.1 backlog)
- 3 cross-cutting docs regenerated against post-Phase-14 reality, preserving section shape: `docs/v1/audit/CROSS_CUTTING_ISSUES.md` (2,154 lines today, ~200-250 issues already closed by Phases 1-14), `docs/v1/audit/SUMMARY_SCORECARD.md`, `docs/v1/audit/METHODOLOGY.md`
- New requirement `DOCS-03` (audit reconciliation complete) added to `REQUIREMENTS.md` at closeout

**Out of scope (locked 2026-05-11):**
- The 20 audit docs for components we do not ship (`control/` 9 -- all of them; `database/` 8 Oracle/MSSql; `file/` 1 `tFileOutputEBCDIC`; `iterate/` 1 `tForeach`; `transform/` 1 `tHashOutput`). These docs stay as-is, broken cross-references and all, because their underlying components have not been built. Phase 15.1 SUMMARY enumerates them so a future roadmap decision (Phase 16 / v2 / never) can be made; this phase does not decide their fate.
- Source code (`src/`) -- doc-only. If an audit surfaces a real production bug, it lands as a P0/P1/P2 inline in the per-component audit doc plus a follow-up note in the plan SUMMARY for a future bugfix phase. No defensive shims, no fallbacks (project memory `feedback_fix_source_no_fallbacks`).
- `CLAUDE.md` -- not edited. References stay structural.
- New top-level docs in `docs/` -- none. No `FINDINGS.md`, no `ROLLUP.md`, no new index files. Findings live inline in the per-component audit doc, in the existing P0/P1/P2/P3 sections.
- CI / pre-commit doc-freshness lint -- explicitly deferred (carries forward Phase 15 D-B1).
- Doc generation tooling -- plain Markdown only (carries forward Phase 15 D-B2).
- Filling in `tests/talend_xml_samples/` reference outputs to validate parity -- that is Phase 16 (manager-led, needs Talend Open Studio access).

<decisions>
## Implementation Decisions

### Scope & approach

- **D-A1: Shipped-component-only coverage.** Only the 66 audit docs whose component name appears in `src/v1/engine/component_registry.py` REGISTRY are reconciled. The 20 non-shipped audit docs are not touched in this phase.
- **D-A2: Inline-only findings, no new docs.** Every reconciliation finding -- resolved issues, net-new gaps, cross-ref fixes -- lands inside the existing audit doc's existing sections. No new top-level docs (no `FINDINGS.md`, no `ROLLUP.md`, no separate per-category index). The existing scorecard sections (Component Identity, Scorecard, P0/P1/P2/P3 issue lists, Resolved, Known Gaps, etc.) are the only sinks.
- **D-A3: Section-preserving edits.** When reconciling an audit doc, sections that do not need changes are not touched. Header date always updates. Other sections update only if the underlying content changed (e.g., Scorecard R/A/G updates only when component status actually changed). This is the doc-equivalent of `feedback_fix_source_no_fallbacks` -- do not "tidy" sections that are still true.
- **D-A4: Net-new audit for `FileOutputXML`.** One new audit doc authored using the same Talaxie-diff workflow as the existing 66. Phase 15.1 closes with zero shipped-component audit gaps.
- **D-A5: Non-shipped audit docs intentionally left broken.** The 20 docs covering non-shipped components retain their stale state (including the broken cross-refs from Phase 15-07's standards drop and 15-09's standards->patterns rename). 15.1-01 SUMMARY enumerates them as known limitations; their fate is a roadmap-level decision, not a doc-edit decision.

### Source of truth (parity reference)

- **D-B1: Talaxie GitHub is the canonical "what Talend does" reference.** Every per-component reconciliation reads the corresponding Talaxie javajet / Java template alongside our `src/v1/engine/components/<x>.py` + `src/converters/talend_to_v1/components/<x>.py` + tests, then diffs. This matches the existing 8-step component workflow (project memory `project_component_workflow`) that Phases 4-14 used to build the components.
- **D-B2: Existing audit docs' "Talend Side" sections are the prior reference, not a re-fetch trigger.** The audit docs were authored 2026-04-03 directly from Talaxie. Re-fetching Talaxie is only required when reconciliation surfaces a suspected drift between the doc's Talend description and current Talaxie semantics. Default mode: trust the existing Talend Side description; re-verify only when the gap diff doesn't make sense.
- **D-B3: Live source is `src/v1/engine/component_registry.py` + the source files it imports.** The static `ETLEngine.COMPONENT_REGISTRY` dict no longer exists (locked in Phase 15-02 ARCHITECTURE.md). REGISTRY membership and class identity are sourced from the decorator-based registry only.

### Per-component reconciliation protocol

- **D-C1: Resolution marker convention -- strike-through inline + phase tag.** When an existing P0/P1/P2/P3 issue is closed by an earlier phase, the issue line is preserved but struck through and tagged `[RESOLVED in Phase N, commit <short_sha>]`. The reader sees what was wrong AND what fixed it. Section line counts stay similar. Format example:
  ```
  - ~~P0: SwiftTransformer not registered in COMPONENT_REGISTRY~~ [RESOLVED in Phase 14, commit b1c2d3e (BUG-SWIFT-001)]
  ```
  Resolution evidence (phase number + commit short SHA) is grep-verified before the strike-through lands.

- **D-C2: Net-new gaps land in the existing P0/P1/P2/P3 issue blocks.** If reconciliation surfaces a real Talaxie-vs-code delta that was not in the original audit (because the original auditor missed it, or because something regressed), the new finding appears as a fresh row in the appropriate severity block. No separate "Phase 15.1 findings" subsection -- net-new and net-old coexist in the same block, distinguished by the tag (`[NEW IN 15.1]` for fresh findings; `[RESOLVED in ...]` for closed ones).

- **D-C3: Verify-before-claim, every time.** Every cited class / function / file path / Talaxie reference is grep-confirmed before the commit lands. Carries Phase 15 D-E2 + project memory `feedback_verify_audit_claims`.

- **D-C4: ASCII-only.** No emoji, no smart quotes, no en/em dashes -- use `--`. Carries Phase 15 D-C1 + project memory `feedback_ascii_logging`.

- **D-C5: Header line.** Every reconciled audit doc has its `*Last updated:* / `> **Audited**:` style header refreshed to 2026-05-11 (or actual close date), preserving the existing header schema. Carries Phase 15 D-C2.

- **D-C6: Atomic commits.** One commit per audit doc (~66 + 1 = 67 commits for the per-component work, organized into category-grouped plans). The cross-cutting regeneration is its own commits (3 separate commits, one per cross-cutting doc). Mirrors Phase 14 D-F2 / Phase 15 D-E1.

### Cross-cutting doc treatment

- **D-D1: Section-shape-preserving regeneration.** `SUMMARY_SCORECARD.md`, `CROSS_CUTTING_ISSUES.md`, and `METHODOLOGY.md` are rewritten with fresh content but retain their existing H2/H3 section structure. A consumer who knew the old shape can still navigate. The content inside each section is regenerated:
  - `SUMMARY_SCORECARD.md` -- regenerated against the current REGISTRY: per-component R/A/G, registered/abstract correctness, coverage status (Phase 14 floor), aggregate counts.
  - `CROSS_CUTTING_ISSUES.md` -- ~200-250 closed issues struck through with phase tag per D-C1; live remaining issues kept; sections preserved.
  - `METHODOLOGY.md` -- rewritten to describe the Talaxie-diff workflow (the 8-step pattern) as the canonical methodology going forward.

- **D-D2: Cross-cutting regeneration lands at the END of execution.** All three cross-cutting docs depend on the 66+1 per-component reconciliations being committed first, because their content rolls up the per-component state. Wave structure: per-component plans first, cross-cutting regeneration second, closeout third.

### Phase mechanics (carries forward from Phases 14 and 15)

- **D-E1: Doc-only.** Zero `src/` changes. If reconciliation surfaces a real bug, capture it inline (D-C2) and as a follow-up note in the plan SUMMARY for a future bugfix phase. No code patches in this phase.
- **D-E2: Phase 14 coverage gate regression check at closeout.** `python -m pytest tests/ -m "not oracle" -n auto --cov=src/v1/engine --cov=src/converters --cov-report=json && python scripts/check_per_module_coverage.py coverage.json --floor 95` MUST exit 0. If it fails, a Phase 15.1 commit accidentally touched source -- revert.
- **D-E3: No `docs/v1/audit/` edits to non-shipped docs.** Phase 15.1 only touches the 66 shipped audit docs + the 1 new `FileOutputXML` doc + the 3 cross-cutting docs (within `audit/`). The 20 non-shipped audit docs are untouched.
- **D-E4: No `CLAUDE.md` edits.** Carries Phase 15 D-B4.
- **D-E5: No top-level `docs/` edits.** The 4 canonical docs + root README from Phase 15 stay locked.
- **D-E6: Manual closeout checkpoint.** Final plan is `autonomous: false` -- the user reviews the inventory before the closeout commits ship. Mirrors Phase 15.10.

### Claude's Discretion

- Plan-level splitting of `file/` (24 docs) and `transform/` (35 docs) into multiple plans if a single plan would exceed an executor's effective context budget. Planner decides splits; the constraint is "one commit per audit doc, grouped sensibly by sub-category or alphabetical batch".
- Exact wording of the `[RESOLVED in ...]` and `[NEW IN 15.1]` tags is fixed by D-C1 / D-C2. Specific commit short-SHA selection uses `git log --follow -- <component_source>` for each component's most recent meaningful change.
- Cross-cutting doc internal section ordering matches the existing files exactly (no re-ordering). Within each section the content is regenerated freely.

</decisions>

## Specific Ideas

- The 66 shipped-component audit docs are the doc deliverable. The implicit deliverable -- arguably more valuable -- is the *list of net-new real gaps* the reconciliation surfaces. Every `[NEW IN 15.1]` tag becomes a candidate follow-up bug for a future bugfix phase or for the Phase 16 integration testing surface.
- The audit docs are forensic-style (Component Identity table -> Scorecard with R/A/G + P0-P3 -> per-issue prose). Phase 15.1 preserves that voice. It does not switch to a "narrative summary" or "parity worksheet" style.
- The cross-cutting `CROSS_CUTTING_ISSUES.md` (2,154 lines) is the most stale doc by far. Regeneration is heavy work but the section-shape constraint keeps it scoped: same H2 sections (e.g., "P0 Issues", "P1 Issues", "Cross-Component Patterns"), regenerated content, struck-through closed items.
- `tests/talend_xml_samples/` is the obvious place to validate parity end-to-end (Talend run vs our engine run, byte-compare output). That is Phase 16's surface (manager-led, needs Talend Open Studio access) -- explicitly not Phase 15.1's job. Phase 15.1's parity check is static (Talaxie source vs our source), not runtime (Talend output vs our output).
- Phase 15-03 (COMPONENT_REFERENCE.md) included a column pointing at `docs/v1/audit/components/<category>/<name>.md` as the per-component truth source. Phase 15.1 makes that promise real -- after 15.1 closes, every linked-to audit doc reflects current source.
