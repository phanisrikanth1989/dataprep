# Phase 15.1 Learnings: documentation-audit-reconciliation

## Decisions

### D-A1: Shipped-component-only coverage
Only the 66 audit docs whose component name appears in `src/v1/engine/component_registry.py` REGISTRY are reconciled in Phase 15.1. The 20 non-shipped audit docs are not touched.

**Rationale:** Scope discipline. Reconciling docs whose engines don't exist would invent content; their "broken" state is honest and informative. Their fate is a roadmap-level call (Phase 15.2 / Phase 16 / never), not a doc-edit call.
**Source:** 15.1-CONTEXT.md

---

### D-A4: Net-new audit doc authored for `tFileOutputXML`
One brand-new audit doc was authored (`docs/v1/audit/components/file/tFileOutputXML.md`) using the same Talaxie-diff workflow as the existing 66.

**Rationale:** `tFileOutputXML` shipped in Phase 12 but was never audited; flagged in 15-03-SUMMARY as Phase 15.1 backlog. Closing it lets Phase 15.1 exit with zero shipped-component audit gaps.
**Source:** 15.1-CONTEXT.md, 15.1-04-SUMMARY.md

---

### D-A5: Non-shipped audit docs intentionally left broken
20 docs covering non-shipped components (`control/` 9, `database/` 8 Oracle/MSSql variants, `tFileOutputEBCDIC`, `tForeach`, `tHashOutput`) retain their stale state — broken cross-refs and all.

**Rationale:** Their underlying components have not been built; there is no source-of-truth to reconcile against. Cleaning them would imply they were verified when they were not. Enumerated as known limitations in `15.1-VERIFICATION.md`.
**Source:** 15.1-CONTEXT.md, 15.1-VERIFICATION.md (Section 4)

---

### D-C1: Strike-through-with-phase-tag for closed issues
Closed P0/P1/P2/P3 issues are preserved but struck through and tagged `[RESOLVED in Phase N, commit <short_sha>]`. Format: `~~P0: <issue>~~ [RESOLVED in Phase 14, commit b1c2d3e (BUG-SWIFT-001)]`.

**Rationale:** Readers see both what was wrong AND what fixed it. Section line counts stay similar; the closed-issue census is greppable (`[RESOLVED in Phase N`). No information loss.
**Source:** 15.1-CONTEXT.md

---

### D-C6: One atomic commit per doc (no batched commits)
Each reconciled audit doc commits independently. 67 doc commits across 8 plans for the per-component work; 3 separate commits for the cross-cutting docs.

**Rationale:** Reverting a single doc must not unwind 6 others. Mirrors Phase 14 D-F2 / Phase 15 D-E1 commit discipline. Made parallel-wave execution tractable — each worktree commit is independently auditable.
**Source:** 15.1-CONTEXT.md

---

### D-D1: Section-shape-preserving regeneration for cross-cutting docs
`SUMMARY_SCORECARD.md`, `CROSS_CUTTING_ISSUES.md`, `METHODOLOGY.md` rewritten with fresh content but retain existing H2/H3 section structure.

**Rationale:** Consumers who knew the old shape can still navigate after regeneration. Distinguishes content-refresh from structural-refactor.
**Source:** 15.1-CONTEXT.md

---

### D-D2: Cross-cutting regeneration lands AFTER per-component reconciliation
Wave structure: per-component plans first (Wave 1), cross-cutting regeneration second (Wave 2), closeout third (Wave 3).

**Rationale:** Cross-cutting content rolls up per-component state. Regenerating them earlier would force re-regeneration after per-component edits land. The 3-wave dependency graph was enforced by the orchestrator via `depends_on` frontmatter.
**Source:** 15.1-CONTEXT.md, 15.1-PHASE-SUMMARY.md

---

### D-E1: Doc-only phase — zero src/ modifications
No `src/` changes anywhere in Phase 15.1. Real production bugs surfaced during reconciliation are captured inline as `[NEW IN 15.1]` findings in the per-component audit doc, NOT patched in this phase.

**Rationale:** Project memory `feedback_fix_source_no_fallbacks` — fix source bugs in a dedicated bugfix phase, not opportunistically during a documentation sweep. Confirmed via `git diff --stat b4463e4..HEAD -- src/` = 0 files.
**Source:** 15.1-CONTEXT.md, 15.1-VERIFICATION.md (Section 1)

---

### D-E2: Phase 14 coverage gate as regression guard at closeout
Phase 14 95%-per-module coverage gate MUST exit 0 at phase close. If it fails, a Phase 15.1 commit accidentally touched source — revert.

**Rationale:** Trivial deterministic check that D-E1 was honored. At closeout: `PASS: all 181 in-scope modules at >= 95.0% line coverage` (7993 passed, 5 skipped, 1 xfailed).
**Source:** 15.1-CONTEXT.md, 15.1-VERIFICATION.md (Section 1)

---

### D-E6: Manual closeout checkpoint required (autonomous: false on plan 15.1-12)
Plan 15.1-12 is non-autonomous. The final task is a user `human-verify` checkpoint before phase Complete is recorded.

**Rationale:** Doc-only phases lack runtime verification signals; a human inventory check (file counts, gate exit code, sample doc spot-check) is the only gate. Approved 2026-05-11.
**Source:** 15.1-12-PLAN.md, 15.1-12-SUMMARY.md

## Lessons

### ASCII compliance is systemic continuous work, not a one-shot fix
Every plan batch (15.1-02 through 15.1-08) hit pre-existing non-ASCII characters (em-dashes, arrows, checkmarks, smart quotes) inherited from the original 2026-04-03/04 audits. Most intensive: `tExtractXMLField.md` (18+) and `tFileInputFullRow.md` (16+). One doc (`tAggregateSortedRow.md`) slipped through plan 15.1-07 entirely and was caught only by the plan 15.1-12 closeout sweep.

**Context:** D-C4 ASCII-only invariant has no automated guard; reconciliation work must include grep-based ASCII spot-checks per doc. Future phase: consider a pre-commit hook on `docs/v1/audit/**` enforcing `grep -nP "[^\x00-\x7F]"` returns empty.
**Source:** 15.1-PHASE-SUMMARY.md (What Was Hard §1), 15.1-12-SUMMARY.md (Rule-1 fix 1)

---

### RED→GREEN status flips trigger multi-section rewrites, not strike-throughs
Components whose audit docs were written BEFORE the engine existed (e.g., `tReplace`, `tFlowToIterate`, `tOracleConnection/Output/Row`) needed Sections 1/5/6/7/8/9/10 rewritten — not just `[RESOLVED in ...]` markers. The strike-through convention (D-C1) is for closing issues, not for flipping the doc's overall verdict.

**Context:** When the original audit said "no engine implementation" and the engine has since been built, the Component Identity table, Scorecard, Testing section, and Known Gaps all need fresh content. Plan-time scope estimates that assumed "all docs are strike-through edits" undercounted these.
**Source:** 15.1-PHASE-SUMMARY.md (What Was Hard §2), 15.1-08-SUMMARY.md (tReplace), 15.1-01-SUMMARY.md (Oracle trio)

---

### Cross-cutting regeneration requires holding full per-component state in mind simultaneously
`SUMMARY_SCORECARD.md` regeneration recomputed rating distributions, traffic-light matrices, and priority counts from all 67 docs at once. `CROSS_CUTTING_ISSUES.md` processed ~42 H3 issues and assigned RESOLVED/STILL LIVE status per issue against the post-Phase-14 reality.

**Context:** This is why D-D2 forces cross-cutting to land at end of execution — the rollup is only consistent after the per-component truth-base is committed. Parallelizing cross-cutting plans across each other (Wave 2) is safe; running them concurrently with Wave 1 would race.
**Source:** 15.1-PHASE-SUMMARY.md (What Was Hard §3), 15.1-09/10/11-SUMMARY.md

---

### Per-component audit docs drift continuously — phase-close regeneration trigger is needed
The 67 reconciled docs will drift again as Phases 16+ add features and fix bugs. `SUMMARY_SCORECARD.md` drifts fastest (it aggregates all 67 per-component docs).

**Context:** A post-phase trigger (whenever a phase closes >=5 issues) should regenerate `CROSS_CUTTING_ISSUES.md` at minimum, and `SUMMARY_SCORECARD.md` at each major phase close. Without it, Phase 16 will inherit a fresh staleness debt.
**Source:** 15.1-PHASE-SUMMARY.md (Key Lessons §1, §2)

---

### Adding `> **Reconciled**` as a NEW line below `> **Audited**` preserves audit history
D-C5 header addition convention does not overwrite the original Audited date; it appends a Reconciled line. Readers see how long the doc was stale before reconciliation, which is meaningful context.

**Context:** Generalizable rule for date-tagged docs: prefer additive timestamping over overwrite. The audit-trail value of the original date is non-obvious until removed.
**Source:** 15.1-PHASE-SUMMARY.md (Key Lessons §4)

---

### Closeout verification sweeps catch silent rule violations that plan-level Self-Check missed
Plan 15.1-12 found 13 transform docs (6 from plan 07, 7 from plan 06) that had been committed WITHOUT the required D-C5 `> **Reconciled**: 2026-05-11` header. Plan 07 and Plan 06 each reported Self-Check: PASSED at their close.

**Context:** Per-plan Self-Check is local to the plan's own claim list. Closeout verification with phase-wide assertions (e.g., `grep -L "Reconciled.*2026-05-11" $(plan-06-files)`) catches violations that no per-plan checker would surface. A future planner should consider a `pre-commit` hook on `docs/v1/audit/components/**` enforcing `Reconciled` header presence for any modified file.
**Source:** 15.1-12-SUMMARY.md (Rule-1 fixes 2, 3)

---

### Worktree-merge state can drift mid-phase if executors commit to the wrong branch
Of 8 Wave-1 executors, 5 landed at least one commit directly on `feature/engine-restructure` rather than only on their `worktree-agent-*` branch. The orchestrator merge step found 3 plans (01, 02, 04) genuinely unmerged while 5 (03, 05, 06, 07, 08) had already partially merged via unknown mechanism.

**Context:** Final phase state was consistent (no lost commits, no duplicates), but the orchestrator's merge accounting got harder. Flagged for post-mortem if it recurs. Possible cause: executor agents running `git -C <main-repo>` instead of staying inside their worktree.
**Source:** Orchestrator merge step (Wave 1 cleanup), no specific plan SUMMARY

## Patterns

### Wave-isolated parallel execution via zero `files_modified` overlap
Plans 15.1-01..08 were structured so that no two plans modified the same file. The orchestrator's intra-wave overlap check passed trivially; 8 worktree agents ran in true parallel.

**When to use:** Any doc-only or independent-module phase where work can be partitioned by file. Author plans with disjoint `files_modified` lists in their frontmatter; let the orchestrator's overlap check verify before dispatch.
**Source:** 15.1-PHASE-SUMMARY.md (What Worked §1)

---

### Pre-lock commit SHAs in RESEARCH.md Section C — don't make parallel executors do live commit-archaeology
RESEARCH.md Section C inventoried the key fix SHAs for Phases 5, 5.1, 5.2, 11, 12, 13, 14 upfront. Each executor received the canonical SHAs in its prompt context and didn't need to run `git log --grep` or risk citing wrong commits.

**When to use:** Any phase where multiple parallel agents need to cite the same historical commits. Pre-locking the SHA inventory in research artifacts makes parallelization safe.
**Source:** 15.1-PHASE-SUMMARY.md (What Worked §2), 15.1-RESEARCH.md (Section C)

---

### Strike-through-with-tag preserves history without information loss
`~~P0: <issue>~~ [RESOLVED in Phase 14, commit b1c2d3e (BUG-SWIFT-001)]` keeps the original issue visible and adds the resolution evidence inline. The closed-issue census is greppable.

**When to use:** Whenever a doc records issues that get fixed over time. Don't delete the issue; strike it through with a phase + commit tag. Mirrors a changelog pattern but inside the doc that owns the claim.
**Source:** 15.1-PHASE-SUMMARY.md (What Worked §3), 15.1-CONTEXT.md (D-C1)

---

### Scope discipline — leave broken-but-honest docs broken when there's no source-of-truth to fix against
The 20 non-shipped audit docs stay as-is. Touching them would imply verification that didn't happen.

**When to use:** When an artifact references something that doesn't exist (deleted file, unbuilt component, retired feature). Enumerate them as known limitations and defer the decision to a roadmap phase; don't fix in-flight.
**Source:** 15.1-PHASE-SUMMARY.md (What Worked §4), 15.1-CONTEXT.md (D-A5)

---

### Verify-before-claim with grep — every cited file/class/SHA is grep-confirmed pre-commit
D-C3 + project memory `feedback_verify_audit_claims`. Every claim that "Phase 14 closed BUG-X via commit Y" was confirmed by running `git log --oneline | grep BUG-X` before the strike-through committed.

**When to use:** Audit, documentation, and architectural artifacts where claims about code state will be cited by future planners. The check is cheap (a grep); the cost of an unverified claim propagating across multiple docs is high.
**Source:** 15.1-PHASE-SUMMARY.md (What Worked §5), 15.1-CONTEXT.md (D-C3)

---

### Reconciled-header-additive convention (don't overwrite Audited date)
Audit docs carry both `> **Audited**: 2026-04-03` (original) and `> **Reconciled**: 2026-05-11` (refresh). The reader sees how long the doc was stale.

**When to use:** Any periodically-refreshed doc where the original timestamp is informative. Additive headers > destructive overwrite.
**Source:** 15.1-PHASE-SUMMARY.md (Key Lessons §4)

---

### Closeout verification sweep with phase-wide assertions
Plan 15.1-12 ran phase-wide assertions (grep `Reconciled` count, non-ASCII presence, src/ touch count) that no per-plan Self-Check would surface. Caught 13 missing headers + 1 ASCII violation.

**When to use:** End of any multi-plan phase where individual plan Self-Checks are local. The closeout plan should re-verify phase-level invariants across the union of touched files. Rule-1 auto-fix commits keep these atomic and surfaceable to future planners.
**Source:** 15.1-12-SUMMARY.md (Self-Check + Rule-1 fixes)

## Surprises

### 13 transform docs from plans 06 & 07 silently shipped without D-C5 Reconciled header
Plans 15.1-06 (Python/Swift/Java code components) and 15.1-07 (heavy hitters incl. tMap) both committed all docs and reported Self-Check: PASSED, but 13 of their 14 files were missing the required `> **Reconciled**: 2026-05-11` header line. Caught only at closeout.

**Impact:** 3 Rule-1 auto-fix commits added during closeout (`737ede3`, `b0ea050`, `2724a70`). Surfaces a planner gap: D-C5 was in CONTEXT.md but not in either plan's explicit Self-Check checklist. Future audit-reconciliation plans should make D-C5 header presence a per-task success criterion, not a phase-level invariant.
**Source:** 15.1-12-SUMMARY.md (Deviations §2 and §3)

---

### Plans 03, 05, 06, 07, 08 had commits already on `feature/engine-restructure` before orchestrator merge
At Wave-1 cleanup time, the orchestrator found 5 of 8 worktree branches already partially merged into the main feature branch. Only plans 01, 02, 04 were genuinely unmerged worktree branches.

**Impact:** Merge accounting became non-obvious. Final state was consistent (no commit loss, no duplicates), but the orchestrator's merge-and-cleanup loop had to tolerate already-merged branches. Mechanism not yet diagnosed — most likely executor agents running `git -C <main-repo>` during their work. Flagged for post-mortem if reproduced.
**Source:** Orchestrator Wave-1 merge output (not in any plan SUMMARY)

---

### 3 plan-04 doc files had merge conflicts from pre-existing stale "Last Updated" annotations on main
`tFileInputMSXML.md`, `tFileInputProperties.md`, `tFixedFlowInput.md` produced 3-way merge conflicts when plan 04's worktree merged into `feature/engine-restructure`. The HEAD versions carried `> **Last Updated**: 2026-04-05 (engine implementation created)` headers from earlier work; plan 04 replaced them with the D-C5-compliant `> **Reconciled**: 2026-05-11`.

**Impact:** Resolved by taking the worktree (theirs) version. Suggests the 2026-04-05 annotations were a previous partial reconciliation attempt that never standardized on D-C5. Future scope-gathering for similar phases should grep for stale "Last Updated" lines and either align them with D-C5 in advance or accept the conflict-resolution step.
**Source:** Orchestrator Wave-1 merge output

---

### `tReplace.md` audit was wrong — said "no engine implementation" but the engine existed
The original 2026-04-04 audit for `tReplace` documented it as having no engine. By the time plan 15.1-08 reconciled it, the engine had been built in a subsequent phase and Phase 14-05 had lifted its coverage to 100%. The reconciliation flipped the doc from RED to GREEN with full Section rewrites.

**Impact:** A reminder that "no engine" audit verdicts decay quickly once engines start landing. Future audits authored ahead of engine work should carry an explicit `predates-engine: true` marker so reconcilers spot them as candidates for full rewrites rather than strike-through edits.
**Source:** 15.1-08-SUMMARY.md, 15.1-PHASE-SUMMARY.md (What Was Hard §2)

---

### ASCII non-conformance was systemic in the original 2026-04 audits
Every per-component reconciliation batch hit em-dashes, smart quotes, arrows (→), or checkmarks (✓/✗) that violated D-C4. This was not a one-off — it was the modal state of the 67 original docs.

**Impact:** Every plan in the per-component wave had to budget time for ASCII cleanup. The original audit-authoring tool/workflow evidently emitted non-ASCII by default. Future audit-authoring should pin ASCII-only at the tool level or include a CI gate so the next reconciliation isn't another sweep.
**Source:** 15.1-PHASE-SUMMARY.md (What Was Hard §1), 15.1-12-SUMMARY.md (Rule-1 fix 1)

---

### Phase 14 coverage gate exit was deterministically clean because of D-E1 enforcement
At closeout, `PASS: all 181 in-scope modules at >= 95.0% line coverage` was the only possible outcome — `git diff --stat b4463e4..HEAD -- src/` returned 0 files. The gate's "regression guard" value in a doc-only phase is therefore trivially zero unless D-E1 was violated.

**Impact:** Suggests treating D-E2 less as a regression-detection mechanism and more as a verifier of D-E1 compliance (i.e., a confidence check that no src/ change snuck in via worktree drift or executor mistake). Same gate is meaningful in src/-touching phases; the rationale shifts for doc-only phases.
**Source:** 15.1-VERIFICATION.md (Section 1), orchestrator pre-merge check
