<approach>
## Approach

Per-component executor flow (one commit per doc, ~67 times):

1. Read the audit doc (file path from Section A row).
2. Read the engine file (Section A row).
3. Read the converter file (Section A row).
4. If reconciliation surfaces a suspected Talaxie drift, fetch the javajet at the path predicted in Section A (column "Talaxie javajet").
5. For each P0/P1/P2/P3 issue in the doc: grep-verify whether the cited file/line/symbol still exists / still has the cited problem.
6. For closed issues: strike-through + tag using the BUG-ID -> commit short-SHA mapping in Section C (or `git log --grep=<BUG-ID>` if not pre-enumerated).
7. For new gaps surfaced by Talaxie diff: append a fresh row in the appropriate severity block with `[NEW IN 15.1]` tag.
8. For broken cross-refs (Section E shipped subset): apply structural-vs-content rule from Phase 15-07-SUMMARY.md guidance.
9. Refresh the `> **Audited**:` / `*Last updated:*` header per D-C5, preserving the existing header schema for that file (Section A column "Pre-existing audit doc header date" shows the schema variant).
10. Single atomic commit per D-C6: `docs(15.1): reconcile <audit_doc_path>`.

Cross-cutting docs run AFTER all 67 per-component commits (D-D2), one commit each:
- `SUMMARY_SCORECARD.md`: regenerated against the REGISTRY-driven per-component R/A/G matrix derived from the 67 reconciled docs.
- `CROSS_CUTTING_ISSUES.md`: walk the existing 7 H2 sections, strike-through closed issues per D-C1 (~200-250 of them per CONTEXT.md), keep live issues, retain H2/H3 shape.
- `METHODOLOGY.md`: rewrite to describe the Talaxie-diff workflow as the canonical methodology going forward, preserving H2 shape (Purpose, Audience, Scope, Priority Definitions, Audit Dimensions, Report Structure, Scoring, Audit Process, Known Cross-Cutting Bugs).

Closeout (sequential on main, manual checkpoint per D-E6): add DOCS-03 to REQUIREMENTS.md, mark Phase 15.1 Complete in ROADMAP.md, write 15.1-VERIFICATION.md + 15.1-PHASE-SUMMARY.md, run Phase 14 coverage gate (D-E2) and confirm `PASS: all 181 in-scope modules at >= 95.0% line coverage`.
</approach>

## H. Risks / Open Questions

1. **Will Talaxie repo path lookups work for every shipped component?**
   Verified for `tAggregateRow` and `tMap`. Highly likely to work for every shipped Talend component (the path pattern is fully regular). Engine-native components (5) have no Talaxie path -- they should not trigger lookup attempts. **MITIGATION:** Section A explicitly marks engine-native as `N/A engine-native` so executor agents skip the Talaxie fetch.

   Phase 12 built `FileOutputXML`, `AdvancedFileOutputXML`, `FileInputMSXML` from scratch (audit-as-spec, not retrospective). The original audit for those wrote `V1 Engine Class: None`. After Phase 12, those classes now exist. **Edge case:** the audit doc's Component Identity section needs an "Engine File" entry added during reconciliation. Pattern is well-established in existing docs (compare tFileOutputDelimited Component Identity which lists actual line counts). **MITIGATION:** Section A pre-fills the correct V1 class + engine file paths for these.

3. **Alias-rich components.**
   Some V1 classes are registered under 3-4 Talend aliases (`UniqueRow`/`tUniqueRow`/`tUniqRow`/`tUnqRow`; `OracleConnection`/`tOracleConnection`/`tDBConnection`). One audit doc covers all aliases. **MITIGATION:** Section A explicitly enumerates the alias list per row; the audit doc's `Registry Aliases` cell must be reconciled against `REGISTRY.list_types()`.

   CONTEXT.md says "~200-250 closed issues". Actual enumeration finds ~70 named H3 issues across 7 H2 sections; the higher count likely counts per-component XCUT references. **MITIGATION:** Regeneration walks H3 issues in `CROSS_CUTTING_ISSUES.md` (the source of the count) rather than chasing per-component XCUT pointers. The number of strike-throughs in this file will be in the dozens, not hundreds.

5. **Phase 6/7/9 had multiple plans without clean per-BUG commit tagging.**
   Phase 4-9 commits often use feat/fix prefixes referencing the requirement ID (FILD-04, MAP-08, AGGR-03) rather than a BUG-ID. **MITIGATION:** Section C's lookup recipe falls back to `git log --grep=<REQ-ID>`; planner can authorize `[RESOLVED in Phase N]` without SHA for Phase 4-9 closures when SHA is ambiguous.

   Some files use `> **Audited**: YYYY-MM-DD`, some add `> **Re-audited**:`, some have trailing whitespace, one uses `Updated:`. **MITIGATION:** D-C5 says "preserve existing header schema per file"; Section A lists each file's existing header string verbatim so the executor knows what schema to preserve when adding `> **Reconciled**: 2026-05-11`.
