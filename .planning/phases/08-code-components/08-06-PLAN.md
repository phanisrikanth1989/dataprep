    6. **Known v1 limitations** (load-bearing -- future readers MUST see this):

       **Limitation 1: Q2 Option A -- batch-level REJECT for tJavaRow**
       The existing JavaBridge.executeJavaRow protocol is all-or-nothing (JavaBridge.java:243-246). Per D-19 the bridge protocol is locked. JavaRowComponent therefore implements batch-level REJECT: when the bridge raises a per-row error, the entire input batch is routed to result["reject"] with errorMessage and errorCode=1. This is an honest divergence from per-row REJECT.
       - Decision rationale: pre-resolved Q2 in 08-RESEARCH.md
       - Future work: a BRDG-* phase that adds an executeJavaRowWithReject variant exposing per-row reject can later upgrade JavaRowComponent transparently
       - python_row_component is unaffected -- it has full per-row REJECT

       **Limitation 2: Python namespace whitelist is hygienic, not adversarial-proof**
       D-11 removes os/sys/subprocess/__import__/open/exec/eval/compile from the user namespace. This guards against ACCIDENTAL misuse by job authors who reach for shell calls. It is NOT a security sandbox -- pure-Python namespace restriction is bypassable via __subclasses__/__mro__ introspection (RESEARCH.md Pitfall #3 + sources).
       - Trust boundary: python_code is owned by internal Citi job authors, not adversarial input
       - Future work: if threat model changes, replace exec with subprocess-isolated execution (codejail / seccomp+container)

       D-26 is hereby SUPERSEDED. The implementations in Plans 01-04 honor the SKIP_RESOLUTION_KEYS protection.
