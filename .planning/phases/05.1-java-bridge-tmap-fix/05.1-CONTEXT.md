# Phase 05.1: Java Bridge tMap Fix - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the Java bridge compiled script execution for tMap. Phase 2's rewrite broke RowWrapper's Arrow type conversion (returns raw Arrow Text objects instead of Java Strings) and removed the 3-arg constructor that compiled tMap scripts depended on. This phase restores proper type extraction so Java expressions (string concatenation, ternary conditionals, comparisons) work in compiled tMap Groovy scripts. Scope is tMap compiled scripts only -- tJava/tJavaRow code components are Phase 8.

</domain>

<decisions>
## Implementation Decisions

### Root Cause (Confirmed by debugging)
- **D-01:** `JavaBridge.buildArrowRowWrapper()` uses generic `vec.getObject()` which returns `org.apache.arrow.vector.util.Text` for VarChar columns instead of `java.lang.String`. Groovy's `+` operator on Text objects causes a silent infinite hang.
- **D-02:** The old RowWrapper (git commit f15cc36) had type-specific dispatch in `getFromArrow()` -- VarChar->String, IntVector->int, Float8Vector->double, BitVector->boolean, DateMilliVector->Date, DecimalVector->BigDecimal. Phase 2 removed all of this.
- **D-03:** IntStream.forEach lambdas cannot access Groovy binding variables (known Groovy/Java interop limitation). Compiled scripts must use plain for-loop. Already fixed in map.py during debug session.

### Fix Approach
- **D-04:** Claude's discretion on whether to fix buildArrowRowWrapper (keep RowWrapper simple) or restore the 3-arg constructor (simpler scripts). Both approaches are valid. The key requirement is that values returned to Groovy scripts are proper Java types, not raw Arrow objects.
- **D-05:** The `buildRowWrapper` Groovy closure injection into `buildTMapBinding()` has been verified to work (added during debug session). If the buildArrowRowWrapper approach is chosen, this closure stays. If 3-arg constructor is restored, the closure can be removed.
- **D-06:** Comment out or skip code only needed for tJava/tJavaRow if it interferes. Focus is tMap compiled scripts only.

### Verification Requirements
- **D-07:** MUST verify with real .item file pipeline: `Job_tMap_0.1.item` with employees.csv + country_lookup.csv. Expected output includes `full_name` (string concat: `row1.first_name + " " + row1.last_name`), `salary_grade` (ternary: `row1.salary >= 75000 ? "Senior" : "Junior"`), and cross-table columns.
- **D-08:** MUST include at least one `@pytest.mark.java` test that runs through the real JVM bridge with actual Java expressions. Mock-only tests are insufficient for bridge integration.
- **D-09:** All 86 existing tMap unit tests + 11 integration tests must continue passing.
- **D-10:** JAR must be rebuilt after any Java changes: `cd src/v1/java_bridge/java && mvn package -DskipTests`

### Claude's Discretion
- Fix approach (buildArrowRowWrapper vs 3-arg constructor vs both)
- Whether to add additional type mappings beyond the core ones (VarChar, Int, Float8, BigInt, Bit, DateMilli, Decimal)
- Whether preprocessingpath (`executeTMapPreprocessing`) also needs the same type fix
- Test structure and organization

</decisions>

- OR: RowWrapper needs 3-arg constructor restored with internal type dispatch

### Integration Points
- Python `_build_compiled_script()` generates Groovy code that calls `buildRowWrapper(inputRoot, i, "tableName")`
- Java `buildTMapBinding()` injects `buildRowWrapper` as a Groovy Closure
- Java `buildArrowRowWrapper()` builds a RowWrapper from Arrow vectors -- THIS is where types must be converted
- Groovy scripts access values via `row1.columnName` (propertyMissing -> get(name) -> inputRow.get(name))

- The old getFromArrow() at commit f15cc36 is the exact reference for what types to handle

- The preprocessingpath (executeTMapPreprocessing) also uses buildArrowRowWrapper and may benefit from the same fix
