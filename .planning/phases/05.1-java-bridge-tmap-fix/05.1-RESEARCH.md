# Phase 05.1: Java Bridge tMap Fix - Research

**Researched:** 2026-04-15
**Domain:** Java bridge Arrow type conversion for compiled tMap Groovy scripts
**Confidence:** HIGH

## Summary

Phase 2's Java bridge rewrite replaced type-specific Arrow vector extraction with generic `vec.getObject()` calls. For VarChar columns, this returns `org.apache.arrow.vector.util.Text` instead of `java.lang.String`. Groovy's `+` operator on Text objects causes a silent infinite hang -- not a crash, not an exception. This breaks all compiled tMap scripts that use string concatenation, ternary conditionals, or any expression beyond simple column reference.

The fix is surgical: add `instanceof` dispatch in `buildArrowRowWrapper()` (6 vector types, ~20 lines of code) and apply the same fix to `executeJavaRow()` which has the identical bug at line 211. The old RowWrapper at commit `f15cc36` provides the exact reference implementation. The `executeTMapPreprocessing()` method delegates to `buildArrowRowWrapper()` so it gets the fix for free.

**Primary recommendation:** Fix `buildArrowRowWrapper()` with type-specific extraction, then fix `executeJavaRow()` with the same pattern. Keep RowWrapper as the simple Map-based class (Phase 2 design). Rebuild JAR. Verify with real pipeline test using employees.csv + country_lookup.csv.

## Architecture Patterns

### Fix Location: `buildArrowRowWrapper()` in JavaBridge.java

The fix goes in one method at line 739 of `JavaBridge.java`. The current code:

```java
// BROKEN: line 745
Object value = vec.isNull(rowIndex) ? null : vec.getObject(rowIndex);
```

Must be replaced with type-specific dispatch:

```java
// Source: Old RowWrapper at commit f15cc36 getFromArrow() method
// VERIFIED: git show f15cc36:src/v1/java_bridge/.../RowWrapper.java
Object value;
if (vec.isNull(rowIndex)) {
    value = null;
} else if (vec instanceof VarCharVector) {
    value = ((VarCharVector) vec).getObject(rowIndex).toString();  // Text -> String
} else if (vec instanceof BigIntVector) {
    value = ((BigIntVector) vec).get(rowIndex);  // long
} else if (vec instanceof Float8Vector) {
    value = ((Float8Vector) vec).get(rowIndex);  // double
} else if (vec instanceof BitVector) {
    value = ((BitVector) vec).get(rowIndex) == 1;  // boolean
} else if (vec instanceof TimeStampNanoVector) {
    // Python sends datetime as pa.timestamp("ns") -> TimeStampNanoVector
    // Old code used DateMilliVector -> Date, but Phase 2 uses TimeStampNanoVector
    long nanos = ((TimeStampNanoVector) vec).get(rowIndex);
    value = new java.util.Date(nanos / 1_000_000);  // nanos -> millis -> Date
} else if (vec instanceof DecimalVector) {
    value = ((DecimalVector) vec).getObject(rowIndex);  // BigDecimal (already correct)
} else if (vec instanceof Decimal256Vector) {
    value = ((Decimal256Vector) vec).getObject(rowIndex);  // BigDecimal
} else {
    // Fallback for unknown types -- toString() is safer than raw getObject()
    value = vec.getObject(rowIndex).toString();
}
```

### Why buildArrowRowWrapper (Not 3-arg Constructor)

**Recommendation: Fix `buildArrowRowWrapper()`, keep RowWrapper as Map-based class.** Rationale:

1. Phase 2 deliberately simplified RowWrapper to decouple it from Arrow vectors. The Map-based design is cleaner -- RowWrapper only knows about Maps, not Arrow internals. [VERIFIED: Current RowWrapper.java has no Arrow imports]
2. The `buildRowWrapper` closure is already injected and working in `buildTMapBinding()`. [VERIFIED: JavaBridge.java lines 719-729]
3. The compiled script format (`_build_compiled_script()` in map.py) already generates calls to `buildRowWrapper(inputRoot, i, "tableName")`. [VERIFIED: map.py lines 1403-1405]
4. Restoring the 3-arg constructor would reintroduce Arrow coupling into RowWrapper, undoing the Phase 2 separation of concerns.
5. A shared `extractTypedValue()` helper fixes both `buildArrowRowWrapper()` and `executeJavaRow()` at once.

### Helper Method Design

Extract the type dispatch into a static helper to avoid duplicating the 20-line switch block:

```java
// Source: Design recommendation based on old RowWrapper pattern and current architecture
/**
 * Extract a properly-typed Java value from an Arrow vector at the given row.
 * VarChar returns String (not Text), numeric types return primitives.
 */
private static Object extractTypedValue(FieldVector vec, int rowIndex) {
    if (vec.isNull(rowIndex)) {
        return null;
    }
    if (vec instanceof VarCharVector) {
        return ((VarCharVector) vec).getObject(rowIndex).toString();
    } else if (vec instanceof BigIntVector) {
        return ((BigIntVector) vec).get(rowIndex);
    } else if (vec instanceof Float8Vector) {
        return ((Float8Vector) vec).get(rowIndex);
    } else if (vec instanceof BitVector) {
        return ((BitVector) vec).get(rowIndex) == 1;
    } else if (vec instanceof TimeStampNanoVector) {
        long nanos = ((TimeStampNanoVector) vec).get(rowIndex);
        return new java.util.Date(nanos / 1_000_000);
    } else if (vec instanceof DecimalVector) {
        return ((DecimalVector) vec).getObject(rowIndex);
    } else if (vec instanceof Decimal256Vector) {
        return ((Decimal256Vector) vec).getObject(rowIndex);
    } else {
        return vec.getObject(rowIndex).toString();
    }
}
```

This helper is used in exactly 2 places:
1. `buildArrowRowWrapper()` line 745 -- replaces `vec.getObject(rowIndex)`
2. `executeJavaRow()` line 211 -- replaces `vec.getObject(i)`

### Anti-Patterns to Avoid

- **Generic `vec.getObject()` for Groovy consumption:** Arrow's `getObject()` returns implementation-specific types. VarChar returns `Text`, not `String`. Always use type-specific extraction.
- **Testing compiled scripts with mocks only:** Mock-only tests cannot catch Arrow type conversion bugs since the data never round-trips through Arrow. At least one `@pytest.mark.java` test must exercise the real JVM bridge.
- **IntStream.forEach in Groovy scripts:** Java lambdas cannot access Groovy Binding variables. Always use plain for-loop. (Already fixed in debug session.)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Arrow type extraction | Custom type inference from data | Type-specific instanceof dispatch | Arrow's getObject() contract is implementation-defined per vector type |
| Row wrapper for Groovy | Direct Arrow vector access in scripts | buildRowWrapper closure delegating to buildArrowRowWrapper | Keeps Arrow internals in Java, Groovy only sees Map-based RowWrapper |
| TimeStamp conversion | String parsing or epoch arithmetic | `TimeStampNanoVector.get()` -> `new Date(nanos / 1_000_000)` | Arrow stores nanos, Java Date uses millis; simple division |

## Common Pitfalls

### Pitfall 1: Text vs String Silent Hang
**What goes wrong:** Groovy's `+` operator on `org.apache.arrow.vector.util.Text` does not throw an exception -- it silently hangs indefinitely. No timeout, no error, no stack trace.
**Why it happens:** Arrow's `Text` class (a simplified Hadoop Text) does not implement `CharSequence`. Groovy's dynamic dispatch tries to resolve `+` but enters an infinite loop.
**How to avoid:** Always call `.toString()` on VarCharVector.getObject() results before passing to Groovy.
**Warning signs:** Script execution that hangs without any exception or log output.
[VERIFIED: Debug session evidence trail in .continue-here.md -- `row1.first_name.toString() + " "` succeeds while `row1.first_name + " "` hangs]

### Pitfall 2: TimeStampNanoVector vs DateMilliVector
**What goes wrong:** The old RowWrapper handled `DateMilliVector` but the current bridge serializes datetime as `pa.timestamp("ns")` which maps to `TimeStampNanoVector`.
**Why it happens:** Phase 2 changed the Arrow schema to use `TimeStampNanoVector` (matching pyarrow's `pa.timestamp("ns")`), but the type dispatch was never updated because it was removed entirely.
**How to avoid:** The fix must handle `TimeStampNanoVector` (what Python sends), not `DateMilliVector` (what old code handled). Converting: `nanos / 1_000_000` -> millis -> `java.util.Date`.
**Warning signs:** Datetime columns returning `LocalDateTime` instead of `java.util.Date` in Groovy scripts.
[VERIFIED: ArrowSerializer.java line 172-173 creates TimeStampNanoVector for datetime; type_mapping.py line 32 maps datetime -> pa.timestamp("ns")]

### Pitfall 3: Forgetting executeJavaRow Has the Same Bug
**What goes wrong:** Only fixing `buildArrowRowWrapper()` leaves `executeJavaRow()` still broken for string operations. Phase 8 (tJavaRow) will hit the same hang.
**Why it happens:** The inline loop at line 211 in `executeJavaRow()` uses the identical `vec.getObject(i)` pattern.
**How to avoid:** Extract the type dispatch into a shared helper method, use it in both locations.
**Warning signs:** tJavaRow string operations hanging in Phase 8.
[VERIFIED: JavaBridge.java line 211 -- same generic getObject() pattern]

### Pitfall 4: JAR Not Rebuilt After Java Changes
**What goes wrong:** Tests pass locally but the bridge uses the old JAR with the bug still present.
**Why it happens:** The JAR at `src/v1/java_bridge/java/target/java-bridge-with-dependencies.jar` must be manually rebuilt after any Java source change.
**How to avoid:** Run `cd src/v1/java_bridge/java && mvn package -DskipTests` after every Java file change.
**Warning signs:** Java changes don't seem to take effect; old behavior persists despite code changes.
[VERIFIED: JAR exists at expected path, last modified 15 Apr 07:57]

## Code Examples

### Example 1: Complete buildArrowRowWrapper Fix

```java
// Source: JavaBridge.java line 739 -- fix target
// Reference: git show f15cc36:src/.../RowWrapper.java getFromArrow() method
private RowWrapper buildArrowRowWrapper(VectorSchemaRoot root, int rowIndex, String tableName) {
    RowWrapper wrapper = new RowWrapper();
    Map<String, Object> rowMap = new HashMap<>();

    for (FieldVector vec : root.getFieldVectors()) {
        String fieldName = vec.getName();
        Object value = extractTypedValue(vec, rowIndex);

        // Store under plain name
        rowMap.put(fieldName, value);

        // Also store under unprefixed name if field has tableName prefix
        if (tableName != null && fieldName.startsWith(tableName + ".")) {
            String shortName = fieldName.substring(tableName.length() + 1);
            rowMap.put(shortName, value);
        }
    }

    wrapper.setInputRow(rowMap);
    return wrapper;
}
```

## Open Questions

1. **TimeStampNanoVector.getObject() return type in Arrow 15.x**
   - What we know: ArrowSerializer.java comments at line 212 say it "often returns Long (nanos since epoch)" and at line 215 says "may return LocalDateTime". The old RowWrapper handled DateMilliVector.
   - What's unclear: Whether Arrow 15.x TimeStampNanoVector.getObject() returns Long or LocalDateTime in practice.
   - Recommendation: Use `get()` (returns `long`) rather than `getObject()` for deterministic behavior. Convert to `java.util.Date` via `new Date(nanos / 1_000_000)`. Verify with a `@pytest.mark.java` test that round-trips a datetime.

2. **IntVector handling in old RowWrapper vs current bridge**
   - What we know: Old RowWrapper had `IntVector` dispatch. Current bridge serializes `int` as `pa.int64()` -> `BigIntVector` (not `IntVector`/`pa.int32()`).
   - What's unclear: Whether any edge case could produce IntVector.
   - Recommendation: Include `IntVector` in the dispatch for safety (returns `int` via `get()`), even though the primary path should always be `BigIntVector`. Cost is one extra `instanceof` check.
