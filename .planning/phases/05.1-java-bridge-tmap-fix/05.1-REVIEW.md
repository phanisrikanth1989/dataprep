## Summary

Phase 5.1 adds the `extractTypedValue` method to `JavaBridge.java` to fix the Arrow Text vs Java String type mismatch that caused Groovy string concatenation to hang indefinitely. The core fix is sound -- it replaces two raw `vec.getObject()` call sites (in `executeJavaRow` and `buildArrowRowWrapper`) with a centralized method that dispatches on Arrow vector type and returns proper Java primitives/String.

The test coverage is strong: 4 new bridge integration tests (`TestTMapCompiledExpressions`) directly exercise the fix scenarios (string concat, ternary, cross-table, null handling), and 1 new full pipeline test (`TestTMapJavaExpressionPipeline`) validates end-to-end CSV-to-tMap-to-CSV with compiled Groovy expressions through a live JVM.

No critical issues found. Three warnings relate to defensive coding gaps in the Java bridge, and three info items flag minor code quality concerns in the test files.

## Warnings

### WR-01: extractTypedValue fallback branch may NPE on toString()

**File:** `src/v1/java_bridge/java/src/main/java/com/citi/gru/etl/JavaBridge.java:764`
**Issue:** The default `else` branch calls `vec.getObject(rowIndex).toString()`. While the null guard at line 743 checks `vec.isNull(rowIndex)`, some Arrow vector implementations can return null from `getObject()` even for non-null rows (e.g., if internal buffers are inconsistent). This would throw a NullPointerException, crashing the entire batch instead of returning null for that cell.
**Fix:**
```java
} else {
    Object raw = vec.getObject(rowIndex);
    return raw != null ? raw.toString() : null;
}
```

### WR-02: loadNextBatch() return value unchecked -- empty Arrow streams silently produce zero rows

**File:** `src/v1/java_bridge/java/src/main/java/com/citi/gru/etl/JavaBridge.java:180,362,461,564`
**Issue:** `ArrowStreamReader.loadNextBatch()` returns `boolean` indicating whether data was loaded. All four call sites ignore this return value. If the Python side sends a valid Arrow stream with zero batches (header only), the code proceeds with `rowCount = 0` and silently returns empty results. This is not a crash risk but violates fail-fast: a corrupt or truncated Arrow stream would be indistinguishable from an intentionally empty DataFrame.
**Fix:** Add a check at each call site. Example:
```java
boolean loaded = reader.loadNextBatch();
if (!loaded) {
    logger.warning("[JavaBridge] No batch loaded from Arrow stream -- input may be empty or corrupt");
}
```

### WR-03: executeTMapPreprocessing silently drops expressions that fail to compile

**File:** `src/v1/java_bridge/java/src/main/java/com/citi/gru/etl/JavaBridge.java:378-380`
**Issue:** When a Groovy expression fails to compile in `executeTMapPreprocessing`, the error is logged but the expression is silently excluded from the `compiledExprClasses` map. The result array for that expression is still allocated (line 386) and stays all-null. Downstream Python code receives nulls without knowing the expression failed to compile vs. legitimately evaluated to null for every row.
**Fix:** Either propagate the compile error to the caller, or include a sentinel/error marker in the results. At minimum, log at WARNING level instead of SEVERE (which implies a system-level failure) and document the silent-null behavior in the method Javadoc:
```java
} catch (Exception e) {
    logger.warning("[JavaBridge] Failed to compile expression '" + exprId
            + "' -- all rows will return null: " + e.getMessage());
}
```

## Info

### IN-01: Missing vector type handlers for Float4Vector, SmallIntVector, TinyIntVector

**File:** `src/v1/java_bridge/java/src/main/java/com/citi/gru/etl/JavaBridge.java:742-766`
**Issue:** `extractTypedValue` handles 8 Arrow vector types but falls through to `.toString()` for less common types like `Float4Vector` (float32), `SmallIntVector` (int16), and `TinyIntVector` (int8). If these types are ever sent by the Python side, numeric values would be coerced to String, causing ClassCastException when Groovy scripts try arithmetic operations. Currently safe because `type_mapping.py` maps floats to `Float8Vector` and ints to `BigIntVector`, but fragile if the type mapping expands.
**Fix:** Consider adding handlers for `Float4Vector`, `SmallIntVector`, and `TinyIntVector` to return numeric primitives.

### IN-02: Unused imports in test_full_pipeline.py

**File:** `tests/v1/engine/test_full_pipeline.py:10,13,15`
**Issue:** `json`, `os`, and `tempfile` are imported but never referenced anywhere in the file. These appear to be leftover from the original `TestFullPipeline` implementation.
**Fix:** Remove the three unused imports:
```python
# Remove these lines:
import json
import os
import tempfile
```

### IN-03: Inconsistent test output -- mix of print() and logger.info()

**File:** `tests/v1/engine/test_full_pipeline.py:260-266,497-502,847-855`
**Issue:** Test methods use `print()` for debug output (e.g., "=== PIPELINE SUCCESS ===") while also using `logger.info()` for structured logging (line 236, 785). This inconsistency means CI log capture settings may miss some output while showing others, and the print output adds noise in default pytest runs.
**Fix:** Replace `print()` calls with `logger.info()` or `logger.debug()` for consistency, or remove them entirely since the assertions validate correctness.
