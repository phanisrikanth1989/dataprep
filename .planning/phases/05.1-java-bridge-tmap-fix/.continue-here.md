### What Fails (The Bug)

When tMap has **complex Java expressions** that require the compiled Groovy script path:
- `row1.first_name + " " + row1.last_name` (string concatenation)
- `row1.salary >= 75000 ? "Senior" : "Junior"` (ternary conditional)
- Any expression that isn't a simple `table.column` reference

The compiled script execution **hangs indefinitely** in the Java bridge.

### Root Cause Chain (Debugged Systematically)

**Root Cause 1: Arrow Text vs Java String**

`JavaBridge.buildArrowRowWrapper()` (line 739 of JavaBridge.java) does:
```java
Object value = vec.isNull(rowIndex) ? null : vec.getObject(rowIndex);
```

For VarChar columns, `vec.getObject()` returns `org.apache.arrow.vector.util.Text` -- NOT `java.lang.String`. When Groovy tries `row1.first_name + " "`, it calls the `+` operator on a `Text` object, which causes an infinite hang (not a crash, not an exception -- a silent hang).

The **old** RowWrapper (pre-Phase 2, commit f15cc36) had explicit type dispatch:
```java
private Object getFromArrow(String fieldName) {
    // ... find vector ...
    if (vector instanceof VarCharVector) {
        return ((VarCharVector) vector).getObject(rowIndex).toString(); // <-- returns String!
    } else if (vector instanceof IntVector) {
        return ((IntVector) vector).get(rowIndex); // <-- returns int
    } else if (vector instanceof BigIntVector) {
        return ((BigIntVector) vector).get(rowIndex); // <-- returns long
    } else if (vector instanceof Float8Vector) {
        return ((Float8Vector) vector).get(rowIndex); // <-- returns double
    } else if (vector instanceof BitVector) {
        return ((BitVector) vector).get(rowIndex) == 1; // <-- returns boolean
    } else if (vector instanceof DateMilliVector) {
        return new Date(((DateMilliVector) vector).get(rowIndex));
    } else if (vector instanceof DecimalVector) {
        return ((DecimalVector) vector).getObject(rowIndex); // BigDecimal
    } else {
        return vector.getObject(rowIndex); // fallback
    }
}
```

Phase 2 removed ALL of this and replaced with generic `vec.getObject()` -- breaking every string expression.

**Root Cause 2: RowWrapper Constructor Removed**

The old RowWrapper had a 3-arg constructor used by compiled scripts:
```java
public RowWrapper(VectorSchemaRoot vectorRoot, int rowIndex, String tableName)
```

Phase 2 removed it, replacing with:
```java
public RowWrapper()  // no-arg, Map-based
public void setInputRow(Map<String, Object> row)
```

The `buildArrowRowWrapper()` private method in JavaBridge was meant to replace the constructor, but it's private and not accessible from Groovy scripts.

**Root Cause 3: IntStream.forEach Can't Access Groovy Bindings**

Java lambdas (used by `IntStream.forEach`) are NOT Groovy closures -- they can't access the script's `Binding` variables like `buildRowWrapper`, `inputRoot`, etc. This was verified:
- `IntStream.range(0, rowCount).forEach(i -> { buildRowWrapper(...) })` -- HANGS
- `for (int i = 0; i < rowCount; i++) { buildRowWrapper(...) }` -- WORKS

**Fix applied in this session:** Replaced IntStream with plain for-loop in `_build_compiled_script()`.

**Root Cause 4: buildRowWrapper Not in Binding**

The `buildRowWrapper` Groovy closure was added to `buildTMapBinding()` in this session. This works:
```java
binding.setVariable("buildRowWrapper",
    new groovy.lang.Closure<RowWrapper>(this) {
        @Override
        public RowWrapper call(Object... args) {
            VectorSchemaRoot root = (VectorSchemaRoot) args[0];
            int rowIdx = ((Number) args[1]).intValue();
            String tblName = (String) args[2];
            return buildArrowRowWrapper(root, rowIdx, tblName);
        }
    }
);
```

This was verified to work -- the closure is callable from Groovy. But the RowWrapper it returns still has the Text-vs-String problem.

### What Phase 5.1 Must Fix

1. **Primary fix:** Add type-specific extraction to `buildArrowRowWrapper()` (or restore RowWrapper's 3-arg constructor with `getFromArrow()`). VarChar MUST return `java.lang.String`, not `org.apache.arrow.vector.util.Text`.

2. **Decision needed:** Where to put the type dispatch:
   - Option A: In `buildArrowRowWrapper()` (keep RowWrapper simple, Phase 2 design)
   - Option B: Restore 3-arg RowWrapper constructor with `getFromArrow()` (old design, scripts are simpler)
   - Option C: Both (constructor for compiled scripts, buildArrowRowWrapper for other paths)

3. **Rebuild JAR** after any Java changes: `cd src/v1/java_bridge/java && mvn package -DskipTests`

4. **Verify with real pipeline:**
   - Sample data: `tests/v1/engine/fixtures/pipeline/employees.csv` (7 employees) + `country_lookup.csv` (8 countries)
   - Config: `/tmp/tmap_pipeline_test.json` (converted from Job_tMap_0.1.item, modified to use local paths + tFileOutputDelimited instead of tLogRow)
   - Expected output: `enriched_output.csv` with full_name (concat), department, salary, country (cross-table), region (cross-table), salary_grade (ternary)

## Critical Anti-Patterns

| Pattern | Severity | Description |
|---------|----------|-------------|
| Mock-only testing for bridge integration | blocking | All 86 tMap unit tests mock the Java bridge. The compiled script path was never tested with a real JVM. Phase 5.1 MUST include at least one `@pytest.mark.java` test that runs a compiled script through the real bridge with Java expressions. |

| Generic vec.getObject() for type extraction | blocking | Arrow's `getObject()` returns implementation-specific types (Text, not String). Any code reading from Arrow vectors for Groovy consumption MUST use type-specific extraction. |
