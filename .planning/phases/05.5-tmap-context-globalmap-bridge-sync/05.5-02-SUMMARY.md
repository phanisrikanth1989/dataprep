**2. [Rule 1 - Bug] PLAN's interfaces snippet got the messages payload shape wrong**

- **Found during:** Task 1 (read-only investigation of the actual compiled-script emission)
- **Issue:** The PLAN's interfaces section L94-101 says the Groovy-emitted shape is `[count: <int>, indices: <List<Integer>>, messages: <List<String>>]`. The actual emission in map.py:3526-3530 is `errorInfo.put("messages", errorMap)` where `errorMap = new HashMap<Integer, String>()` and `errorMap.put(i, e.getMessage() ...)` keys are row indices. So `messages` is a `Map<Integer, String>`, NOT a parallel `List<String>`. RESEARCH.md L246-285's pseudocode uses `messages.get(i)` where `i` is the count loop index -- this is ambiguous and could be misread as parallel-list semantics; the implementation correctly uses `messages.get(indices.get(i))` so the row index from the indices list becomes the lookup key.
- **Fix:** Implemented `errorMessage[i] = messages.get(indices.get(i))` and `rowIndex[i] = indices.get(i)`, treating `messages` as a row-index-keyed map. This matches the empirical compiled-script emission shape.
