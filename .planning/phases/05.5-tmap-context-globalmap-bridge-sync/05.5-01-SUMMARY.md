## Decisions Made

- **Registration site = JavaBridge.start() after gateway connect.** Py4J `register_input_converter` is module-global, but registering inside `start()` (rather than at module import) ensures the registration only fires when a JVM is actually launched. Class-level `_converters_registered` flag prevents re-registration when multiple `JavaBridge` instances are created in the same Python process (e.g., test-session-scoped bridges).
- **DatetimeConverter registered FIRST.** `datetime.datetime` is a subclass of `datetime.date`. Py4J iterates registered converters and uses the first whose `can_convert` returns True. Registering `DatetimeConverter` first ensures the subclass branch wins for `datetime` instances. As belt-and-suspenders, `DateConverter.can_convert` also explicitly excludes `datetime` via `and not isinstance(obj, datetime.datetime)`.
- **No setter changes.** Per SPEC Boundaries, `set_context` / `set_global_map` `str(value)` coercion is untouched in this plan -- the setter bypass is deferred to plans 05.5-04 / 05.5-05 (Wave 2). The converter registration only kicks in when callers write through the `bridge.context[k] = v` dict path; current setters still stringify.
- **ASCII-only log line.** New `logger.info("[OK] Registered Py4J date/datetime input converters")` uses plain ASCII per project memory `feedback_ascii_logging`. Matches existing `[OK]` / `[WARN]` prefix convention in the same module.
