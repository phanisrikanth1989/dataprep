## Xfail Justifications

Each xfail's `reason=` string in test_map_bridge.py is one of these 4 centralized constants. Each maps to a single root cause and a concrete fix surface for a future phase.

### `_XFAIL_FLOAT_PY4J_UNBOX` (1 cell: column-float-context)

**Root cause:** Py4J auto-unboxes `java.lang.Float` into a Python `float` on the input-converter path. Plan 05.5-04's helper wraps via `gateway.jvm.java.lang.Float(v)` -- empirical probe (live JVM, 2026-05-16) confirmed the wrap RETURNS A PYTHON FLOAT, not a Py4J JavaObject proxy. Subsequent context-write puts a Python float into `bridge.context["X"]`, which Py4J wire-serializes as `java.lang.Double` on the Groovy side. `instanceof Float == false`, `instanceof Double == true` (both verified).

**Fix surface:** Java-side Float box reconstruction in `JavaBridge.setContext` (or equivalent), keyed off a side-channel type hint that the Python helper passes. Pure-Python wrap cannot work because Py4J's bidirectional primitive unboxing is hard-wired.

**Out of scope for Phase 05.5;** tracked for a follow-up bridge phase.

### `_XFAIL_FLOAT_GM_NO_TYPE_REGISTRY` (1 cell: column-float-globalMap)

**Root cause:** GlobalMap (`src/v1/engine/global_map.py`) is a plain dict with no Talend type registry. The Plan 05.5-04 helper's id_Float wrap branch is keyed off `ContextManager.context_types[k] == 'id_Float'`, which doesn't apply to globalMap writes. So globalMap-id_Float values are written as Python float, arrive as Java Double, fail `instanceof Float`.

**Fix surface:** GlobalMap needs an optional type-tag attribute (parallel to `context_types`) so the bridge-push helper can apply the same id_Float reconstruction. This is also blocked on _XFAIL_FLOAT_PY4J_UNBOX -- even with a type tag, the Python-side wrap doesn't work.

**Out of scope;** future GlobalMap type-tagging phase.

### `_XFAIL_DATE_CTX_STR_COERCION` (2 cells: column-date_pydate-context, column-date_pydatetime-context)

**Root cause:** `ContextManager.set(value, value_type='id_Date')` invokes `_convert_type` which uses `_TYPE_CONVERTERS['id_Date']=str` (verified at `src/v1/engine/context_manager.py:63`). The Python `date` / `datetime` is coerced to a string at the ContextManager level BEFORE the Plan 05.5-04 helper push runs. The Py4J DateConverter registered in Plan 05.5-01 NEVER FIRES for this code path because the value reaching `bridge.context[k] = v` is already a `str`, not a date instance. Value arrives in Groovy as `java.lang.String`; `instanceof Date == false`.

**Fix surface:** Either
- a context_types-aware push that bypasses `ContextManager._convert_type` for date types (preserves the original Python date/datetime through to the bridge), OR
- removing the str converter from id_Date in `_TYPE_CONVERTERS` and delegating format conversion entirely to component-side `parseDate` (which is what the 8 format-cells already exercise successfully on `path (a)`)

Either is a Phase 05.6+ change.

### `_XFAIL_DATE_GM_CONVERTER_DISPATCH` (4 cells: both surfaces x both date types on globalMap)

**Root cause:** When `bridge.global_map["X"] = date_value` is executed, Py4J's input-converter machinery dispatches `convert(gateway_client, value)`. The Plan 05.5-01 `DateConverter` / `DatetimeConverter` access `gateway_client.jvm`, but Py4J passes the `GatewayClient` instance (no `.jvm` attribute) instead of the higher-level `JavaGateway`. Empirically raises `AttributeError: 'GatewayClient' object has no attribute 'jvm'` mid-execute, bubbling up as `ComponentExecutionError`. The tMap execution explodes on BOTH surfaces (column and filter) -- the dispatch fires at dict-write time, before any Groovy evaluation begins.

**Fix surface:** Rebuild the `java.util.Date` via the gateway_client's plain-protocol API (e.g. via `java_collections_converter` or direct command-string construction) rather than the high-level `.jvm` proxy. This is a converter-side bug-fix in Plan 05.5-01's bridge.py converters; follow-up.

## Decisions Made

- **Flat 4-axis parametrize instead of stacked.** The naive approach of stacking `@parametrize('surface', [...])` over `@parametrize('label,...', [...])` (the PLAN's suggested shape, L124-128) cannot express per-surface xfail marks: a single param entry's `marks` attribute applies to ALL surface instances it fans out to. Empirically, the `float-context` column cell is xfail (Py4J unbox) but the `float-context` filter cell passes (filter only checks null) -- these need different marks. Flattening into a single `pytest.param(surface, label, ..., marks=...)` per cell solves this cleanly.

- **Surface-aware xfail decisions.** Float cells xfail only on column surface (filter still passes); id_Date context cells xfail only on column surface (filter still passes); id_Date globalMap cells xfail on BOTH surfaces (the dispatch error fires at dict-write time, before any Groovy runs). Each surface decision is grounded in empirical probe data captured pre-implementation.

- **None cells without xfail.** The None type-row uses a null-check expression (`(ns.X == null) ? "null" : "not_null"`) instead of an instanceof check, so it tests the null-routing contract rather than type fidelity. Both surfaces pass: column returns the literal string `"null"`, filter `ns.X != null` evaluates false and drops the row.

- **Datetime-format cells use the str-path (path a).** SPEC L88-95 disambiguates id_Date into two paths: (a) Python str source under id_Date type, parsed by Groovy's parseDate; (b) Python date/datetime source arriving as java.util.Date, instanceof-checked. The 8 format-cells exercise path (a) and all pass (parseDate accepts the locked formats). Path (b) is covered by the date_pydate / date_pydatetime rows in the 36-cell matrix, which are currently xfailed (see DATE_CTX_STR_COERCION / DATE_GM_CONVERTER_DISPATCH above) -- so the SPEC's R6 escalation policy is exercised in both directions: path (a) green, path (b) xfail-with-reason.

- **Centralized _XFAIL_* reason strings.** Each distinct root-cause failure mode gets a single module-level constant; cells sharing the same root cause reuse the same constant. This makes the xfail surface area visible in one place and supports the 05.5-08-VERIFICATION acceptance gate, which can match on these constants to confirm "exactly the SPEC's permitted escalations, no silent skips" without parsing 8 separate reason strings.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Test fidelity] PLAN suggested stacked @parametrize, empirical xfail dispositions required flat 4-axis parametrize.**

- **Found during:** Task 1 first verify pass
- **Issue:** PLAN L124-128 suggests `@parametrize('label,...', _TYPE_ROWS)` stacked with `@parametrize('namespace', [...])` stacked with `@parametrize('surface', [...])`. After the first verify pass, 4 cells XPASSed-strict because xfail marks attached at the type-row level applied to all surfaces -- but float-context-filter passes (the filter only checks `!= null`). Stacked parametrize cannot express per-surface marks within a single type-row.
- **Fix:** Flattened to one `@parametrize('surface,label,...,namespace', _build_type_cell_params())` with all 4 axes baked into each `pytest.param`. The builder function constructs 36 params and attaches marks at each cell individually via a `marks_grid` keyed on (namespace, surface).
- **Files modified:** Only the test file; commit `e9ea623` final shape includes the fix.
- **Tracked as deviation** because PLAN's example code section L124-128 explicitly showed stacked decorators.

**2. [Rule 1 - Bug discovery] Plan 05.5-04's id_Float wrap is a no-op; xfailed instead of fixed (architectural -- escalates to Rule 4 but accepted because SPEC R6 allows escalation).**

- **Found during:** Task 1 empirical probe (pre-implementation)
- **Issue:** Plan 05.5-04 introduced `gateway.jvm.java.lang.Float(v)` wrap for id_Float values. Live JVM probe confirmed the wrap returns a Python float (verified via `type()`), so the value still arrives in Groovy as java.lang.Double. The wrap is effectively a no-op.
- **Decision:** xfail the affected cell (column-float-context) with `_XFAIL_FLOAT_PY4J_UNBOX` reason rather than fixing in this plan. SPEC R6 explicitly permits per-cell escalation. Fix surface (Java-side Float box reconstruction in JavaBridge.setContext) is an architectural change that crosses into bridge.py and likely the Java side -- out of Plan 05.5-06's testing-only scope.
- **Escalated to:** Future bridge phase. Documented in this SUMMARY's xfail justifications.

**3. [Rule 1 - Bug discovery] DateConverter dispatch broken for globalMap dict-write path.**

- **Found during:** Task 1 empirical probe
- **Issue:** Writing `bridge.global_map["X"] = date(2026, 5, 15)` raises `AttributeError: 'GatewayClient' object has no attribute 'jvm'` because the Plan 05.5-01 converters access `gateway_client.jvm`. Py4J passes `GatewayClient`, not `JavaGateway`.
- **Decision:** xfail with `_XFAIL_DATE_GM_CONVERTER_DISPATCH` reason rather than fix in this plan. Fix surface is in Plan 05.5-01's bridge.py converter code -- out of Plan 05.5-06's testing-only scope. The bug only surfaces on the globalMap dict-write path; the context dict-write path doesn't hit it because the value is already str-coerced by `_TYPE_CONVERTERS['id_Date']=str` upstream (so the converter dispatch isn't even attempted -- see `_XFAIL_DATE_CTX_STR_COERCION`).
- **Escalated to:** Plan 05.5-01 follow-up.
