# Phase 05.5: tMap Context/GlobalMap Bridge Sync + `__errors__` Arrow Round-trip — Research

**Researched:** 2026-05-15
**Domain:** Python-Java Py4J state synchronization + Arrow IPC serialization (Java side)
**Confidence:** HIGH (root cause pinpointed via empirical spike + source inspection; Py4J protocol verified against upstream)

## Summary

The fix has TWO physically distinct mechanisms that the planner must keep separate:

1. **Map / FilterRows (R1, R2, R3, R4, R7):** A typed-state push from Python `ContextManager` / `GlobalMap` into the `JavaBridge` Python wrapper's internal `self.context` / `self.global_map` dicts at every bridge-call site. The current call into `executeTMapPreprocessing` / `executeCompiledTMap` already passes these dicts as Py4J arguments; Java already merges them into its `Binding`. The only missing piece is the upstream push from `ContextManager.get_all()` and `GlobalMap.get_all()`. The pattern proven on `origin/main` and on the working `BaseComponent._resolve_java_expressions` (base_component.py L371-384) uses the existing `JavaBridge.set_context(k, v)` / `set_global_map(k, v)` setters — but those setters have a `str(value)` coercion bug at bridge.py L790 / L801 that breaks R6 type fidelity (every value becomes a Java String). The cleanest fix is to write directly into the bridge's internal `context` and `global_map` dicts (bypassing the lossy setter) since they flow through the same `executeTMapPreprocessing` / `executeCompiledTMap` call argument unchanged.

2. **PyMap (R5):** A different mechanism entirely. PyMap is pure Python `eval()` — there is no Java bridge involved. SPEC R5's framing as a "regression" is inaccurate: py_map is net-new on this branch and does not exist on `origin/main`. The `_build_namespace` helper at py_map.py L379-401 does not bind `context` or `globalMap` into the eval namespace. The fix is to inject `context` (a dict-like view of `self.context_manager.get_all()`) and `globalMap` (a dict-like view of `self.global_map.get_all()` plus a `.get(key)` shim) into the eval namespace.

3. **Java `__errors__` Arrow serialization (R8):** JavaBridge.java L846 currently `continue`s past the `__errors__` entry in `convertTMapOutputsToArrow`. The Java side has all the building blocks (LinkedHashMap of column data, `ArrowSerializer.createOutputRootFromData`) to emit a structured Arrow stream. The work is to design a fixed reserved-column schema (`errorMessage`, `errorStackTrace`, optionally row-index column) and emit an Arrow batch using the same `ArrowStreamWriter` path the active outputs use.

4. **R9 prediction:** Of the 6 strict-xfails, the 2 catch tests (test_map_reject_catch.py:169, 207) ARE about `__errors__` round-trip — R8 directly addresses them. The 4 filter-reject compiled-path xfails (test_map_reject_filter.py:384-468) are about a script-emission gap (`_build_compiled_script` not emitting `evalOutput_<rej>` in the active-mode loop) — this is **orthogonal** to R1+R8 and will likely stay xfail unless an extra plan task addresses the script-emission gap.

**Primary recommendation:** Plan three independent task swimlanes that converge:
- **Swimlane A (R1, R2, R3, R4, R7):** Add `_push_runtime_state_to_bridge` to `Map` + `FilterRows`. Bypass `set_context`'s `str(value)` coercion by writing directly to `bridge.context[k] = v` / `bridge.global_map[k] = v`. Fire at all 3 map.py sites (L2113 compile is OK to also receive sync — defensive — but the active sites are L2143, L2177, L3607) and 1 filter_rows.py site (`_handle_advanced` before `execute_tmap_preprocessing`).
- **Swimlane B (R5):** Add `context` and `globalMap` to py_map's `_build_namespace` eval namespace; no bridge involved.
- **Swimlane C (R6 type matrix):** A small Py4J input-converter for `datetime.date` and `datetime.datetime` → `java.util.Date` (must be registered before bridge starts); an explicit `gateway.jvm.java.lang.Float(value)` construction in `set_context` / `set_global_map` when `context_types[key] == 'id_Float'`. Verify Decimal already round-trips via native Py4J protocol.
- **Swimlane D (R8):** Modify Java `convertTMapOutputsToArrow` to emit `__errors__` as Arrow; modify `_route_catch_output_rejects` to consume the DataFrame shape (already wired-ready per Phase 05.4-04).

### Py4J protocol facts (verified against py4j/protocol.py upstream)

The Py4J wire protocol natively serializes these Python types to typed Java values, no converter registration needed:

| Python type | Type tag | Java target | Notes |
|-------------|----------|-------------|-------|
| `None` | `n` | `null` | [VERIFIED: protocol.py — `NULL_TYPE = "n"`] |
| `bool` | `b` | `java.lang.Boolean` | Checked before int (since bool subclasses int) |
| `int` in `[-2^31, 2^31-1]` | `i` | `java.lang.Integer` | Maps to `id_Integer` cleanly |
| `int` outside range | `L` | `java.lang.Long` | Maps to `id_Long` cleanly |
| `float` | `d` | `java.lang.Double` | **Cannot distinguish `id_Float` natively — workaround below** |
| `decimal.Decimal` | `D` | `java.math.BigDecimal` | **Maps to `id_BigDecimal` natively. No converter needed.** |
| `str` | `s` | `java.lang.String` | |
| `bytes` / `bytearray` | `j` | `byte[]` | Not used in this phase |

**NOT supported natively:** `datetime.date`, `datetime.datetime`. These require either a `register_input_converter` registration at bridge startup, or explicit construction (e.g. `gateway.jvm.java.util.Date(epoch_millis)`) at `set_context` time. [CITED: py4j/py4j-python/src/py4j/protocol.py:get_command_part on GitHub master]

**Float vs Double:** Py4J always serializes Python `float` as Java `Double`. To force `Float`, the workaround is explicit JVM construction: `gateway.jvm.java.lang.Float(value)`. [VERIFIED: py4j GitHub Issue #271 "Proxy handler does not allow returning a float"]

### Type converter registration pattern (PySpark reference)

PySpark uses this exact pattern for datetime → Timestamp; we adapt it for datetime → `java.util.Date`:

```python
# Pattern reference: pyspark.sql.types DatetimeConverter, DateConverter
from py4j.protocol import register_input_converter
import datetime, calendar, time

class DateConverter(object):
    def can_convert(self, obj):
        return isinstance(obj, datetime.date) and not isinstance(obj, datetime.datetime)
    def convert(self, obj, gateway_client):
        # midnight UTC; java.util.Date constructor takes epoch millis
        seconds = calendar.timegm(obj.timetuple())
        return gateway_client.jvm.java.util.Date(int(seconds * 1000))

class DatetimeConverter(object):
    def can_convert(self, obj):
        return isinstance(obj, datetime.datetime)
    def convert(self, obj, gateway_client):
        if obj.tzinfo is not None:
            seconds = calendar.timegm(obj.utctimetuple())
        else:
            seconds = time.mktime(obj.timetuple())
        millis = int(seconds * 1000 + obj.microsecond // 1000)
        return gateway_client.jvm.java.util.Date(millis)

# Registration order matters: datetime subclass FIRST so its can_convert
# is tested before date's
register_input_converter(DatetimeConverter())
register_input_converter(DateConverter())
```

Registration must happen once before the gateway starts (or before the first call). Best home: `JavaBridge.__init__` or top of `bridge.py`. [CITED: pyspark.sql.types]

|------------|-----------|----------|

Bypassing keeps the setter's existing contract for non-tMap callers; fixing it is the cleaner long-term answer but expands blast radius. Plan-phase decision. |
| Per-component `_push_runtime_state_to_bridge` helper | Shared helper on BaseComponent | SPEC explicitly rejects the shared-helper option to avoid implicit-sync semantics. Three near-identical 8-line helpers is the locked pattern. |
| Py4J `register_input_converter` for date/datetime | Explicit conversion inside `set_context` | Converter approach is automatic (any code path) and matches PySpark precedent. ### Pattern 1: Per-Component Runtime State Push

**What:** A small private method per component that flushes Python-side `ContextManager` and `GlobalMap` state into the JVM bridge's internal dicts immediately before any bridge invocation that runs row-level Groovy.

**When to use:** Inside every method that calls `self.java_bridge.execute_*` for per-row evaluation. Three call sites in `Map` (L2143, L2177, L3607 via `_evaluate_with_bridge`), one in `FilterRows._handle_advanced` (L435).

**Example (Map):**
```python
# Source: pattern from origin/main map.py L399-403 (verified via git show)
#         and base_component.py L372-384 (verified working today)
def _push_runtime_state_to_bridge(self) -> None:
    """Flush Python-side context + globalMap to the Java bridge.

    Must be called immediately before any bridge invocation that
    executes per-row Groovy. Writes typed values directly into the
    bridge's internal dicts (NOT via set_context/set_global_map which
    coerce to str — breaks R6 type fidelity).
    """
    if self.java_bridge is None:
        return
    if self.context_manager is not None:
        for k, v in self.context_manager.get_all().items():
            self.java_bridge.context[k] = v
    if self.global_map is not None:
        for k, v in self.global_map.get_all().items():
            self.java_bridge.global_map[k] = v
```

Call sites in `Map._execute_compiled_tmap_chunked` (L2113 onward):
```python
self.java_bridge.compile_tmap_script(...)   # compile (script doesn't need state)
self._push_runtime_state_to_bridge()        # <<< NEW: before execute
raw_result = self.java_bridge.execute_compiled_tmap_chunked(...)  # L2143
# ... reject pass:
self._push_runtime_state_to_bridge()        # <<< NEW: before reject-mode execute
reject_raw = self.java_bridge.execute_compiled_tmap_chunked(...)  # L2177
```

Call site in `Map._evaluate_with_bridge` (L3572):
```python
self._push_runtime_state_to_bridge()        # <<< NEW
return self.java_bridge.execute_tmap_preprocessing(...)  # L3607
```

Call site in `FilterRows._handle_advanced` (filter_rows.py L435):
```python
self._push_runtime_state_to_bridge()        # <<< NEW (mirror helper)
results = self.java_bridge.execute_tmap_preprocessing(...)
```

### Pattern 2: PyMap Namespace Binding

**What:** PyMap evaluates expressions via Python `eval()` against a namespace dict. The current `_build_namespace` does not include `context` or `globalMap`. Fix: add them as live dict views.

**Example:**
```python
# In py_map.py _build_namespace (around L393):
ns: dict[str, Any] = {}
ns.update(_SAFE_NAMESPACE_GLOBALS)
ns["__builtins__"] = _build_safe_builtins()
ns.update({name: _Row(rd) for name, rd in row_dicts.items()})
ns["Var"] = var_dict
ns["Decimal"] = Decimal
# NEW R5:
ns["context"] = _ContextView(self.context_manager) if self.context_manager else {}
ns["globalMap"] = _GlobalMapView(self.global_map) if self.global_map else {}
```

`_ContextView` and `_GlobalMapView` are thin wrappers that:
- Support both `context.X` (attribute access via `__getattr__`) and `context["X"]` (item access)
- For `globalMap`, also support `globalMap.get(key)` (Talend-style)
- Are read-only from the eval namespace's perspective (`__setattr__` raises or no-ops — pyMap shouldn't mutate state through expressions)

This is the equivalent of `Map`'s `_Row` dict subclass that supports attribute-style access (py_map.py L70-80).

### Pattern 3: Java `__errors__` Arrow Emission

**What:** Replace the `continue` at JavaBridge.java:846 with a structured emission path. Schema: `{rowIndex: int, errorMessage: str, errorStackTrace: str}`.

**Example (Java):**
```java
// In convertTMapOutputsToArrow, REPLACE:
//     if ("__errors__".equals(outputName)) { continue; }
// WITH a real branch that emits Arrow:

if ("__errors__".equals(outputName)) {
    // outputResult is the Groovy dict produced by the compiled script:
    //   { count: int, indices: List<Integer>, messages: List<String>,
    //     stackTraces: List<String> }
    int errCount = ((Number) outputResult.get("count")).intValue();
    List<Integer> indices = (List<Integer>) outputResult.get("indices");
    List<String> messages = (List<String>) outputResult.get("messages");
    // Add stackTraces emission to _build_compiled_script if absent
    List<String> stacks = (List<String>) outputResult.getOrDefault(
            "stackTraces", new ArrayList<>());

    Map<String, String> errSchema = new LinkedHashMap<>();
    errSchema.put("rowIndex", "int");
    errSchema.put("errorMessage", "str");
    errSchema.put("errorStackTrace", "str");

    Map<String, Object[]> errColumns = new LinkedHashMap<>();
    Object[] idxArr = new Object[errCount];
    Object[] msgArr = new Object[errCount];
    Object[] stArr = new Object[errCount];
    for (int i = 0; i < errCount; i++) {
        idxArr[i] = indices.get(i);
        msgArr[i] = messages.get(i);
        stArr[i] = i < stacks.size() ? stacks.get(i) : "";
    }
    errColumns.put("rowIndex", idxArr);
    errColumns.put("errorMessage", msgArr);
    errColumns.put("errorStackTrace", stArr);

    try (VectorSchemaRoot errRoot = ArrowSerializer
            .createOutputRootFromData(allocator, errColumns, errSchema)) {
        ByteArrayOutputStream out = new ByteArrayOutputStream();
        ArrowStreamWriter writer = new ArrowStreamWriter(errRoot, null, out);
        writer.start();
        writer.writeBatch();
        writer.close();
        outputArrowData.put("__errors__", out.toByteArray());
    }
    continue;
}
```

The compiled script in `_build_compiled_script` must populate `stackTraces` alongside `messages` if it doesn't already. Verify by reading the script generation in map.py — if absent, plan a small Groovy-side edit.

### Anti-Patterns to Avoid

- **Calling `bridge.set_context(k, v)` for typed values:** The setter coerces every value to `str(value)` (bridge.py L790, L801). This breaks R6 entirely — every Date / Decimal / Float / Integer arrives in Groovy as a String. Either bypass with direct dict write OR fix the setter to preserve type. Plan-phase chooses; both are valid.
- **Adding a shared sync helper on BaseComponent:** SPEC explicitly rejects this (see SPEC L100-111 Boundaries). The three private helpers are the locked pattern.
- **Calling `_push_runtime_state_to_bridge` from `_resolve_expressions` for tMap:** Won't help — `Map._resolve_expressions` skips the parent and the bridge is invoked later inside `_process`. Push at the call sites, not at config resolution.
- **Relying on the existing `_sync_from_java` after-call sync:** It's correct as a backstop but doesn't substitute for the pre-call push. `_sync_from_java` reads Java state back to Python; we need the opposite direction.
- **Mutating `bridge.context` outside the call site:** It already carries `__rejectMode__` from D-09 (bridge.py L603-618). A naive `bridge.context.clear()` before push would break the reject-mode flag. Use direct key writes (`bridge.context[k] = v`) instead of clear-and-set, OR snapshot/restore the `__rejectMode__` key.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Python ↔ Java type conversion for primitives | Custom serializer | Native Py4J protocol (Decimal, int, bool already typed) | Py4J already does it correctly per the verified type table. |
| `datetime` → Java `Date` conversion | Manual `str(dt)` + parseDate in Groovy | Py4J `register_input_converter` (PySpark pattern) | One registration covers ALL call sites and produces native `java.util.Date` — Groovy can use it directly without parseDate. |
| Arrow IPC stream from `__errors__` | Custom byte protocol | `ArrowStreamWriter` + existing `ArrowSerializer.createOutputRootFromData` | Java side already uses this pattern for every output; identical machinery, just a new column schema. |
| Per-call diff of context state | Manual change detection | Push all keys every call | The dict is small (typically <50 keys), bridge is in-process, and idempotent overwrite is cheaper than diff bookkeeping. |
| Bidirectional race protection | New lock around context | Sequential execution invariant (already locked) | `_call_java_with_sync` already runs `_sync_from_java` at the end of every bridge call (bridge.py L860), establishing causal ordering. SPEC R1 explicitly states this. |

**Key insight:** Every primitive type listed in R6 except `id_Float` and `id_Date` survives Py4J's native protocol unchanged. The phase's type-fidelity work boils down to: (a) register two date converters, (b) one Float construction when type ID says `id_Float`, (c) verify Decimal works as expected, (d) confirm None survives the wire. Most of the matrix is already correct once R1's push happens.

## Common Pitfalls

### Pitfall 1: `set_context` `str(value)` coercion shadowing R6 type work
**What goes wrong:** Plan adopts `_push_runtime_state_to_bridge` that calls `self.java_bridge.set_context(k, v)`. R1-R4 spike tests start passing because `context.MY_VAR` is now bound, but R6 type-fidelity tests fail because every value arrives as a Java String.
**Why it happens:** The bridge.py L790 `setContext(key, str(value))` coercion is invisible from the helper's perspective — the helper looks correct.
**How to avoid:** Use direct dict write (`self.java_bridge.context[k] = v`) OR fix the setter to preserve type. Make the choice explicit in the plan task description.
**Warning signs:** R6 tests pass for `id_String` but fail for `id_Integer` / `id_BigDecimal` / `id_Date`.

### Pitfall 2: `__rejectMode__` clobbered by clear-and-push
**What goes wrong:** Plan writes `_push_runtime_state_to_bridge` as `bridge.context = dict(context_manager.get_all())` (assignment) or starts with `bridge.context.clear()`. The reject-pass invocation at map.py L2177 fires after `bridge.context["__rejectMode__"] = True` (bridge.py L604). If the push happens between L604 and L2177, the flag is overwritten and the compiled script silently executes in active mode.
**Why it happens:** `execute_compiled_tmap_chunked` sets `__rejectMode__` inside its own body (bridge.py L597-618), wrapping the call in try/finally. Any push between the public entry-point's mutation and the actual bridge call must preserve that key.
**How to avoid:** (a) Push BEFORE calling `execute_compiled_tmap_chunked` (i.e. at the map.py call site, not inside the bridge method), and (b) use per-key writes, not dict replacement. The proposed helper pattern (per-key `bridge.context[k] = v` loop) preserves any pre-existing keys including `__rejectMode__` if the user's ContextManager doesn't also have that key.
**Warning signs:** Reject-pass tests intermittently report 0 reject rows; user catch outputs appear in `result["catch"]` populated with active-mode column values instead of reject-mode ones.

### Pitfall 3: Float silently becoming Double
**What goes wrong:** Spike test for `id_Float` passes `1.5` and asserts `context.X instanceof Float` in Groovy. Test fails because Py4J serialized it as `java.lang.Double`.
**Why it happens:** Py4J's protocol always sends Python `float` as Double; there's no way to mark it as 32-bit from the Python side. [VERIFIED: py4j GitHub Issue #271]
**How to avoid:** In `_push_runtime_state_to_bridge`, inspect `self.context_manager.context_types[k]` (ContextManager L96 tracks type IDs) — if it's `id_Float`, explicitly construct `gateway.jvm.java.lang.Float(value)` before assigning to `bridge.context[k]`.
**Warning signs:** `context.X instanceof Float` returns false in Groovy when the Talend type ID is `id_Float`; precision is preserved but the type assertion fails.

### Pitfall 4: TalendDate.parseDate(format, context.X) when context.X is already a Date
**What goes wrong:** The R6 acceptance criterion says `routines.TalendDate.parseDate(format, context.X)` must succeed for the 4 format variants. But if `context.X` arrives as `java.util.Date` (via the Py4J converter), parseDate's signature is `parseDate(String, String)` — passing a Date second arg either throws or calls `.toString()` and parses the JVM default format.
**Why it happens:** Two semantically incompatible expectations: (a) context.X is a typed Date for direct Groovy use, (b) context.X is a String that user code parses with parseDate.
**How to avoid:** Pick one. The SPEC's matrix says "parseDate(format, context.X) succeeds" — so context.X should be a String when the Talend type ID is `id_Date`. ContextManager L63 already stores `id_Date` values as strings (its converter is `str`). The Py4J date converter should be applied ONLY when the Python value is actually a `datetime.date` / `datetime.datetime` (not when it's a stringified date from ContextManager). This means: don't blindly convert; respect the value's actual Python type.
**Warning signs:** parseDate throws `ParseException` saying "Unparseable date: 'Fri May 15 ...'" — that's `java.util.Date.toString()` output.

### Pitfall 5: Existing test depending on empty bridge.context
**What goes wrong:** The R1 push starts populating `bridge.context` with values that ALL ContextManager keys carry. Some pre-existing test asserts that bridge state stays clean across components, or that an unrelated bridge call returns null for an unset context key.
**Why it happens:** The original "context starts empty" wasn't a contract, it was an accident — but a test may have ossified that accident.
**How to avoid:** Pre-implementation, grep `tests/` for `bridge.context\s*==\s*\{\}` / `assert.*context.*empty` / `getContext` and review hits. If any test asserts the prior accidental behavior, classify as: (a) genuinely testing isolation (which we still preserve since the bridge is per-component or per-engine, not global) — likely keeps passing, (b) testing the accident — needs update or deletion.
**Warning signs:** Phase 14 coverage gate or test suite regresses on tests that haven't touched their assertions.

### Pitfall 6: addRoutinesToBinding called per-row vs once
**What goes wrong:** No-op risk: confirming routines still work post-fix.
**Why it happens:** `buildTMapBinding` calls `addRoutinesToBinding(binding)` exactly once at binding construction time (JavaBridge.java L756). Per-row, the same binding is reused. The push happens BEFORE buildTMapBinding fires, so routines remain bound. Routines on `row.X` already pass (spike E3) — they will continue to pass.
**How to avoid:** Just verify in tests, no code change needed.
**Warning signs:** Spike E3 regresses. (Unlikely — completely orthogonal mechanism.)

### Pitfall 7: ContextManager's nested-context flatten on origin/main
**What goes wrong:** Plan copies `origin/main`'s exact loop (which expects `context_all` to be a dict-of-dicts and flattens it). The current branch's `ContextManager.get_all()` returns a flat dict (verified at context_manager.py L351-357). The flatten logic would treat single-level values as nested and corrupt them.
**Why it happens:** The two branches have diverged in how context is stored.
**How to avoid:** Use the current branch's flat `get_all()` directly. Do NOT copy `origin/main`'s flatten loop verbatim.
**Warning signs:** Context values appear as dicts in Groovy bindings.

### Type-aware push (recommended for this phase)
```python
def _push_runtime_state_to_bridge(self) -> None:
    if self.java_bridge is None:
        return
    if self.context_manager is not None:
        types = getattr(self.context_manager, "context_types", {})
        for k, v in self.context_manager.get_all().items():
            t = types.get(k)
            if t == "id_Float" and isinstance(v, float):
                # Force Java Float (Py4J defaults to Double otherwise)
                self.java_bridge.context[k] = \
                    self.java_bridge.gateway.jvm.java.lang.Float(v)
            else:
                # Py4J handles Decimal, int, bool, str, None correctly;
                # date/datetime handled by registered input converter
                self.java_bridge.context[k] = v
    if self.global_map is not None:
        for k, v in self.global_map.get_all().items():
            self.java_bridge.global_map[k] = v
```

## Open Questions

1. **Should the plan also fix `bridge.set_context` / `set_global_map` setters' `str(value)` coercion, OR keep them as-is and bypass via direct write?**
   - What we know: Direct write fixes R6 immediately for tMap/filter_rows/py_map without touching the setters. Fixing the setters would also benefit any future caller and the existing `base_component._resolve_java_expressions` flow (which currently has the same coercion bug for any typed value passed through batch one-time expression resolution).
   - What's unclear: Whether any existing caller depends on the str-coercion (e.g. for the executeOneTimeExpression Java side that may be more permissive with Strings than typed values).
   - Recommendation: Plan-phase decides. Lean toward fixing the setter (one line: drop `str(...)`) because the Java side accepts `Object` already and Py4J handles type dispatch. Lower-risk path is to bypass via direct dict write — keeps the change surface area minimal to this phase's scope.

2. **What is the exact reserved schema for `__errors__`? rowIndex required?**
   - What we know: Current Groovy dict shape is `{count, indices, messages}`; the Phase 05.4-04 D-06 contract mentions `errorMessage` and `errorStackTrace`.
   - What's unclear: Whether `rowIndex` should be a column (useful for join-back to user data) or absorbed elsewhere. SPEC L86-88 says "at minimum errorMessage (str) and errorStackTrace (str) columns".
   - Recommendation: Include `rowIndex` as a column for join-back to the user's catch-output user columns; document explicitly in 05.5-CONTEXT.md / plan-phase as a HOW decision.

3. **For R5 py_map, what's the right interface — bare dict or attribute-access wrapper?**
   - What we know: Talend convention is `context.X` (attribute), `globalMap.get("X")` (method). py_map's existing `_Row` (L70-80) demonstrates the attribute-access pattern.
   - What's unclear: Whether users expect `context["X"]` (dict access) to also work, and whether `globalMap.X` (attribute on globalMap) should work in addition to `globalMap.get("X")`.
   - Recommendation: Provide both interfaces — Mirror Talend exactly. A dict subclass with `__getattr__` (like `_Row`) covers both for context. For globalMap, add `.get(key)` method on top.

4. **Should R6 attempt the Float fix in this phase or defer with documented xfail?**
   - What we know: Py4J's protocol is hard-coded to send `float` as Double; the workaround (`gateway.jvm.java.lang.Float(value)`) requires type info from ContextManager.
   - What's unclear: Whether the workaround introduces autoboxing edge cases in Groovy (e.g. `Float` may auto-promote to `Double` in arithmetic context, defeating the test).
   - Recommendation: Attempt in this phase; if Groovy auto-promotion bites the test, escalate to xfail with documented gap per SPEC L79.

5. **Are there any other components beyond the 3 named (tMap, py_map, filter_rows) that run per-row Java/Groovy?**
   - What we know: SPEC explicitly enumerates these three and excludes others.
   - What's unclear: Whether tJavaRow exists in the new restructure and has the same gap.
   - Recommendation: SPEC is the authority; plan-phase should not expand scope. Note for future phase: tJavaRow (java_row_component) is in PROJECT.md's Active list as not-yet-built — it will need this same pattern when implemented.
