# Phase 05.5: tMap Context/GlobalMap Bridge Sync + `__errors__` Arrow Round-trip — Specification

**Created:** 2026-05-15
**Ambiguity score:** 0.07 (gate: ≤ 0.20)
**Requirements:** 9 locked (R6 expanded for production type coverage)

## Goal

Restore call-site bridge synchronization of `ContextManager` and `GlobalMap` state in tMap, py_map, and filter_rows so that `context.X` and `globalMap.X` references inside per-row Java/Groovy expressions resolve to their current Python-side values, and fix the Java-side `JavaBridge.convertTMapOutputsToArrow` to serialize `__errors__` as a real Arrow DataFrame so the D-06 reserved-column round-trip works end-to-end.

## Background

The engine restructure (current `feature/engine-restructure` branch) regressed the call-site sync pattern that `origin/main` had in place. On `main`, every bridge invocation site in `map.py` (3 sites: L399-403, L538-543, L856-860) explicitly pushed all `context_manager.get_all()` values via `java_bridge.set_context(k, v)` and all `global_map.get_all()` values via `java_bridge.set_global_map(k, v)` immediately before calling the bridge for row-level evaluation.

In the current branch, `Map._resolve_expressions` (map.py:226) and `PyMap._resolve_expressions` (py_map.py:104) explicitly skip `BaseComponent._resolve_java_expressions` — by design, because their `{{java}}` markers reference live row data that doesn't exist at config-resolution time. `FilterRows._resolve_java_expressions` (filter_rows.py:354) pops `advanced_cond` and calls `super()`, but if `advanced_cond` is the only `{{java}}` expression, `super()` finds no work and early-returns before reaching the context-sync block at base_component.py:372-384.

Net result: `JavaBridge.context` (Python wrapper's internal dict at bridge.py:98) and `JavaBridge.global_map` (bridge.py:99) start empty and only get populated by accident — if some other component in the same engine session triggered a config-level `{{java}}` substitution that incidentally pushed values. When the compiled Groovy script runs (`binding.setVariable("context", context)` at JavaBridge.java:736), `context` is an empty/stale Map. Every `context.X` reference evaluates to null.

**Empirical evidence (15-test spike, 2026-05-15):**

| Surface | context.X | globalMap.X | routine(context.X) | routine(row.X) |
|---|---|---|---|---|
| tMap output column | FAIL | FAIL | FAIL | PASS |
| tMap input filter | FAIL | FAIL | n/a | PASS |
| tMap output filter | FAIL | FAIL | n/a | PASS |
| tMap variable | FAIL | FAIL | n/a | n/a |
| FilterRows advanced_cond | FAIL | FAIL | n/a | PASS |

12 of 15 spike tests fail. Routines on row data pass because Java binds routine classes statically via `addRoutinesToBinding`; only context/globalMap require Python-side sync.

**Separately**, `JavaBridge.convertTMapOutputsToArrow` (Java side, near line 846) currently skips `__errors__` during Arrow serialization. The Phase 05.4-04 rewrite of `_route_catch_output_rejects` handles both shapes (raw Groovy `{count, indices, messages}` dict and the future DataFrame shape) but the Java-side gap means the D-06 reserved-column row-data round-trip can't fully verify today, leaving 2 of the Phase 05.4-07 strict-xfail tests pinned.

## Requirements

1. **R1 — tMap call-site context/globalMap sync**: Every bridge invocation site in `src/v1/engine/components/transform/map.py` that runs row-level Java/Groovy evaluation must push current `context_manager.get_all()` and `global_map.get_all()` state into the bridge before invocation, via a private helper method on `Map`.
   - Current: `map.py` L2143 (`execute_compiled_tmap_chunked`, active mode), L2177 (same, reject_mode), L3607 (`execute_tmap_preprocessing`) call the bridge without any preceding sync.
   - Target: A private method (suggested name `_push_runtime_state_to_bridge`) is defined on `Map`. It iterates `self.context_manager.get_all()` calling `self.java_bridge.set_context(k, v)` for each key, and iterates `self.global_map.get_all()` calling `self.java_bridge.set_global_map(k, v)` for each key. The helper is invoked at all 3 bridge-call sites before invocation. Bidirectional sync model: before pushing Python state, any pending Java-side writes from upstream tJavaRow have already flowed back via the existing `_sync_from_java` mechanism (this is the engine's sequential-execution invariant — no new sync code required here).
   - Acceptance: Spike tests A1, A2, A3, A4 (tMap column expression with `context.X`, `globalMap.X`, `routine(context.X)`, `routine(globalMap.X)`) pass against the real JVM bridge. `grep -c "_push_runtime_state_to_bridge\|set_context\|set_global_map" src/v1/engine/components/transform/map.py` returns ≥ 3 unique invocation sites.

2. **R2 — tMap input/output filter context/globalMap sync**: Filter expressions on tMap inputs and outputs must resolve `context.X` and `globalMap.X` correctly when routed through the compiled Groovy path.
   - Current: Spike tests B1, B2 (input filter with `context.X` / `globalMap.X`) and C1, C2 (output filter with same) all fail — filters return all rows because `context.threshold` evaluates to null in Groovy and the comparison fails silently.
   - Target: R1 fix transitively covers these (same bridge call sites). No additional code required if R1 is implemented at all three call sites.
   - Acceptance: Spike tests B1, B2, C1, C2 pass. Filter masks produced by `execute_compiled_tmap_chunked` honor the live context/globalMap values.

3. **R3 — tMap variable expressions with context/globalMap**: Variable definitions (`Var.X = expr`) that reference `context.X` or `globalMap.X` must resolve correctly so downstream column expressions see the computed Var values.
   - Current: Spike tests D1, D2 fail — `Var.envTag` computes `"[null]"` because context binding inside the compiled script is empty.
   - Target: R1 fix transitively covers this; no separate code path needed.
   - Acceptance: Spike tests D1, D2 pass.

4. **R4 — filter_rows advanced_cond context/globalMap sync**: `FilterRows._handle_advanced` must push context/globalMap to the bridge before invoking `execute_tmap_preprocessing` for advanced-condition evaluation.
   - Current: `filter_rows.py` `_handle_advanced` calls `execute_tmap_preprocessing` with no preceding sync. Spike tests E1, E2 fail (filter passes all rows because `context.threshold` is null in Groovy).
   - Target: A private helper (same pattern as R1) on `FilterRows` is invoked inside `_handle_advanced` before the bridge call.
   - Acceptance: Spike tests E1, E2 pass.

5. **R5 — py_map context/globalMap eval-namespace binding** *(reframed post-research: py_map is pure Python — has zero `java_bridge` references)*: `PyMap`'s per-row Python eval namespace must include `context` and `globalMap` views so that user expressions like `context.X` and `globalMap.get("X")` resolve at row evaluation time. The mechanism is namespace-binding, NOT bridge sync.
   - Current: `py_map.py` `_resolve_expressions` (L104) skips parent class. Per-row eval builds its namespace in `_build_namespace` (~L379-401) which omits `context` and `globalMap`. Any `context.X` reference in a py_map expression raises `NameError` or evaluates against an unrelated symbol.
   - Target: `PyMap._build_namespace` adds `context` (as `types.SimpleNamespace(**context_manager.get_all())` for attribute-style access, mirroring tMap's pattern at map.py:2368) and `globalMap` (as the live `GlobalMap` object so `globalMap.get("X")` works) to the eval namespace.
   - Acceptance: New unit tests in `tests/v1/engine/components/transform/test_py_map.py::TestPyMapContext` cover (a) `context.X` attribute access in a py_map output column expression, (b) `globalMap.get("X")` call in same. Pure-Python tests — no `@pytest.mark.java` required.

6. **R6 — Type fidelity matrix**: Context and globalMap values across the full Talend-supported scalar type system must round-trip correctly through `set_context` / `set_global_map` and be readable inside compiled Groovy expressions with native Java type semantics matching the corresponding Talend `id_*` type.
   - Current: No tests cover type fidelity for context/globalMap values in the matrix. `None` specifically caused Groovy ambiguous-method-overload crashes in spike test 5. Py4J's default Python→Java serialization behavior for `Decimal`, `date`, `datetime`, and the float/double distinction has not been verified against `set_context` / `set_global_map`. Talend production jobs use all of these types in context variables (`id_Date` is especially common for batch-window parameters).
   - Target: The phase locks the following 9 types as the acceptance matrix, mapping each to its Talend type ID and target Java runtime type:

     | Python type / source                          | Talend type ID  | Target Java type            | Notes                                       |
     |-----------------------------------------------|-----------------|-----------------------------|---------------------------------------------|
     | `int`                                         | `id_Integer`    | `java.lang.Integer`         | Talend `Integer` is 32-bit                  |
     | `int` (large)                                 | `id_Long`       | `java.lang.Long`            | Distinguished from `id_Integer` via type ID |
     | `str`                                         | `id_String`     | `java.lang.String`          |                                             |
     | `float` (32-bit aware)                        | `id_Float`      | `java.lang.Float`           | Python has no native float32; convert       |
     | `float` (default Python float, 64-bit)        | `id_Double`     | `java.lang.Double`          | Default Py4J path; verify precision         |
     | `decimal.Decimal`                             | `id_BigDecimal` | `java.math.BigDecimal`      | Py4J does NOT serialize Decimal natively    |
     | `datetime.date`                               | `id_Date`       | `java.util.Date` (00:00:00) |                                             |
     | `datetime.datetime`                           | `id_Date`       | `java.util.Date`            |                                             |
     | `None`                                        | any             | Java `null`                 | Must NOT crash Groovy compilation           |

     For `id_Date` (both `date` and `datetime`), the matrix also locks **4 datetime format variants** that production Talend jobs commonly use: ISO `yyyy-MM-dd HH:mm:ss`, ISO date-only `yyyy-MM-dd`, US `MM/dd/yyyy`, European `dd/MM/yyyy HH:mm`. Each format must round-trip such that `routines.TalendDate.parseDate(format, context.X)` inside a Groovy expression succeeds and produces the same epoch-millis value as the Python-side value would in pandas.

     Each of the 9 types has explicit test coverage for both `context.X` and `globalMap.X` in both column expressions and filter expressions. `None` values must produce a deterministic Groovy null without crashing the compiled script (downstream cast logic is the caller's responsibility, not this phase's).
   - Acceptance: Parametrized test matrix with **9 types × 2 namespaces (context, globalMap) × 2 surfaces (column, filter) = 36 test cells**, plus **4 datetime format variants × 2 namespaces = 8 format cells**, all pass against the real bridge. Any type that requires explicit Py4J serialization handling (likely `Decimal` and `datetime`) gets a small dedicated converter in `JavaBridge.set_context` / `set_global_map` to ensure correct Java runtime type. If a type cannot be made to round-trip cleanly within the phase budget, it is escalated to a separate phase with a written rationale — but the test for it stays in the matrix as `xfail(strict=True)` with a documented surviving gap (same pattern as Phase 05.4-07).

7. **R7 — Reject-output columns with context/globalMap**: Reject output columns of all three reject types (`is_reject`, `inner_join_reject`, `catch_output_reject`) must resolve `context.X` and `globalMap.X` references correctly when reject-routing fires.
   - Current: Phase 05.4 fixed reject column expressions for `row.X` references. No coverage for `context.X` / `globalMap.X` in reject column expressions. The compiled dual-invocation path (Phase 05.4-06) inherits the same bridge-sync gap from R1.
   - Target: A test per reject type pins that a reject output column expression containing `context.X` and `globalMap.X` produces correct values in the reject DataFrame.
   - Acceptance: Three new tests, one per reject type, all pass against the real bridge.

8. **R8 — Java-side `__errors__` Arrow serialization**: `JavaBridge.convertTMapOutputsToArrow` must serialize `__errors__` as a structured DataFrame instead of skipping it, so `_route_catch_output_rejects` receives a real DataFrame with `errorMessage` and `errorStackTrace` columns alongside user-defined catch columns.
   - Current: `JavaBridge.convertTMapOutputsToArrow:846` skips the `__errors__` entry; Python side receives a raw Groovy `{count: int, indices: List<Integer>, messages: List<String>}` dict.
   - Target: When `__errors__` is present in the compiled script's output map, the Java side serializes it as an Arrow stream with at minimum `errorMessage` (str) and `errorStackTrace` (str) columns, indexed by the error row position. `_route_catch_output_rejects` reads it via the same Arrow IPC path as other outputs. The current dual-shape handling in `map.py` can remain as a defensive fallback or be simplified — implementation detail for plan-phase.
   - Acceptance: A test catches a deliberately failing row, asserts the catch-reject DataFrame has the user's column values populated via `_evaluate_output_columns_py` semantics PLUS the framework-reserved `errorMessage` populated by Java side (D-06 contract from Phase 05.4-04). The 2 strict-xfail tests in `tests/v1/engine/components/transform/test_map_reject_catch.py::TestCatchOutputRejectCompiledD06` flip to PASS.

9. **R9 — Retroactive evaluation of Phase 05.4-07 strict-xfails**: After R1-R8 land, the 6 strict-xfail tests pinned by Phase 05.4-07 must be re-evaluated. Any that flip to GREEN with the new fixes are promoted to active (`xfail` removed); any that remain RED are documented with the surviving root cause and either pinned for a future phase or rewritten.
   - Current: 6 strict-xfail tests in `tests/v1/engine/components/transform/test_map_reject_filter.py::TestFilterRejectCompiled` (4 tests, L384/L406/L426/L447) and `tests/v1/engine/components/transform/test_map_reject_catch.py` (2 tests, L169/L207).
   - Target — research-grounded prediction: The 2 catch xfails directly cite `__errors__` Arrow round-trip and are expected to FLIP GREEN once R8 lands. The 4 filter-reject xfails have an **orthogonal root cause** (`_build_compiled_script` does not emit `evalOutput_<rej>` methods in the active-mode loop) which is NOT in this phase's scope as written. ## Boundaries

**In scope:**
- Call-site sync of `context_manager.get_all()` and `global_map.get_all()` to `java_bridge.context` / `java_bridge.global_map` (direct dict writes, NOT via `set_context` setter) in `map.py` and `filter_rows.py`
- `PyMap` eval-namespace binding of `context` (as `SimpleNamespace`) and `globalMap` (live `GlobalMap` object) — separate mechanism, no `java_bridge` involved (py_map is pure Python)
- Private helper per component: `_push_runtime_state_to_bridge` in Map / FilterRows (bridge-sync), and an inline namespace addition in PyMap's `_build_namespace` — three different code paths, no shared abstraction in BaseComponent
- Java-side `__errors__` Arrow serialization fix in `JavaBridge.convertTMapOutputsToArrow`
- Test matrix expansion: parametrize over **9 Talend-typed values** (`id_Integer / id_Long / id_String / id_Float / id_Double / id_BigDecimal / id_Date` with `date` and `datetime` Python sources / `None`) × `context / globalMap` × `column expression / filter expression`, plus **4 datetime format variants** (ISO datetime, ISO date, US, European) × 2 namespaces
- Small Py4J converter helpers in `JavaBridge.set_context` / `set_global_map` for types that don't natively serialize (`Decimal` → `java.math.BigDecimal`, `datetime.date` / `datetime.datetime` → `java.util.Date`, explicit `float`→`Float` vs `float`→`Double` distinction driven by Talend type ID)
- Reject-output column tests for all three reject types with context/globalMap references
- Retroactive review of the 6 strict-xfail tests from Phase 05.4-07 (promote where fix lands, document where it doesn't)
- Phase 14 coverage gate must still pass on `map.py` (≥ Phase 05.4 baseline of 88.7%)

**Out of scope:**
- Changes to `base_component.py` — its `_resolve_java_expressions` is correct for its one-time config-substitution purpose; per-row sync is the component's concern
- Changes to `JavaBridge` Python API signatures (`execute_compiled_tmap_chunked`, `execute_tmap_preprocessing` keep their current contracts)
- Fixing the `set_context` / `set_global_map` setter's `str(value)` coercion at bridge.py:790/801 — bypassed via direct dict write; setter remains as-is for callers that rely on it
- Shared utility helper on `BaseComponent` — explicitly rejected to avoid the appearance of automatic sync; per-component private helpers are the locked pattern
- Other components with `{{java}}` markers (`row_generator`, `set_global_var`, `file_output_delimited`) — they handle config-level `{{java}}` substitution via base class, which already syncs context for its own use. Per-row evaluation only exists in the three components named above
- Lifting `map.py` / `py_map.py` coverage to the 95% Phase 14 floor — separate coverage phase, not this fix's responsibility
- `JavaBridge` write-back races beyond the existing `_sync_from_java` model — sequential component execution remains the invariant; no new locking
- Changes to `ContextManager.resolve_string` or `ContextManager.resolve_dict` — the Python-side resolution paths used by `_join_context_only` and config substitution are correct and stay untouched

## Constraints

- **No breaking changes** to JSON config format or JavaBridge public API — engine restructure has already shipped to teammates' workflows on this branch
- **Talend parity is non-negotiable** — every test case must produce the same output a Talend job with the equivalent `.item` config produces (the 12 currently-failing spike cases all have well-defined Talend-correct outputs)
- **Java bridge architecture is fixed** — Py4J + Arrow, no replacement
- **Phase 14 coverage gate** must remain passing on `src/v1/engine` per-module floor of 95% (allowing `map.py` and `py_map.py` to stay at their current sub-floor baselines per the Phase 14 known-deficit list; this phase must not regress those baselines)
- **No new dependencies** — fix uses only existing `JavaBridge` setters, existing test harness, and Py4J converter primitives that already exist in the runtime
- **Py4J serialization compatibility** — type-converter helpers must not break existing callers; if a type cannot be cleanly serialized in Py4J's protocol (theoretical edge case), the phase escalates rather than introducing a workaround that violates the no-new-deps rule
- **Sync mechanism is direct `bridge.context[k] = v` / `bridge.global_map[k] = v` writes** *(research-locked)* — NOT the existing `bridge.set_context(key, value)` setter. The setter at `src/v1/java_bridge/bridge.py:790` does `self.java_bridge.setContext(key, str(value))` which string-coerces every value before sending to Java. In practice this is masked by the `this.context.putAll(contextVars)` overwrite that fires inside each bridge method (verified at JavaBridge.java:178/279/308/368/473/579), but bypassing the setter (a) removes the masking dependency, (b) avoids a redundant round-trip per key, and (c) preserves type fidelity by relying on Py4J's native protocol serialization of `self.context` as a Map at the bridge-call boundary. The setter itself is NOT modified in this phase (out of scope).
- **`__rejectMode__` flag preservation** *(research-locked)* — the per-key direct-write pattern (`bridge.context[k] = v` in a loop) is mandatory; `bridge.context.clear()` followed by reassignment OR `bridge.context = dict(...)` is FORBIDDEN. The `__rejectMode__` flag is set inside `execute_compiled_tmap_chunked` (bridge.py L604) and any sync that runs between L604 and the actual Java call MUST NOT clobber it. The sync helper runs BEFORE `execute_compiled_tmap_chunked` enters (so before the flag is set) — but per-key writes also guarantee that even if ordering changes later, the flag survives.
- **ASCII-only logging** — no emojis/unicode in any new log lines (project memory `feedback_ascii_logging`)
