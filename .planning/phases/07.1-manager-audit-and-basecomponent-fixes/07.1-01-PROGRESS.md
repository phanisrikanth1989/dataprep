## Finding Verdicts

### CR-01 (= G-13): validate_schema applied to reject DataFrame raises DataValidationError on expected NULLs

VERDICT: REAL
- **Evidence:** `base_component.py:693-694` — `_apply_output_schema_validation` calls `self.validate_schema(reject_df, reject_schema)` without relaxing nullability. `validate_schema:737-740` raises `DataValidationError` when `not nullable and result[col_name].isna().any()`.
- **Reject rows often contain NULLs in non-nullable columns** (that is the reason they were rejected). This crashes on the first reject row that triggered the rejection.
- **Alias:** G-13 == CR-01. Recorded once.

### CR-02 (= G-14): _apply_decimal_precision crashes on string precision

VERDICT: REAL
- **Evidence:** `base_component.py:780` — `quantize_str = Decimal(10) ** -precision`. The parameter `precision: int` in the signature is a type hint only; Python does not enforce it. JSON configs can produce `"precision": "4"` (string). `-"4"` raises `TypeError: bad operand type for unary -: 'str'`.
- **Alias:** G-14 == CR-02. Recorded once.

### WR-01 (= G-01): _enforce_schema_column_order uses wrong default for datetime columns

VERDICT: REAL
- **Evidence:** `base_component.py:596-605` — missing column fill uses `if nullable: main_df[col] = pd.NA` for ALL types, including datetime. `pd.NA` is not a valid datetime value and will produce an object-dtype column of NA scalars. The non-nullable branch uses `""` for everything except int/float/Decimal/bool.
- Missing branch: `datetime` + nullable → should be `pd.NaT`; `datetime` + non-nullable → should be `pd.Timestamp(0)`.
- **Alias:** G-01 == WR-01. Recorded once.

### WR-02: _enforce_schema_column_order adds missing columns via scalar assignment, breaking dtype on empty DataFrames

VERDICT: REAL
- **Evidence:** `base_component.py:596-605` — `main_df[col] = pd.NA` uses scalar broadcast. On populated DataFrames this produces an `object` dtype column. On empty DataFrames the assignment silently does nothing (pandas ignores scalar assignment to missing columns on empty frames), so the column is never added at all.
- The fix must use `pd.Series([value] * len(df), dtype=...)` to work on both empty and populated frames.

### WR-03: _apply_output_schema_validation early-exits on empty DataFrames

VERDICT: REAL
- **Evidence:** `base_component.py:686` — `if main_df is not None and isinstance(main_df, pd.DataFrame) and not main_df.empty:` — empty DataFrames skip validation entirely.
- **Evidence:** `base_component.py:571` — `_enforce_schema_column_order` has `if main_df is None or not isinstance(main_df, pd.DataFrame) or main_df.empty: return result` — same early exit.
- **Evidence:** `base_component.py:622` — `not reject_df.empty` guards the reject path in `_enforce_schema_column_order`.
- Fix: remove `empty` guards so column order + schema validation applies to empty DataFrames.

### G-02: Decimal coercion not applied when no precision specified

VERDICT: REAL
- **Evidence:** `base_component.py:755-758` — `_apply_decimal_precision` is only called when `precision is not None`. When `precision` is absent, `_coerce_column_type` falls through to the object type path (line 812 — no explicit branch for Decimal type in `_coerce_column_type`). String "123.45" stays a string.
- Fix: In `_coerce_column_type`, add explicit Decimal branch that converts string/numeric values to `Decimal` objects, separate from precision step.

### G-03: Float precision not applied

VERDICT: REAL
- **Evidence:** `base_component.py:755-758` — `_apply_decimal_precision` is only called for `col_type == "Decimal"`. Float columns with `precision` set in schema get no rounding. Talend rounds float/double columns to schema precision.

### G-04: date_pattern schema attribute not used in _coerce_column_type

VERDICT: REAL
- **Evidence:** `base_component.py:815-816` — `_coerce_column_type` for `datetime` type: `df[col_name] = pd.to_datetime(df[col_name], errors="coerce")`. The `col_def` dict is not passed in at all (current signature is `(self, df, col_name, col_type, nullable)` — no `col_def`), so `date_pattern` cannot be consulted.
- Talend's `date_pattern` (Java pattern like "dd/MM/yyyy") should drive format-aware parsing.

### G-05 (D-11): die_on_error not checked in schema validation

VERDICT: REAL
- **Evidence:** `base_component.py:737-740` — `validate_schema` raises `DataValidationError` unconditionally when `not nullable and result[col_name].isna().any()`. `die_on_error` is read at line 163 and stored on `self.die_on_error` but never consulted in `validate_schema` or `_apply_output_schema_validation`.
- When `die_on_error=False`, schema violations should route to reject flow instead of raising.

### G-10 (D-12): Streaming validates after concat, not per-chunk

VERDICT: REAL
- **Evidence:** `base_component.py:414-455` — `_execute_streaming` collects chunk results, then at line 448 calls `pd.concat(flow_chunks[key], ignore_index=True)` for each flow key and returns the merged result. Steps 7b (`_enforce_schema_column_order`) and 7c (`_apply_output_schema_validation`) run in `execute()` AFTER `_execute_streaming` returns the full concatenated result. No per-chunk validation happens.

### G-12 (D-10): No treat_empty_as_null schema attribute

VERDICT: REAL
- **Evidence:** `base_component.py:794-835` — `_coerce_column_type` has no `treat_empty_as_null` logic. For string columns, empty strings `""` pass through unchanged. For numeric columns, `pd.to_numeric("", errors="coerce")` silently produces `NaN` — the coercion behavior is not configurable per column.
- Fix: consult `col_def.get("treat_empty_as_null")` with defaults: True for numeric/datetime/Decimal, False for str.
