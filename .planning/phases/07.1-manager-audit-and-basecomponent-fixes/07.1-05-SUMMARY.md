## Fixes Applied

### ENG-CR-05: Remove manual validate_schema from filter_rows._process
`src/v1/engine/components/transform/filter_rows.py` lines 250-253 removed. BaseComponent step 7c
(`_apply_output_schema_validation`) owns schema validation. Subclasses must not duplicate it.

### ENG-CR-07: Symmetric pop+restore in _resolve_java_expressions
Old code set `self.config["advanced_cond"] = ""` which mutated config. New code uses
`original = self.config.pop("advanced_cond")` with `finally: self.config["advanced_cond"] = original`.
Only pops when `{{java}}` marker is present (gate per T-7.1-05-01).

### ENG-WR-07: removeprefix for {{java}} marker stripping
Both `_build_reject_error_message` and `_handle_advanced` now use
`advanced_cond.removeprefix("{{java}}")` (Python 3.9+) instead of `advanced_cond[8:]`.

### WR-07/ENG-IN-03: Numeric path always taken when value parses as numeric
Removed `and numeric_col.notna().any()` guard from `_compare`. Now: if `value` parses as float,
numeric comparison is always attempted (NaN values in column produce False, consistent with
Talend null-never-matches).

### WR-08: getattr guard for self.inputs
`self.inputs[0] if self.inputs else "row1"` -> `inputs_attr = getattr(self, "inputs", None);
main_table_name = inputs_attr[0] if inputs_attr else "row1"`. No AttributeError in standalone use.

### ENG-IN-06: numpy import at module top
`import numpy as np` moved from inside `_handle_advanced()` to module-level imports section.

### Talend parity: use_advanced=True applies ONLY advanced mask
Old code ANDed `advanced_mask & simple_mask` when `use_advanced=True`. Talend grays out simple
conditions when advanced mode is active. Fixed to use only `advanced_mask`.

### BaseComponent stats ownership
Removed manual `self._update_stats(...)` call from `filter_rows._process`. With 7.1-01
BaseComponent, if a component calls `_update_stats`, the base skips its own
`_update_stats_from_result` (no double-count). Removing the call lets base handle stats cleanly.
Updated `test_stats_updated` to expect single-counted stats: NB_LINE=5, OK=4, REJECT=1.

### ENG-CR-04 PRODUCER: per-flow inputs map in _propagate_input_schemas
`to_schema["input"] = list(upstream_output)` (last-write-wins, lost first flow for multi-input)
replaced by:
```python
inputs_map = to_schema.setdefault("inputs", {})
flow_name = flow.get("name") or from_id
inputs_map[flow_name] = list(upstream_output)
to_schema["input"] = list(upstream_output)  # back-compat preserved
```

### ENG-WR-09: Case-tolerant outputs lookup
```python
norm_map = {str(k).upper(): v for k, v in outputs_map.items()}
connector_key = (flow.get("type") or "").upper()
upstream_output = norm_map.get(connector_key)
```
Handles lowercase outputs keys (e.g. `{"filter": [...]}`) matching uppercase flow types.

### ENG-CR-04 CONSUMER: _schema_for_flow() in tMap + engine.py initialization
`engine.py`: `component.schema_inputs_map = comp_config.get('schema', {}).get('inputs', {})`.
`map.py`: new `_schema_for_flow(flow_name)` method reads from `self.schema_inputs_map[flow_name]`
when present, falls back to `self.input_schema` for pre-7.1 back-compat.

### ENG-WR-08: Remove false eval() and FUNCTION claims from converter
Removed `_engine_gap_keys` block with two false claims. No current engine gaps exist.
Updated converter component tests to not expect the stale entries.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] use_advanced=True logic applied simple conditions incorrectly**
- **Found during:** Task 3 (pre-existing test `test_reads_advanced_cond_not_advanced_condition` still failing)
- **Issue:** When `use_advanced=True`, filter_rows ANDed `advanced_mask & simple_mask`. Talend's tFilterRow grays out simple conditions when advanced mode is active -- only the advanced expression should apply.
- **Fix:** Changed `if use_advanced:` branch to use only `advanced_mask = self._handle_advanced(input_data)`.
- **Files modified:** `src/v1/engine/components/transform/filter_rows.py`
- **Commit:** `715193d`

**2. [Rule 2 - Missing critical functionality] BaseComponent stats ownership not reflected in existing test**
- **Found during:** Task 2 (test_stats_updated had wrong double-counting expectation)
- **Issue:** test_stats_updated expected NB_LINE=10 (old: component + base both count). With 7.1-01 BaseComponent, `_update_stats` in `_process` sets `_stats_set_by_component=True` so base skips. New correct expectation is NB_LINE=5.
- **Fix:** Updated test to expect NB_LINE=5, NB_LINE_OK=4, NB_LINE_REJECT=1. Removed manual `_update_stats` call from filter_rows._process.
- **Files modified:** `tests/v1/engine/components/transform/test_filter_rows.py`
- **Commit:** `ec70928`, `715193d`

**3. [Rule 2 - Missing critical functionality] ENG-CR-04 consumer requires engine.py initialization**
- **Found during:** Task 5 (map.py has no access to schema.inputs without engine.py change)
- **Issue:** The plan example used `self.config.get("inputs", {})` but that's the join config (not schema). The per-flow schema lives at `comp_config["schema"]["inputs"]` which the engine sets as `component.input_schema` only for the single `"input"` key. Needed a new `component.schema_inputs_map` attribute.
- **Fix:** Added `component.schema_inputs_map = comp_config.get('schema', {}).get('inputs', {})` to engine.py initialization, and `_schema_for_flow()` reads from `getattr(self, "schema_inputs_map", None)`.
- **Files modified:** `src/v1/engine/engine.py`, `src/v1/engine/components/transform/map.py`
- **Commit:** `2bf750e`

**4. [Rule 1 - Bug] Existing converter filter_rows tests expected stale needs_review entries**
- **Found during:** Task 6 (test_needs_review_has_entries expected >= 1 entries after removal)
- **Issue:** `TestNeedsReview::test_needs_review_has_entries` asserted `len >= 1`, but all entries were false and correctly removed.
- **Fix:** Updated test to check `test_needs_review_no_false_eval_claim` (no eval claim) instead of asserting presence of entries.
- **Files modified:** `tests/converters/talend_to_v1/components/transform/test_filter_rows.py`
- **Commit:** `0815841`

### ENG-WR-06: Confirmed no action needed in filter_rows

D-21 `_rename_reserved_reject_columns` in BaseComponent handles the MAIN flow only. The test
`test_user_errormessage_renamed` was updated to verify this behavior: user `errorMessage` is
renamed to `errorMessage_user` in the MAIN output. In the REJECT output, filter_rows adds its
engine diagnostic under `errorMessage` (user's value superseded, documented). This is correct
per the D-21 scope statement ("Only applies to the 'main' flow").
