## Findings Verification

### CR-05: list_object returns Python list, not delimited string

**VERDICT: REAL — Phase 6 D-09 superseded by Talaxie messages.properties source**

**Current code (aggregate_row.py lines 138-144):**
```python
if func_name in ("list", "list_object", "union"):
    if func_name == "union":
        logger.warning("'union' function treated as list aggregation -- ...")
    if ignore_null:
        return lambda x: list_delimiter.join(x.dropna().astype(str))
    else:
        return lambda x: list_delimiter.join(x.astype(str))
```

All three functions produce a delimited string. This is wrong for `list_object`.

**Talaxie source confirms:** `tAggregateRow_messages.properties` line:
`LIST_DELIMITER.NAME=Delimiter (only for list operation)`

The `list_delimiter` property explicitly applies ONLY to the `list` operation (i.e., `list` / string join). `list_object` produces a Java `List<Object>` — the Python equivalent is a Python list. Phase 6 D-09 was written when the Talaxie source was not consulted; it assumed `list_object` behaved like `list`. That assumption was wrong.

**Fix required:** Separate the three branches. `list_object` returns `.tolist()`. `list` returns delimiter-joined string.

### CR-05-bis: union should deduplicate and sort before joining

**VERDICT: REAL**

**Current code:** `union` falls through to the same lambda as `list` — no deduplication. Talend's `union` aggregator (in the tAggregateRow context) collects distinct values and joins them.

**Fix required:** `union` returns `list_delimiter.join(sorted(set(...)))`

### WR-09: median with use_financial_precision falls back to float silently

**VERDICT: REAL**

**Current code (lines 147-150):**
```python
if func_name == "median":
    if use_financial_precision:
        # For median, convert to float since Decimal median is complex
        return lambda x: x.dropna().astype(float).median() if skipna else x.astype(float).median()
    return lambda x: x.median(skipna=skipna)
```

The comment "convert to float" is silent — no warning is emitted. For financial/regulatory calculations that explicitly enable `use_financial_precision`, silent precision loss is a correctness gap.

**Fix required:** Emit a one-time-per-execute warning via `_warned_median_decimal` flag.

### WR-11: groupby uses sort=True (alphabetizes groups, not input-order)

**VERDICT: REAL**

**Current code (line 394):**
```python
df.groupby(group_cols, sort=True)
```

Talend's tAggregateRow uses a LinkedHashMap internally, which preserves insertion order (first-seen group order). `sort=True` reorders output alphabetically by group key. This breaks job output ordering for any downstream component that relies on group order matching input order.

**Fix required:** `sort=False`

### WR-10: count + ignore_null=False should count all rows (including nulls)

**VERDICT: WRONG — keep current behavior, do NOT fix**

**Current code (line 125-127):**
```python
if func_name == "count":
    # count always counts non-null values
    return "count"
```

**Why WRONG:** Pandas `"count"` agg = `COUNT(col)` in SQL = non-null count. This is the SQL convention and matches Talend's documented behavior: "calculates the number of rows" without a null-inclusion clause. The audit finding assumed Talend would count all rows when `ignore_null=False`, but:
1. No Talend job-runtime source available to verify.
2. SQL convention (ANSI): `COUNT(col)` = non-null regardless of any null-handling flag.
3. Talend docs (help.qlik.com/talend tAggregateRow): "count = counts the number of rows" — no specification of null inclusion under ignore_null=False.

**Action:** Add a clarifying comment in the code. Document as WRONG in SUMMARY.
