## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] CR-06 validation checks effective (unescaped) separator, not raw config string**
- **Found during:** Task 3 (test failure: `test_tab_separated_output`)
- **Issue:** `_validate_config` initially checked `len(field_sep_raw) > 1` where `field_sep_raw = "\\t"` (2-char string). The escape sequence `"\\t"` should be valid (unescapes to 1-char tab), but the raw check raised incorrectly.
- **Fix:** Changed check to `len(_unescape_separator(field_sep_raw)) > 1`, so only EFFECTIVE multi-char separators (like `"|;"`) trigger the error.
- **Files modified:** `src/v1/engine/components/file/file_output_delimited.py`
- **Commit:** `6b6a075`

**2. [Rule 1 - Bug] TestMultiCharDelimiter tests conflicted with CR-06**
- **Found during:** Task 3 (pre-existing tests failed after CR-06 implementation)
- **Issue:** 4 of 6 `TestMultiCharDelimiter` tests used `fieldseparator="|;"` without `csv_option=True` and expected successful writes via the old silent raw-mode fallback. CR-06 now raises `ConfigurationError` for this case.
- **Fix:** Updated tests to add `csv_option=True` (correct usage) and added `test_multichar_delimiter_no_csv_raises` to explicitly test the CR-06 error path.
- **Files modified:** `tests/v1/engine/components/file/test_file_output_delimited.py`
- **Commit:** `6b6a075`

**3. [Rule 1 - Design] TestPassthrough assertion redesigned for BaseComponent post-validation reality**
- **Found during:** Task 3 (test design)
- **Issue:** Plan spec said "returned main has untruncated values". But BaseComponent's step 7c (`_apply_output_schema_validation`) runs AFTER `_process` returns and truncates string values via `output_schema.length`. With the CR-09 fix, truncation happens ONCE (BaseComponent), not twice (component + BaseComponent). The passthrough integrity test should verify the ORIGINAL df is not mutated in-place, not that the returned result bypasses BaseComponent.
- **Fix:** Test now verifies `df["name"]` values are unchanged AFTER calling `execute(df)` — proving `_process` does not mutate the original in-place. This is the actual CR-09 regression proof.
- **Files modified:** `tests/v1/engine/components/file/test_file_output_delimited.py`
- **Commit:** `44c5256`

## Addendum (2026-04-29) -- CR-06 contract superseded

The `CR-06-ConfigurationError-gate` contract recorded above was post-7.1 review found to be **over-strict** -- it rejected multi-char delimiter configs that Talend itself accepts and processes correctly. Primary-source review of Talaxie tdi-studio-se confirmed:

- `csv_option=True` + multi-char delimiter: Talend silently truncates to first char (`com.talend.csv.CSVWriter.setSeparator(char)` API constraint, tFileOutputDelimited_main.javajet:645-651)
- `csv_option=False` + multi-char delimiter: Talend preserves full multi-char string (BufferedWriter.write)

Updated contract (commit 43762c8 + this quick task 260429-hc2):

- `_validate_config` no longer rejects multi-char delimiters
- csv_option=True + multi-char -> truncate to first char with warning (Talend parity)
- csv_option=False + multi-char -> preserved as-is (Talend parity)
- Test suite updated to match.
