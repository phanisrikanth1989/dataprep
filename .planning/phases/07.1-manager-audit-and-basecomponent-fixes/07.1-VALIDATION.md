# Phase 07.1 — Validation Strategy

> Per-phase validation contract for feedback sampling during execution. Fix-phase scope: most "validation" is built into D-13's verify-findings task at the start of every plan. Per-task automated verification map populated by planner during step 8.

---

## Test Infrastructure

| Property | Value |
|----------|-------|
| **Framework** | pytest (Python), JUnit (Java routines) |
| **Config file** | `pyproject.toml` (pytest section), `src/v1/java_bridge/java/pom.xml` (JUnit via Surefire) |
| **Quick run command** | `pytest tests/v1/engine/ -m unit -q` |
| **Full suite command** | `pytest tests/ && (cd src/v1/java_bridge/java && mvn test)` |
| **Estimated runtime** | ~30s unit / ~3 min full (incl. @pytest.mark.java + Maven) |

---

## Sampling Rate

- **After every task commit:** `pytest tests/v1/engine/{affected_module}/ -q` (file-scoped)
- **After every plan completion:** `pytest tests/v1/engine/ -m unit -q` (full unit suite for engine)
- **After Wave 1 complete:** Full suite — must be green before Wave 2 starts (BaseComponent contract is foundation)
- **Before `/gsd-verify-work`:** Full suite green
- **Max feedback latency:** ~30s for unit, ~3 min full

---

## Per-Task Verification Map

To be populated by the planner during step 8 of plan-phase. Each task gets a row with:
- Task ID, plan, wave
- Finding ID resolved (e.g., CR-01, ENG-CR-04, G-04)
- Test type: unit / integration / java-bridge / regression
- Automated command (pytest selector or mvn target)
- Verification status
