## Summary

This is a **fix phase**, not a feature phase. Research focused on (a) verifying disputed audit claims against authoritative sources (Talaxie GitHub mirror of Talend Open Studio, OpenDAS Talend routines snapshot, official Talend docs at help.qlik.com), (b) reading current source to confirm bug presence, (c) cross-checking findings against prior phase decisions where they contradict.

**Headline findings:**
1. **CR-05 (`list_object` returns string, not list) — REAL bug.** Phase 6 D-09 was wrong. Talaxie's `tAggregateRow_messages.properties` has `LIST_DELIMITER.NAME=Delimiter (only for list operation)` — `list_delimiter` is **not** used for `list_object`. `list_object` produces a Java List/Object; engine must produce a Python list.
2. **CR-07, CR-08, WR-14, IN-01 — WRONG findings.** Talend upstream source (OpenDAS mirror) shows our code matches upstream verbatim:
   - `Mathematical.INT(String)` is `Integer.valueOf(e)` in Talend → throws on `"100.5"` (parity preserved).
   - `TalendDate.parseDate` wraps `ParseException` in `new RuntimeException(e)` (parity preserved).
   - `StringHandling.LEN(null)` returns `-1` in Talend (parity preserved).
   - `DataOperation.CHAR(int)` uses `Character.forDigit(i, 10)` in Talend (parity preserved).
   - The audit assumed "sensible" Talend behavior that **doesn't match what Talend actually ships**. Per CLAUDE.md "feature parity is non-negotiable" — fixing these would *break* parity.
3. **WR-15 (INSTR bounds check uses wrong variable) — REAL bug.** Code uses `defaultStart` (literal `1`) in length guard before assigning user `start`. Identical to upstream Talend bug. Fixing improves robustness without breaking parity (because the broken path is never hit when start is in-range, and Talend itself crashes the same way).
4. **WR-10 (count + ignore_null=False) — NEEDS-INVESTIGATION (verdict: WRONG / not a bug).** Talend docs do not specify count's behavior under `ignore_null=False`. SQL convention: `COUNT(col)` = non-null count regardless. Phase 6 D-11 said `skipna=False` for non-count aggregates only — count is the conventional carve-out. Recommendation: leave current behavior, document.
5. **CR-09 / ENG-CR-06 (validate_schema mutates input) — REAL.** Confirmed via direct read of `file_output_delimited.py:160-161, 214` — the prior fix was never applied. Manager added a *second* mutation site (`_apply_date_patterns`) which compounds the leak.
6. **ENG-CR-04 (`_propagate_input_schemas` multi-input overwrite) — REAL.** Direct read of `converter.py:317`: `to_schema["input"] = list(upstream_output)` runs once per flow with target = same component, so the LAST flow wins. Multi-input components (tMap with main + lookup) can't get distinct schemas per connector.
7. **G-12 (`treat_empty_as_null`) — Talend default for string is "treat as empty string, not null"**, for numeric is "coerce empty → null and reject if non-nullable" (matches Phase 4 D-06 reject behavior). Our default in D-10 is correct.

**Primary recommendation:** Plans 7.1-08 (Java routines) and 7.1-06 (aggregate) should drop the WRONG findings before any code changes — task-1 verification will save the team from breaking parity. All other plans proceed as scoped.

### Pattern 1: BaseComponent contract for `_validate_config()`

**What:** Abstract method on `BaseComponent`. Returns `None`. Raises `ConfigurationError` on invalid config. Receives unresolved config (context vars not yet substituted).

**When to use:** Every subclass. Phase 1 D-13 makes this required.

**What:** Reserve `errorMessage` and `errorCode` as engine-managed columns on the reject DataFrame. If the input DataFrame already has a column named `errorMessage` (legitimate user data), rename it to `errorMessage_user` before the engine attaches its own.

**When to use:** Any component that produces a reject flow with `errorMessage` (file_input_delimited, file_output_delimited, filter_rows, base_component for SCHEMA_VIOLATION).

### Pattern 3: Per-chunk streaming validation (D-12, G-10 fix)

**What:** In `_execute_streaming`, run `_enforce_schema_column_order` and `_apply_output_schema_validation` *per chunk* (not after concatenation). Reject rows from each chunk get appended to a per-chunk reject list, concatenated at the end. Memory stays bounded.

**When to use:** Inside the streaming branch of `BaseComponent._execute_streaming`. Replaces the current flow which concatenates everything then validates.

**Example:**
```python
def _execute_streaming(self, input_data):
    chunk_size = self.config.get("chunk_size", 10000)
    main_chunks, reject_chunks = [], []
    other_chunks = {}
    for start in range(0, len(input_data), chunk_size):
        chunk = input_data.iloc[start:start + chunk_size]
        chunk_result = self._process(chunk)
        # Validate per chunk (G-10 fix)
        chunk_result = self._enforce_schema_column_order(chunk_result)
        chunk_result = self._apply_output_schema_validation(chunk_result)
        if chunk_result.get("main") is not None:
            main_chunks.append(chunk_result["main"])
        if chunk_result.get("reject") is not None:
            reject_chunks.append(chunk_result["reject"])
        for k, v in chunk_result.items():
            if k not in ("main", "reject") and isinstance(v, pd.DataFrame):
                other_chunks.setdefault(k, []).append(v)
    return {
        "main": pd.concat(main_chunks, ignore_index=True) if main_chunks else pd.DataFrame(),
        "reject": pd.concat(reject_chunks, ignore_index=True) if reject_chunks else None,
        **{k: pd.concat(v, ignore_index=True) for k, v in other_chunks.items()},
    }
```

### Pattern 4: `treat_empty_as_null` per-column attribute (D-10)

**What:** New schema column attribute. Default `true` for `int`/`float`/`Decimal`/`datetime` columns; default `false` for `str` columns. Consulted in `_coerce_column_type` to decide whether `""` → `NaN` (for numeric) or stays as `""` (for string).

**When to use:** All `_coerce_column_type` calls (every component flowing through `_apply_output_schema_validation`).

**Example:**
```python
def _coerce_column_type(self, df, col_name, col_type, nullable, treat_empty_as_null):
    if col_type == "str":
        # String columns: empty stays empty unless explicitly opted in
        if treat_empty_as_null:
            df[col_name] = df[col_name].replace("", pd.NA)
        return df
    # Numeric / datetime: empty → NaN (unless explicitly opted out)
    if not treat_empty_as_null:
        # Preserve empty string as a non-null sentinel? Not coherent for numeric;
        # raise instead so callers don't silently get the wrong type.
        empty_mask = (df[col_name] == "") if df[col_name].dtype == object else None
        if empty_mask is not None and empty_mask.any():
            raise DataValidationError(
                f"Column '{col_name}' has empty strings but treat_empty_as_null=false "
                f"and column is non-string type"
            )
    # ... existing coercion logic ...
```

### Anti-Patterns to Avoid

- **Component calling `validate_schema` inside `_process()`** (ENG-CR-05 in current `filter_rows.py:251-253`): The base class runs validation in step 7c automatically. Subclass calls duplicate work and can race with `_enforce_schema_column_order` (which fills missing columns). [VERIFIED: `src/v1/engine/components/transform/filter_rows.py:251-253` — direct read]
- **Mutating `self.config` mid-execute as a "trick"** (ENG-CR-07 in `filter_rows._resolve_java_expressions`): Phase 1 D-14 says config is re-derived at start of every `execute()`, so transient mutation works *if* the contract is one-execute-per-call. But the pattern is brittle and breaks on iterate. Use `pop` + restore symmetrically OR (cleaner) signal "skip this key" via a class attribute the parent checks. [VERIFIED: `filter_rows.py:346-363`]
- **Mutating input passthrough** (CR-09 in `file_output_delimited.py:160-161, 214`): A sink component must return the *upstream* DataFrame as `main`, not its own truncated/coerced view. Apply formatting to a working copy. [VERIFIED: `file_output_delimited.py:158-161, 214` — direct read]
- **Per-row `iterrows()` for vectorizable transforms** (ENG-CR-02 in `normalize.py:153`): Build vectorized via `.str.split().explode()`; `iterrows()` erases dtypes and is O(n²) memory. [VERIFIED: `normalize.py:151-205`]

## Common Pitfalls

### Pitfall 1: "Fixing" a routine to match what looks reasonable when Talend itself is the unreasonable behavior

**What goes wrong:** Audit reads `Numeric.INT(String e) { return Integer.valueOf(e); }`, sees that this throws on `"100.5"`, and concludes "Talend must accept '100.5' and truncate" — proposes change to `Double.parseDouble(e.trim())` cast to int.

**Why it happens:** Confidence bias from "what would a reasonable ETL tool do" overrides the empirical question "what does this specific tool actually do?"

**How to avoid:** For every routine fix, fetch upstream Talend source (Talaxie or OpenDAS mirror) and grep for the exact method. If the upstream and our code are byte-identical, the audit finding is WRONG. Apply a "preserve parity" rule per CLAUDE.md.

**Warning signs:** Audit text uses phrases like "Talend behavior is well-documented to..." without a citation, or "users expect..." without a job example. Compare to actually-cited findings (e.g., WR-15 INSTR bounds — has line numbers, has a reproducer).

[VERIFIED: Compared `routines/Numeric.java`, `routines/StringHandling.java`, `routines/DataOperation.java`, `routines/TalendDate.java` against OpenDAS mirror at https://github.com/OpenDAS/opendas-talend — fetched via raw.githubusercontent.com. All four flagged routines are byte-identical to upstream Talend.]

### Pitfall 2: Re-validating in `_process()` because "I might know more about my output schema than the base class"

**What goes wrong:** Subclass calls `self.validate_schema(df, output_schema)` inside `_process()`, then BaseComponent's `_apply_output_schema_validation` runs it again. Double work; race with column-fill in `_enforce_schema_column_order` if first validation rejects rows that the base class would have filled.

**Why it happens:** Component author either (a) didn't know base does it, or (b) saw the base class run *after* `_process` and tried to "front-load" validation.

**How to avoid:** ENGINE_COMPONENT_PATTERN.md Rule 11 says "Use `self.validate_schema(df, schema)`... Note: `self.output_schema` and `self.input_schema` are set by the engine's `_initialize_components()`". This is misleading — the manual call is exactly what ENG-CR-05 flags. Plan task in 7.1-01 should add a note to ENGINE_COMPONENT_PATTERN.md: "Subclasses MUST NOT call `self.validate_schema` inside `_process()`. The base class runs validation automatically in step 7c."

**Warning signs:** `_process` calls `self.validate_schema` (grep for `self.validate_schema`); look at `aggregate_row.py:355` and `filter_rows.py:253` — both will be removed.

### Pitfall 3: pandas 3.0 chained assignment + per-chunk streaming = view/copy confusion

**What goes wrong:** In streaming mode, chunks come from `input_data.iloc[start:end]` — a view in older pandas, a copy in pandas 3.0 with CoW (per project_pandas3_installed.md memory). Modifying chunk contents (e.g., `chunk[col] = pd.NaT`) may or may not affect the parent depending on pandas version. With CoW enabled (default in 3.0), it's a copy — safe but allocations cost.

**Why it happens:** pandas 3.0 CoW is a behavior change from 2.x; manager-era code may assume mutating the chunk is fine without `.copy()`.

**How to avoid:** Use `.copy()` explicitly when assigning to chunk columns, OR (preferred) use vectorized rebuild patterns: build a fresh DataFrame from the chunk rather than mutating it.

**Warning signs:** SettingWithCopyWarning in tests (silenced in pandas 3.0 most paths); subtle test failures only on streaming path.

### Pitfall 4: Empty string vs null collapse in `pd.to_numeric(..., errors="coerce")`

**What goes wrong:** `pd.to_numeric("", errors="coerce")` returns `NaN`. So `""` and `None`/`null` look identical post-coercion. The user's intent ("this string column has empty strings, those are meaningful") gets erased.

**Why it happens:** pandas defaults; no per-column knob in current BaseComponent.

**How to avoid:** D-10 introduces `treat_empty_as_null` schema attribute. Default `true` for numeric (preserves current behavior, since for numeric `""` → NaN is the only sane interpretation), `false` for string (preserves intent — empty strings stay empty).

**Warning signs:** A column declared `str` ends up with `pd.NA` instead of `""` after the engine processes it. Compare against Talend output: Talend's tFileInputDelimited reads `""` as `""` for string columns by default.

### Pitfall 5: `_propagate_input_schemas` overwriting per-input schemas

**What goes wrong:** `to_schema["input"] = list(upstream_output)` runs once per flow. For a multi-input target (tMap with main + lookup), the second flow's assignment overwrites the first. Engine sees only the last upstream's schema.

**Why it happens:** Original design assumed single-input targets only. Multi-input (tMap, future tJoin) was never wired.

**How to avoid:** Write to `to_schema["inputs"][flow_name] = list(upstream_output)` (a per-flow map) in addition to the legacy `to_schema["input"] = ...` (kept for single-input back-compat). Plan 7.1-05 fix.

**Warning signs:** A tMap with both main and lookup inputs has identical schemas in both inputs even though the upstreams are different shapes. [VERIFIED: `converter.py:317`]

### Vectorized normalize for ENG-CR-02 fix

```python
# Source: synthesized from Talaxie tNormalize_main.javajet logic
# (https://raw.githubusercontent.com/Talaxie/tdi-studio-se/master/main/plugins/
#  org.talend.designer.components.localprovider/components/tNormalize/tNormalize_main.javajet)

def _process(self, input_data):
    if input_data is None or input_data.empty:
        return {"main": pd.DataFrame() if input_data is None else input_data,
                "reject": None}

    norm_col = self.config["normalize_column"]
    sep = self.config.get("item_separator", ",")
    trim = self.config.get("trim", False)
    discard_trailing = self.config.get("discard_trailing_empty_str", False)
    dedupe = self.config.get("deduplicate", False)

    if norm_col not in input_data.columns:
        # Talend tNormalize errors out if the column doesn't exist
        raise ConfigurationError(f"[{self.id}] Column '{norm_col}' not found")

    df = input_data.copy()
    # Talend semantics: split with limit=-1 keeps trailing empties; limit=0 drops them
    # https://raw.githubusercontent.com/Talaxie/.../tNormalize_main.javajet
    if discard_trailing:
        # Java's String.split(delim) drops trailing empties by default
        splits = df[norm_col].astype("string").fillna("").str.split(
            sep, regex=False
        )
    else:
        # Java's String.split(delim, -1) keeps trailing empties
        splits = df[norm_col].astype("string").fillna("").str.split(
            sep, regex=False, n=-1, expand=False
        )

    # Talend ordering: discard_trailing → trim → dedupe
    # (per help.qlik.com tNormalize docs: "When both Discard the trailing
    #  empty string and Trim resulting values check boxes are selected,
    #  the former works first.")
    if discard_trailing:
        # Strip trailing empties using a per-row Python list trim
        # (vectorized version requires dropping right-aligned reduce)
        splits = splits.apply(_strip_trailing_empties)
    if trim:
        splits = splits.apply(lambda lst: [v.strip() for v in lst])
    if dedupe:
        splits = splits.apply(lambda lst: list(dict.fromkeys(lst)))

    df[norm_col] = splits
    out = df.explode(norm_col, ignore_index=True)
    # Talend semantics: empty result emits ZERO rows (not one empty-string row)
    # — verified via Talaxie source which iterates lastNoEmptyIndex_ from 0,
    # producing nothing when the count is 0.
    out = out.dropna(subset=[norm_col]) if discard_trailing else out
    out[norm_col] = out[norm_col].fillna("")
    return {"main": out, "reject": None}


def _strip_trailing_empties(lst):
    """Mirror Talaxie tNormalize's lastNoEmptyIndex_ logic."""
    last_idx = len(lst)
    for i in range(len(lst), 0, -1):
        if lst[i - 1] != "":
            last_idx = i
            break
        last_idx = 0
    return lst[:last_idx]
```

### Multi-input schema propagation fix (ENG-CR-04)

```python
# Source: src/converters/talend_to_v1/converter.py:268-322 (post-fix)

def _propagate_input_schemas(components_map, flows):
    for flow in flows:
        from_id, to_id = flow["from"], flow["to"]
        from_comp = components_map.get(from_id)
        to_comp = components_map.get(to_id)
        if not from_comp or not to_comp:
            continue
        from_schema = from_comp.get("schema")
        to_schema = to_comp.get("schema")
        if not isinstance(from_schema, dict) or not isinstance(to_schema, dict):
            continue

        # Per-connector lookup with case-tolerant normalization (fixes
        # the fragile re-uppercase pattern; ENG-WR-09)
        upstream_output = None
        outputs_map = from_schema.get("outputs")
        if isinstance(outputs_map, dict):
            norm_map = {str(k).upper(): v for k, v in outputs_map.items()}
            connector_key = (flow.get("type") or "").upper()
            upstream_output = norm_map.get(connector_key)
        if upstream_output is None:
            upstream_output = from_schema.get("output")
        if upstream_output is None:
            continue

        if "input" not in to_schema:
            continue

        # FIX ENG-CR-04: write per-flow input schema for multi-input targets
        inputs_map = to_schema.setdefault("inputs", {})
        flow_name = flow.get("name") or from_id
        inputs_map[flow_name] = list(upstream_output)

        # Preserve legacy single 'input' for back-compat single-input components
        # — last-write-wins is OK because (a) single-input targets only see one
        # flow, (b) multi-input targets must read from inputs[flow_name] going
        # forward.
        to_schema["input"] = list(upstream_output)
```

## Open Questions

1. **WR-10 (count + ignore_null=False)**
   - What we know: Talend doc lists `count` as "calculates the number of rows". SQL convention: `COUNT(col)` = non-null count.
   - What's unclear: Whether Talend's per-column IGNORE_NULL flag changes count's semantics specifically. No public Talend job-runtime source available to inspect.
   - Recommendation: Mark as WRONG / no change. Plan 7.1-06 task 1 records the verdict and rationale. If a real Talend-converted job is found where count differs, file a Phase 12 follow-up.

2. **List delimiter and `union` semantics for non-geometry usage**
   - What we know: Talend doc says `union (geometry)` — implies geometry-specific (PostGIS-style geometry union).
   - What's unclear: How Talend handles `union` for non-geometry columns. The audit's claim "union should dedupe" is reasonable for non-geometry but not actually documented.
   - Recommendation: For non-geometry, treat as "distinct list, joined" (dedupe + sort + join via list_delimiter). Document the deviation explicitly.

3. **Talend default date pattern chain ordering**
   - What we know: Talend's tFileInputDelimited has a per-column `date_pattern` parameter. When not specified, it errors on parse.
   - What's unclear: BaseComponent's chain (Talend defaults → explicit → ISO 8601 → inference) is more lenient than Talend. Per CONTEXT.md Claude's discretion this is an acceptable improvement, but the planner should be aware our chain is ahead of Talend.
   - Recommendation: Document in BaseComponent docstring: "Date parsing chain is intentionally more lenient than Talend; explicit `date_pattern` always wins."
