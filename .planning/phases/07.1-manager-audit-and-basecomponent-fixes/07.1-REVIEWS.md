# Per-Plan Concerns

### 07.1-01: BaseComponent Rewrite
*   **Concern 1 (Medium):** `validate_schema` relaxed nullability for rejects. The plan suggests modifying the `reject_schema` on the fly to set `nullable=True`. Ensure this doesn't accidentally mutate the shared schema object if it's cached or passed by reference.
*   **Concern 2 (Low):** `treat_empty_as_null` logic for non-string types. If `False`, it raises `DataValidationError`. Ensure this doesn't conflict with the `nullable=True` default for numeric columns which might expect `NaN` (coerced from `""`) to be acceptable.

### 07.1-02: Maven pom.xml Fix
*   **Concern 1 (High):** Dependency Extraction. Task 3 relies on extracting classes from a Windows-absolute path that the review already established will likely not exist on the executor's machine. The "fallback to Option 2 (copy sources from Talaxie)" should be elevated to a primary sub-task or the agent should be explicitly directed to fetch the 4 helper classes via `web_fetch` if the JAR is missing.

### 07.1-03: FileOutputDelimited Rewrite
*   **Concern 1 (Medium):** Passthrough integrity. The plan correctly copies the DataFrame for writing, but ensure `_apply_date_patterns` does not accidentally trigger a `SettingWithCopyWarning` by modifying a slice of the original if the copy logic is shallow.

### 07.1-04: FileInputDelimited Patch
*   **Concern 1 (Medium):** Exception Hierarchy. Switching `DataValidationError` to `ValueError` in inner loops solves the catch leakage, but ensure that when `die_on_error=True`, these are re-wrapped into a descriptive `DataValidationError` at the component boundary to maintain the engine's exception contract.

### 07.1-05: FilterRows + Converter Patch
*   **Concern 1 (High):** Multi-input Schema Consumption. The converter fix (ENG-CR-04) adds `to_schema["inputs"][flow_name]`. However, the existing `tMap` component (Phase 5) only knows how to read `self.input_schema` (the legacy single map). tMap will remain broken (last-input-wins) until it is updated to read from the new `inputs` map. This update is missing from the plan.
*   **Concern 2 (Medium):** Logical Operator Mapping. `_LOGICAL_OP_MAP` handles `&&` and `||`. Ensure the converter always produces these tokens; Talend XML sometimes uses `AND`/`OR` depending on the version.

### 07.1-06: AggregateRow Patch
*   **Concern 1 (Medium):** `list_object` return type. In pandas, a column containing lists is an `object` dtype. Ensure downstream components (like `tNormalize` in 07.1-07) are tested against this specific return type, as it is the first time the engine will emit nested Python lists in a DataFrame.

### 07.1-07: Normalize Rewrite
*   **Concern 1 (Low):** `explode` index reset. The plan uses `ignore_index=True` in `explode`. Verify that tNormalize in Talend preserves the original row index signal (e.g. for `NB_LINE` tracking of the *input* row).

### 07.1-08: Java Routines Patches
*   **Concern 1 (Medium):** JUnit Prerequisite. This plan **must** execute after 07.1-02. While the wave structure correctly places them in Wave 2 vs Wave 1, the agent must ensure a clean `mvn package` is run after the pom.xml fix before attempting to run the new JUnit tests.
*   **Concern 2 (Low):** `Mathematical.INT` vs `Numeric.INT`. The plan guards `Numeric.INT` but notes `Mathematical.INT` might not exist. Talend usually keeps these in separate classes; ensure the regression guards are applied to the correct class based on the source code.

# Risks the Plans Don't Address

*   **Downstream Multi-Input Breaking:** As noted in 07.1-05, fixing the converter to support multi-input schemas is only half the battle. `tMap` and future `tJoin` need to be updated to read from `schema["inputs"]`. This risk should be addressed by adding a task to 07.1-05 to at least audit/update `map.py`.
*   **Binary Bloat:** Committing a 5MB+ JAR (`routines-system.jar`) in 07.1-02. This is accepted for portability but should be monitored if more vendored JARs are added.
*   **Performance vs Parity:** The `union` aggregator fix in 07.1-06 uses a `set` and `sorted`. If the input is massive, this Python-level lambda will be slow. Since this is an audit/correctness phase, this is an acceptable risk, but should be noted for Phase 12.
