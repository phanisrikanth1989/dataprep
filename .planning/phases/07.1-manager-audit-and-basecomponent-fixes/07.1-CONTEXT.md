### Empty string vs null semantics (Area 2 / G-12)
- **D-10:** Configurable per-column via `nullable` plus a new schema attribute `treat_empty_as_null` (default `true` for numeric/date columns to preserve current pandas-like behavior; default `false` for string columns to preserve user intent). Conversion to NaN on `_coerce_column_type` consults this attribute, not blanket coercion. Applies only to `BaseComponent._apply_output_schema_validation` and downstream coercion sites — does not retrofit input parsing in Phase 4 file I/O (those rejects remain governed by Phase 4 D-06).

### `die_on_error` semantics (Area 4 / G-05)
- **D-11:** When `die_on_error=true`, schema validation failures (non-nullable null, type coercion error, length truncation in strict mode) raise `DataValidationError` with the row index and column. When `die_on_error=false`, the offending row routes to the **reject flow** with `errorCode="SCHEMA_VIOLATION"` and `errorMessage` describing the specific violation. Aligns with Phase 4 D-06/07 (REJECT flow already has errorCode/errorMessage). No silent drops, no silent nulls.

### Streaming + schema validation (Area 5 / G-10)
- **D-12:** Per-chunk type coercion + per-chunk reject routing inside `_execute_streaming`. Cross-chunk / global checks (none currently exist for this scope) defer to a single batch pass at end-of-stream. The current behavior (concat full DF, then validate) is the bug — streaming was meant to keep memory bounded; the BaseComponent rewrite (D-01) implements the per-chunk path.

### Audit claim re-verification policy (Area 7)
- **D-13:** Every plan in this phase opens with a **"verify findings"** task as task 1. For each REVIEW.md / REVIEW-engine.md / REVIEW-javabridge.md finding in the plan's scope, the task records a verdict: REAL / WRONG / NEEDS-INVESTIGATION, citing current code line and any prior-phase decision that contradicts the finding. Fixes only proceed for REAL findings. WRONG findings get a one-line note in the SUMMARY explaining why they were not fixed. Applies the standing `verify_audit_claims` rule structurally, not just culturally.
- **D-14:** ~~Pre-flagged for verification: CR-05 likely WRONG~~ → **SUPERSEDED by RESEARCH.md verdict: CR-05 IS REAL.** Phase 6 D-09 was wrong. Talaxie source (`tAggregateRow_messages.properties` — `LIST_DELIMITER.NAME=Delimiter (only for list operation)`) confirms `list_object` should produce a Python list, not a delimited string. 7.1-06 fixes CR-05 (real), WR-09 (real), WR-11 (real); drops WR-10 per D-15.
- **D-15:** ~~Pre-flagged for verification: WR-10 cross-check~~ → **SUPERSEDED by RESEARCH.md verdict: WR-10 is WRONG / no change.** No public Talend job-runtime source available, but SQL convention + Talend docs both indicate count = non-null regardless of `ignore_null` flag. 7.1-06 SUMMARY notes WR-10 as a WRONG finding. Three additional WRONG findings surfaced by research (supersede CONTEXT scope below):
  - **CR-07** (Numeric.INT NumberFormatException on "100.5") — WRONG. OpenDAS Talend upstream is byte-identical; fixing breaks Talend parity.
  - **WR-14** (StringHandling.LEN returns -1 for null) — WRONG. Talend upstream returns -1, not 0. Reviewer was mistaken.
  - **IN-01** (Mathematical.CHAR uses Character.forDigit) — WRONG. Matches Talend upstream exactly.
  Net: 7.1-08 fix scope drops from 5 items to **3** (WR-15 INSTR bounds, WR-16 deprecated `new Float(double)`, CR-08 partial — wrap-cause-as-message but keep RuntimeException type per Talend). New assumption: vendoring Talend `routines.system.*` JAR (A1) needs Apache 2.0 license sign-off in 7.1-02 task 1.

## Existing Code Insights

### Reusable Assets
- **BaseComponent template method** (`src/v1/engine/base_component.py`) — `execute()` orchestrates `_validate_config` → `_resolve_expressions` → `_select_mode` → `_execute_batch`/`_execute_streaming` → `_apply_output_schema_validation` → `_update_stats`. Rewrite preserves this skeleton; the bug surface is in the body of each step.
- **`exceptions.py` hierarchy** (`src/v1/engine/exceptions.py`) — `ETLError`, `ConfigurationError`, `DataValidationError`, `ComponentExecutionError`, `FileOperationError`, `JavaBridgeError`. All fix code raises typed exceptions, never bare `Exception`.
- **REJECT flow plumbing** (Phase 1 D-04) — engine routes any named output flow to downstream connectors. Manager's `errorMessage` collision (ENG-WR-06) is the only blocker; namespace fix unblocks the existing plumbing.
- **`@REGISTRY.register()` decorator** — components self-register; rewrites preserve this pattern.

### Established Patterns
- **Component config snapshot/restore** (Phase 1 D-14) — `_original_config` deepcopied at construction, never mutated. ENG-CR-07 violates this; fix restores compliance.
- **`_validate_config()` abstract contract** (Phase 1 D-13) — every component must implement and raise `ConfigurationError`. ENG-CR-01 (normalize returns list) violates; fix restores compliance.
- **Converter outputs canonical config keys** (Phase 4 D-04) — engine reads them directly. No mapping shims allowed in fix code.
- **Talend default for file I/O encoding is `ISO-8859-15`, fieldseparator `;`** (Phase 4 D-05) — preserve in 7.1-03 rewrite.
- **`list_object` aggregator returns delimited string per Talend** (Phase 6 D-09) — preserve; do NOT "fix" to return Python list.

### Integration Points
- **`base_component.py`** is consumed by every engine component class. The rewrite must not break the abstract contract that 50+ components depend on. Wave 1 → Wave 2 dep is exactly this contract surface.
- **`_apply_output_schema_validation` + `_enforce_schema_column_order`** are the two BaseComponent methods every component's `_process()` flows through. New behavior (per-chunk validation, reject routing on `die_on_error=false`, `treat_empty_as_null` attribute) lives here.
- **`pom.xml`** is consumed by Maven and the JAR build script (`mvn package`). Verify on a clean Mac (`brew install maven` if not present) before merging.
- **Java routines** are loaded into the Py4J bridge classpath via `routine_jars` (Phase 9 design). Routine fixes ship in the `java-bridge-with-dependencies.jar`; no Python-side change needed.
- **Converter `_propagate_input_schemas`** feeds engine `BaseComponent` input handling for multi-input components (tMap, future tJoin). Fix unblocks tMap multi-input correctness; verify against an existing tMap test fixture as a regression gate.
